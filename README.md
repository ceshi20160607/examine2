# examine2

低代码平台 + 独立审批引擎的一体化方案。

**本文档为项目实施的约束与决策来源**；实现与本文冲突时，以本文为准，变更须评审并按需补充 ADR。

## 1. 文档约定

| 用语 | 含义 |
|------|------|
| **必须** | 后端/接口须实现，不可仅依赖前端 |
| **禁止** | 明确不允许的做法 |
| **默认（MVP）** | 第一版采用的路径；演进须 ADR |
| **ADR** | Architecture Decision Record：记录理由、影响面、迁移策略 |

## 2. 术语表（读后续章节前先对齐）

| 术语 | 含义 |
|------|------|
| **Context** | 单次请求必须可还原的上下文，见 **D1**：`platformAccountId` + `systemId` + `tenantId` |
| **AuthContext** | 权限快照：模块/Action/字段/数据域/流程操作等，用于后端硬过滤 |
| **待办** | **`record_node` / `record_node_user` 上当前待处理项**；「我的待办」按此查。**不是**业务主表字段名 |
| **业务主表 / 主数据** | 模块业务记录表（含槽位、`record_data` 等），与审批运行时表区分 |
| **flow_definition** + **flow_node** | 流程**模板/配置**（节点顺序、规则、审批人等） |
| **flow_record** | **一条审批整单**（审批实例）；**整单状态**只在此维护 |
| **record_node** | **某 flow_record 上的运行时节点**（待办载体、节点结论） |
| **record_node_user** | 多人审批时**具体审批人**维度的待办/结论 |
| **after** | 节点提交产生的**临时快照**；常态下一节点一次有效提交；**或签**时多人可能先后提交，可存多条，**以最后一次提交为准**（**D13**）；落库形态见 **D21** |
| **relation_id** | 业务侧主键，与 **flow_record** 关联时表示「本审批单对应哪条业务数据」（逻辑名，实现可映射列名） |

**旧库说明**：`doc/examine.sql` 仅供对照；**新库须单独设计与迁移**，下文表名均为**逻辑名**。

## 3. 产品目标形态

- **平台账号**：注册登录平台，不依赖是否已建系统/模块。
- **系统（低代码创建）**：平台内创建「系统」（类比飞书/钉钉应用空间）；内含组织、成员、角色、权限、模块与数据。
- **系统用户**：复用平台账号；可有**系统专属扩展字段**。
- **多租户开关**：按系统；关 = 单租户（`tenantId` 固定默认）；开 = 同系统下多租户隔离。
- **模块**：业务对象、表单、列表、规则、动作、扩展能力。
- **流程**：独立引擎；模块可配置使用，也可对外协议接入；含异常策略。

## 4. 架构决策清单（ADR）

修改下列决策须补充 ADR。

| # | 决策 | 内容 |
|---|------|------|
| D1 | **Context** | 所有业务请求必须携带并可还原：`platformAccountId` + `systemId` + `tenantId`（关多租户时 `tenantId` 为固定默认值）。 |
| D2 | **多租户 MVP** | 模块**元数据**默认**全租户共享**；**业务数据**按 `tenantId` **强隔离**。每租户独立元数据为后续能力。 |
| D3 | **主表槽位** | 槽位列**列集固定**，**禁止**业务侧手动扩表；仅允许调整「字段 → 槽位」映射，且须**数据迁移**（回填、索引等）。 |
| D4 | **record_data** | **一字段一行（EAV）**为主；复杂结构可用 JSON 子列；可搜索字段须落 typed-value。 |
| D5 | **审批分层状态** | **节点级**状态只在 **record_node / record_node_user**；**整单**状态只在 **flow_record**。**禁止**两处各维护一套「真相」。 |
| D6 | **模板 → 运行时完整复制** | 生成 **flow_record** 时，须将当时 **flow_definition** 下**全部 flow_node** 及配置**复制**为 **record_node**（运行时自有副本，**非**运行时再引用模板不落库）。发起时固化 **flow_definition_id**；**改模板** = 新 **flow_definition**，**不**改已发起的 **flow_record**。 |
| D7 | **权限** | **Deny-first**；判定顺序见 **§7.5 AuthContext**。 |
| D8 | **异步** | MVP 仅 **Outbox 轻量版**（DB + 轮询）；不依赖 Kafka。 |
| D9 | **流程引擎** | 以现有 `examine` 能力为演进基线，不另起一套简化表替代。 |
| D10 | **ID** | 业务主键统一 **Snowflake 类**（或项目既定 ID 服务）；对外与内部一致，**禁止**客户端自造（开放 API 另有约定除外）。 |
| D11 | **幂等键** | 写接口幂等键作用域：`systemId + tenantId + principal + Idempotency-Key`（principal 为平台用户或第三方 **appId**）；TTL 默认 **24h**（可配置）。 |
| D12 | **查询 DSL** | SearchBO 仅**白名单**操作符/字段/深度（含 `like`）；禁止任意 SQL；须上限：条件数、排序字段数等。 |
| D13 | **after 链** | **after** 均为临时数据。**常态**下一 **record_node** 仅一次有效提交；**或签**时多人可能先后提交，可落多条 **after**，**本节点生效数据以最后一次提交为准**；**下一节点**初始数据 = **上一节点最后一次生效的 after**。无业务层「冲突合并」语义；历史行可保留供审计/展示。 |
| D14 | **审计** | **业务操作审计 / 流程轨迹 / 技术排障日志** 三类分离，禁止混为一张表。 |
| D15 | **权限缓存** | AuthContext 变更须**失效**（优先 `memberId + systemId + tenantId`）；最大 TTL **5min**（可配置）。 |
| D16 | **幂等 HTTP** | 同键重复请求且首次已成功：返回 **HTTP 200**， body 含 **`idempotent_replay: true`**，不重复副作用；首次若为 **201**，重放仍 **200**。**禁止**用 **409** 表示幂等重放。 |
| D17 | **运行中加签** | 仅在**当前 flow_record 已复制出的运行时节点图**上、**尚未进入**的后续路径**追加 record_node**；**不修改** **flow_definition** 模板（同 D6）。**与模板无并发关系**：加签发生在运行时副本上，模板仅用于发起时复制。插入后待办与 **record_node** 须与运行图一致并重算。 |
| D18 | **审批结果形态** | **审批中 / 通过 / 拒绝 / 终止 / 暂停 / 作废 / 驳回** 语义与接口、枚举**一一对应**，见 **§11.3**。 |
| D19 | **拒绝/作废后再发起** | **flow_record** 终态为拒绝或作废后，可改业务数据，再发起须**新建 flow_record**（绑定**当时**的 **flow_definition**）。终止、暂停语义见 D18。 |
| D20 | **主数据 vs 审批** | **默认**审批过程**不**写业务主表；**仅整单通过**时回写（或 Outbox/回调）。**例外**：flow 配置显式开启「**字段变更立即同步主数据**」时允许写入（须审计）。 |
| D21 | **after 存储** | 每次提交**单独落库**一条 **after** 记录（或签时可能多条）；**同节点生效口径**见 D13；**不与**节点状态混在同一行覆盖写。 |

## 5. 全局技术约定

### 5.1 ID

`recordId`、`flow_record_id`、`record_node_id`、`fileId`、`eventId` 等须可追溯；**禁止**客户端随意指定雪花 ID 写入（开放 API 明确允许并校验者除外）。

### 5.2 幂等与 HTTP

- **副作用**：相同幂等键（D11）**至多一次**；返回同一业务结果。
- **HTTP**：幂等重放表现见 **D16**；与 **409** 等资源冲突区分。

## 7. 横切能力

### 7.1 Context（D1）

- **platformAccountId**：登录态  
- **systemId**：当前系统  
- **tenantId**：关多租户时为**固定默认值**，**禁止**客户端随意覆盖；开多租户时**必传**且须校验成员身份。  

**禁止**跨 `systemId` / `tenantId` 读写（含管理端）。

### 7.2 基础能力（在 D1 范围内）

**消息**、**待办**、**日志**为系统默认能力；存储与查询均须在 Context 内。

- **消息**：当前用户在 **AuthContext** 下**可见**的消息列表（含通知等），后端按权限过滤，**非**全员广播同一套数据。  
- **待办**：当前用户在 **AuthContext** 下**可见**的待办（与 **§2 待办**、`record_node` / `record_node_user` 查询一致），即「我的待办」= 权限范围内的待处理项。  
- **日志**：操作与排障日志，检索范围仍在 Context 内。

**说明**：MVP **不**对**内部业务接口**做统一网关限流；第三方开放 API 的限流等仍见 **§8.3**。

### 7.3 平台账号与系统成员

- **平台账号**：登录与平台级资料（手机/邮箱/密码/头像等）。  
- **系统成员**：某系统（及租户）内身份；支持一人多部门、多角色、多数据权限与扩展字段。

### 7.4 多租户（D2）

- **关闭**：默认单租户，数据仍带默认 `tenantId`，便于日后开启。  
- **开启**：组织/成员/数据按租户隔离；读写须后端带 `tenantId` 过滤。

### 7.5 AuthContext

**必须**缓存（会话或短期）；**所有**业务接口基于其硬过滤。

**包含**：模块与可见 Action；字段读/写/必填；数据域（含 TEAM）；对 **flow_record** 的流程操作与审批中编辑。

**判定顺序（须按序短路）**：

1. 系统/租户成员 → 非成员 **403**  
2. 模块可见性  
3. Action 权限  
4. 数据域  
5. 字段权限（Deny 优先于 Allow）  
6. 流程节点策略（如 `edit_policy`）  

**冲突规则**：Deny-first；禁止仅前端隐藏。

**缓存失效（D15）**：成员角色、字段权限、Action、数据域、模块可见性、租户成员关系等变更须触发失效；优先按 `memberId + systemId + tenantId`；全量失效仅应急。

### 7.6 自定义扩展

自定义 Action、字段类型、规则 DSL、Webhook 等：**须沙箱、限额、审计**；**禁止**绕过 AuthContext。

## 8. 低代码模块

### 8.1 元数据

- **Field**：类型、校验、默认值、字典、可搜索/排序、扩展等  
- **FormLayout**：布局、分组、联动、显隐、只读  
- **ListView**：列、排序、筛选、视图  
- **Relation**：主从、引用、lookup、级联  

### 8.2 Action 与策略

- **标准 Action**：LIST / DETAIL / ADD / EDIT / DELETE / IMPORT / EXPORT / PRINT / TRANSFER / TRANSFORM / LOCK / UNLOCK / TEAM  
- **ActionPolicy**：角色/条件、字段可写、数据域、是否挂流程、审计与通知  

### 8.3 对外开放 API（第三方调本系统）

模块可对已鉴权的第三方开放数据与 Action（仍受 Context + AuthContext 约束）。

- **身份与授权**：**`appId` + `appKey` + `appSecret`** 与绑定的 **`systemId` / `tenantId`、模块/Action 授权** 共同确定**唯一调用方**及可访问范围；与 **D11** 幂等域中的 **principal（appId）** 一致，无需再引入并列身份模型。  
- **凭证**：统一 **`appId` + `appKey` + `appSecret`** 换 token 或签名校验（禁止多套并列命名）  
- **令牌**：短期 `access_token` 或请求级签名（timestamp + nonce + HMAC）  
- **授权范围**：绑定 `systemId`/`tenantId`；模块级/Action 级；**字段级与数据域**仍按 **AuthContext**（与上条身份一致，第三方不可绕过）。  
- **写接口**：头 **`Idempotency-Key`**（D11）；限流（可按 app / system / tenant）；审计调用方与结果；防重放（时间窗 + nonce）；可选 IP 白名单等  
- **服务主体**：映射为 **service principal**，仍走 AuthContext，禁止旁路

## 9. 数据引擎

### 9.1 槽位与 record_data（D3、D4）

- 槽位示例：`number1..5`、`text1..10`、`date1..5`、`option1..5`、`user1..2` 等；**同一模块同一槽位仅映射一个字段**；高频筛选/排序走槽位，其余 **record_data**。  
- **列集固定、禁止手工扩表**；改映射须迁移。  
- **record_data**：EAV 为主；复杂结构可 JSON + typed-value 承载可搜索子列。

### 9.2 typed-value

按类型选用：`string_value` / `number_value` / `decimal_value` / `date_value` / `option_value` / `user_value` / `dept_value`；JSON 用于多值/表格等。

### 9.3 查询 DSL（D12）

- 字段白名单：仅「当前用户可读」字段  
- 操作符白名单：`eq/ne/gt/.../in/between/like/isNull`；**禁止 rawSql**  
- 上限：条件数、`in` 长度、排序字段数、`like` 长度等  
- MVP：**禁止**任意跨模块 join；仅预置 Relation 的受控 join（单独授权）

### 9.4 附件

独立服务；业务存 `file_id` / `batch_id`；下载鉴权后临时链接；模块与流程复用。

## 10. 出站连接器（本系统调第三方）

用于字段补全、自定义 Action、流程节点拉取/推送外部接口等。

- **封装**：HTTP、签名、重试、超时、熔断；业务只配参数与映射  
- **节点**：在 **flow_node / record_node** 上配置 URL、参数与上下文字段映射、响应 JSON → 字段映射（推/拉）  
- **密钥**：加密存储，按 system/tenant 隔离，可轮换  
- **失败**：同步是否阻断流程；异步优先 **Outbox**

## 11. 流程引擎

### 11.1 主链路（端到端）

1. 在 **flow_definition / flow_node** 配置流程；模块绑定流程与触发时机（如新建即发起）。  
2. 业务产生主键 **relation_id** 时**同时**创建 **flow_record**，固化 **flow_definition_id**，并**完整复制**模板节点为 **record_node**（及 **record_node_user**）— **D6**。  
3. 审批在 **record_node / record_node_user** 推进；过程内字段修改默认在 **after** 与流程轨迹中；**默认**不写业务主表 — **D20**。  
4. 全部节点完成后，在 **flow_record** 上写**整单**终态。  
5. **整单通过**时按 **D20** 回写主数据或 Outbox/回调；拒绝/作废后再发起见 **D19**。

### 11.2 flow 作用域

- **平台级**：对外开放的审批能力  
- **系统内模块级**：绑定某模块、由 Action 触发  

鉴权仍满足 **D1**；**flow_definition** 须能区分场景。

### 11.3 整单状态 vs 节点状态（D5、D18）

**整单（flow_record）— 实现前将状态与迁移写死为表/枚举，禁止随意加状态**

| 逻辑状态 | 说明 |
|----------|------|
| 审批中 | 未终态；含进行中、部分节点已完成等 |
| 暂停 | 非终态，可恢复 |
| 通过 | 终态；触发 **D20** 主数据回写 |
| 拒绝 | 终态；**D19** |
| 终止 | 终态；主动停止；**不回写**通过类业务约定 |
| 作废 | 终态；**D19** |

对外可用 `terminal_reason` 区分终态原因；**暂停**与**终止**禁止混用。

**节点（record_node / record_node_user）**：待处理 / 已通过 / 已驳回 / 转交等**仅记在节点侧**，不替代整单状态。

**驳回（D18）**：回到上一节点；**整单仍为审批中**；会签/并行下「上一节点」须可解析且合法。

**说明**：同一业务是否允许多条进行中的 **flow_record** 由产品定义；**D6** 已覆盖版本与运行时副本，不另设「并发条数」ADR。

### 11.4 主数据与审批过程（D19、D20）

- **默认**：节点编辑**不**写业务主表；**仅整单通过**时一次性回写或异步通知。  
- **例外**：flow 开启「字段变更立即同步主数据」时，按 **D20** 写主表并审计。  
- **展示**：发起基准沿 **after** 链递进（**D13**）；中间节点通过默认不落主表（同上例外）。  
- **模块元数据**：审批过程中不因流程改模块定义。  

### 11.5 after 链与存储（D13、D21）

- **常态**：一个 **record_node** 对应一次有效提交，一条 **after** 即可表达该节点出口数据。  
- **或签**：多人可能先后提交，可存多条 **after**，**本节点生效数据以最后一次提交为准**；下一节点初始 = 上一节点**最后一次生效**的 **after**。  
- **节点之间**：链式传递规则不变（上一节点出口 → 下一节点入口）。  
- **D21**：每次提交独立插入 **after** 行，不与节点状态混写覆盖。

### 11.6 节点可编辑与出站集成

- **edit_policy**：可编辑字段、时机、是否允许改影响分支的字段（默认慎改分支关键字段）。  
- 节点级 **推/拉** 第三方：见 **§10**。  

### 11.7 运行中加签（D17）

仅在当前 **flow_record** 发起时**已复制好的运行时节点图**上、**尚未进入**的后续路径**追加 record_node**；**不修改** **flow_definition** 模板。加签是对**运行时副本**的扩展，与模板定义无并发冲突。须权限与全量审计；待办与 **record_node** 与运行图一致。

### 11.8 外部系统接入本流程（协议）

- 发起 / 查询 / 审批：流程 key、业务标识、快照、候选人等  
- 写接口：**Idempotency-Key**（D11），同键不得产生第二条 **flow_record**  
- 错误：统一结构（`code`、`message`、`requestId`）  
- 回调：可配置 URL；**至少一次**投递 + 消费者幂等  

### 11.9 模块内接入

ActionPolicy 触发流程；条件可读模块字段（含自定义字段）。

## 12. 审计分层（D14）

| 类型 | 用途 | 保留 |
|------|------|------|
| 业务操作审计 | 对业务数据的操作（合规） | 长期，按模块/记录 |
| 流程轨迹 | 节点、意见、转交、审批中编辑等 | 绑定 **flow_record** 展示 |
| 技术排障日志 | Outbox、回调、Connector、慢查询 | 较短 + 可采样 |

## 13. Outbox（D8）

- 事务内写 `outbox_event`；轮询 + 线程池消费  
- **至少一次**投递；消费者幂等（`event_id` / 业务幂等键）  
- 重试、死信、人工重放；可观测：积压、失败率、延迟  

**必须异步**：导入导出、附件后处理、外部回调、索引刷新等。  
**禁止异步**：AuthContext 计算、权限校验、主事务成败判定。

## 14. 里程碑与验收

| 阶段 | 交付 | 验收要点 |
|------|------|----------|
| A | 平台账号、建系统、进入系统、Context | 无 Context 或非法租户 → 403；关多租户时 tenant 固定 |
| B | 成员、多部门、多角色、数据权限、AuthContext | 同用户不同角色列表不同；Deny 生效 |
| C | 模块元数据、槽位、record_data、Action CRUD | 列表走槽位/typed-value；无权限绕过写 |
| D | flow_record、record_node、D6/D13/D21、D20、D18 | 主链路可测；仅通过回写；拒绝/作废可再发起（D19） |
| E | Outbox、回调、开放 API 幂等 | 同 Idempotency-Key 不产生第二条 flow_record；死信可重放 |

## 15. 工程约束

| 项 | 约定 |
|----|------|
| **构建** | **Maven** 多模块；父工程 `pom.xml` 为聚合与依赖管理。 |
