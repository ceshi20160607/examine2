## 1. 产品与价值主张

### 1.0 平台能力概览

平台可按以下能力理解（实现上对应后端模块，名称以代码为准）：
| 能力 | 说明 |
|------|------|
| **用户与账号（user）** | 账号登录、平台控制台、系统列表/创建、进入系统。 |
| **无代码（lowcode）** | 创建并维护**自建业务系统**、模块、数据、规则与动作。 |
| **审批（flow）** | 平台审批服务（模板、实例、节点、待办与终态）；**不发起业务审批**，只提供能力给调用方。细则见 **第 9 章**。 |
| **消息（message）** | 平台级能力，当前阶段做「**全平台可见**」。 |
| **待办（todo）** | 平台级能力，当前阶段做「**全平台可见**」。 |

### 1.1 平台与自建系统（固定 `systemId=0`）

- 平台本身逻辑上是一个固定系统：**`systemId = 0`**。用户先登录平台，处于**平台态**（`systemId=0`），可使用平台能力（flow/message/todo/user）并看到自己创建的系统列表。  
- 用户点击某个系统进入后，Context 切换到该系统的 **`systemId != 0`**，进入**系统态**，在该系统内建模块、配权限、操作数据。  
- **平台态没有租户（tenant）**：`systemId=0` 时不使用 `tenantId`。  
- 多租户只作用于自建系统：若某自建系统开启多租户，则进入系统时需要**选择 `tenantId`**，之后所有业务读写按 `tenantId` 隔离。

### 1.2 登录与鉴权（内部 vs 对外）

**内部（平台用户）**

1. 登录成功下发 **token**；请求携带 token。  
2. 后端根据 token 从 **Redis**（或等价会话存储）取会话，解析 **`platId`** 与当前 **`systemId`**。  
3. 进入某自建系统时，更新会话中的 `systemId`；若该系统开启多租户，则进入系统时选择的 `tenantId` 也写入会话。  
4. Redis 中查不到有效会话（过期/注销/非法 token）→ 视为未登录，要求重新登录。  
5. 自建系统内操作权限由该系统内配置（AuthContext）；平台态与系统态权限/菜单范围按产品规则区分。

### 1.3 自建系统切换方案（本项目约定）

为避免「前端当前页面所属系统/租户」与「后端实际执行的 systemId/tenantId」不一致导致串数据，本项目约定采用**token 会话权威**的切换方式：

- **登录**：登录成功下发 **token**。  
- **会话存储**：后端以 token 作为 key，在 **Redis** 中保存会话，包含 `platId`、当前 `systemId`、当前 `tenantId` 等 Context。  
- **请求携带**：除登录/平台公共入口外，所有请求只需携带 token（例如 `Authorization: Bearer <token>`）；**systemId/tenantId 由后端从 Redis 会话解析**，不从请求头解析。  
- **进入系统**：调用「进入系统」接口后，后端**更新该 token 对应会话**中的 `systemId`（以及默认 `tenantId`）。token **不变**。  
- **选择租户**：系统开启多租户时，调用「选择租户」接口后，后端**更新该 token 对应会话**中的 `tenantId`。token **不变**。

**对内 vs 对外调用（总览）**

| 调用方 | 鉴权 | 路径前缀 | systemId / tenantId 来源 |
|--------|------|----------|------------------------------|
| 内部平台用户 | token，会话在 Redis | 对内前缀（以 OpenAPI 为准） | **从 token 对应 Redis 会话解析** |
| 第三方对外应用 | `appId` + `secret` | **`/v1/**`** | 由 `appId` 鉴权与模板/资源授权解析（**第 6.2 节** 补充归属与授权表） |

**会话校验约定（写死）**

- 自建系统内接口：必须先通过 token 鉴权；后端从 Redis 会话读取当前 `systemId/tenantId` 作为 Context。  
- 未进入系统时：会话中 `systemId=0`、`tenantId=0`，仅允许访问平台态接口；访问系统内接口应返回 **HTTP 403**（或按 OpenAPI 约定）。  
- 未开启多租户时：会话中 `tenantId` 固定为 **0**。  
- 系统开启多租户时：必须先选择租户，后端校验 `tenantId` 属于当前 `systemId` 且可用，否则返回 **HTTP 403**。  
- flow：内部调用按上表第一行与本条校验；对外应用调用见上表第二行（不依赖内部会话）。

**平台公共接口（不要求进入系统/选择租户）**

以下接口属于平台态或公共入口（以 OpenAPI 的路径前缀/tag 为准），不要求客户端已进入系统/选择租户：登录、系统列表/创建、进入系统、选择租户、创建/管理对外应用等。除上述范围外，均视为「自建系统内接口」，必须在会话中存在有效 `systemId/tenantId`。

**API 路径约定（对内 vs 对外）**

- **对内**（平台用户 token）：按模块划分路径前缀（例如平台控制台与自建系统业务可使用不同前缀），具体以 OpenAPI 与网关路由为准；**作用域**由 token 对应 Redis 会话中的 `systemId/tenantId` 表达，不靠 OpenAPI `tag` 单独承担鉴权。  
- **对外**（第三方 `appId` + `secret`）：统一挂在 **`/v1/**`** 下，版本号 **`v1`** 为路径中的一段；后续兼容策略（是否并行 `/v2/**`、弃用窗口）在 OpenAPI 与发布说明中约定。

**对外（第三方系统调本系统的 flow）**

总览见上文「对内 vs 对外调用」表。补充：**对外应用**创建、`appId`/`secret`、**`/v1/**`** 与 **`flowId`** 的用法，以及限流/审计/防重放等，以 OpenAPI 与安全基线为准（**第 6.2 节** 写模板归属与 app 授权思路）。

## 2. 架构鸟瞰

### 2.1 领域划分

| 域 | 职责 |
|----|------|
| **platform** | 账号、会话、控制台；平台态（`systemId=0`）；系统列表/创建、进入系统（切换 `systemId`） |
| **system** | 自建系统内组织、成员、角色、数据权限、多租户、成员扩展字段 |
| **lowcode** | 模块、字段、表单、列表、字典、关系、Action；业务通过本域**调用 flow** |
| **data** | CRUD、查询 DSL、权限过滤、导入导出、打印 |
| **flow** | 审批引擎域；`flow_record` / `record_node`；对内与对外协议。定位与边界见 **第 9 章** |
| **message** | 平台消息（当前阶段全平台可见） |
| **todo** | 平台待办（当前阶段全平台可见） |

### 2.2 原则（摘要）

- **请求范围**：业务读写落在「平台用户 + 当前系统」内；若当前系统开启多租户，则还落在「当前租户」内。禁止跨系统、跨租户篡改数据。  
- **数据分层**：整单状态只在 `flow_record`；节点与审批待办在 `record_node` / `record_node_user`。  
- **模板与实例**：发起时把模板**完整复制**为运行时；改模板不追溯已发起实例。  
- **权限**：后端按 **AuthContext** 硬过滤（顺序见 **第 5.4 节**）。  
- **flow 与业务**：见 **第 9 章**（flow 只交付终态与约定字段；业务后续动作由调用方定义）。

## 3. 术语与缩写

| 词 | 含义 |
|----|------|
| **Context** | `platId` + `systemId`；`systemId=0` 为平台态；`systemId!=0` 为自建系统态。若该系统开启多租户，则 Context 还包含 `tenantId`。 |
| **AuthContext** | 当前用户在当前 Context 下的权限快照，后端接口硬过滤。 |
| **EAV** | `record_data` 一字段一行存法。 |
| **Outbox** | 事务内记待投递，异步真正外呼。 |
| **DSL** | 白名单查询，禁止任意 SQL。 |
| **principal** | 人或开放 API 的 `appId`；参与幂等域。 |
| **after** | 节点提交快照；或签多人多条以最后一次通过类提交为准；若出现拒绝等终态则整单按终态结束。 |

## 4. 全局技术约定

### 4.1 ID

主键须可追溯；禁止客户端随意自造雪花 ID（开放 API 另有约定除外）。

### 4.2 幂等与 HTTP

- 幂等键：`systemId` + `tenantId(可选)` + `principal` + `Idempotency-Key`；其中 `tenantId` 仅在进入了开启多租户的自建系统时参与；**未开启多租户时 `tenantId = 0`**；TTL 默认 24h（可配）。  
- 同键已成功重放：HTTP 200 + `idempotent_replay: true`；禁止用 409 表示幂等重放。  

### 4.3 请求失败与事务

失败须明确错误；已开事务应回滚；对用户可提示「未成功，请稍后重试」。与幂等、Outbox 语义不冲突。

### 4.4 业务唯一与「进行中审批」校验

- **平台内（自建系统）**：业务唯一键 = `systemId + moduleId + 数据主键`。  
- **平台外（第三方经对外应用调用）**：唯一键 = `appId + flowId + 业务id`。若 flow 中已存在绑定该标识且尚未终态的实例，则不允许新建，提示「存在审批，不能新建」。  

**字段命名建议（占位）**：平台外传入的「业务id」建议统一落库为 `external_type_id`（或同义字段），避免不同模块各自命名导致后续查询/去重困难。

说明：`external_type_id` 表示「业务主键 id」（用于去重/关联），无论内部还是外部接入，只要使用外部传入的业务主键，都落该字段。

## 5. 横切能力

### 5.1 Context

- platId、systemId。  
- tenantId：仅当进入了开启多租户的自建系统时才存在；平台态（`systemId=0`）不使用 tenant。  
- 未开启多租户时：统一约定 `tenantId = 0`。  
- 禁止跨 systemId / tenantId 读写。  

### 5.2 消息、待办、日志

- message：平台级；当前阶段全平台可见；后端仍按**systemId 与角色/权限**过滤。  
- todo：平台级；当前阶段全平台可见；后端仍按**systemId 与角色/权限**过滤。  
- 日志：操作与排障，不跨 Context。  

说明：「全平台可见」指**一个入口聚合展示**用户在多个自建系统中的消息/待办，不代表“无权限可看所有系统数据”。最终可见范围仍以 AuthContext 与 systemId/tenantId 过滤为准。

### 5.3 多租户

多租户只对自建系统生效：平台态无租户概念。自建系统若开启多租户，则进入系统时选择 tenantId；业务数据按 tenantId 强隔离；元数据（MVP）默认全租户共享。未开启多租户时，统一约定 `tenantId = 0`。

### 5.4 AuthContext

须缓存；所有业务接口硬过滤。判定顺序：成员 → 模块可见性 → Action → 数据域 → 字段（Deny 优先）→ 流程节点策略。最大 TTL 5min（可配）。

## 6. 低代码与开放 API

### 6.1 元数据与 Action

Field / FormLayout / ListView / Relation；标准 Action 与 ActionPolicy 等以实现为准。

### 6.2 对外应用与开放 API（第三方调本系统的 flow）

路径前缀、`appId`/`secret` 与「不依赖内部会话 systemId/tenantId」的总览见 **§1.3**「对内 vs 对外调用」表；本节只补充**模板归属、唯一性与 app→资源授权**。

**前置：对外应用**

1. 在平台创建对外应用，生成 `appId` 与 `secret`（secret 仅展示/轮换时可见，须安全存储）。  
2. 在该应用内创建并维护 flow 模板。  
3. 自建系统模板与对外应用模板**共用一套表结构更合适**，但必须按归属字段区分（自建系统归属 `systemId!=0`；对外应用归属 `appId`），并据此鉴权与查询。  

**模板归属与唯一性（推荐）**

- `owner_type`：`system` / `app`  
- `owner_id`：自建系统用 `systemId`；对外应用用 `appId`  
- `flow_id`：在 **(owner_type, owner_id)** 范围内唯一（即同一系统或同一对外应用下不重复）

**主体 ↔ 资源（授权思路）**

- `appId` 是**对外安全主体**（可轮换 secret、可停用、可限流与审计），不等同于自建系统的 `systemId`。  
- `systemId` 是**内部业务空间/资源边界**。对外应用能访问哪些系统/租户/模板，应通过「主体 ↔ 资源」授权关系控制（例如 app → system/tenant/flow 的授权表），避免把安全主体与业务空间混成一个 id。  
- **与内部 Context 的关系**：`appId` 鉴权通过后，依据 **app 授权**解析可用模板与（若引擎需要）内部 `systemId`/`tenantId` 映射，再执行 flow；**内部 `systemId` 不暴露给第三方作为调用参数**（与 §1.3 表一致）。

**第三方调用链**

1. 用 `appId + secret` 鉴权（token/签名形态见 OpenAPI）。  
2. 请求落在 **`/v1/**`**；用 `flowId` 指明模板，再发起/办理审批。

## 9. flow（只做审批服务）

### 9.1 定位

- flow 只负责审批：模板、实例、节点、待办、终态、约定字段/快照结果。  
- 平台不发起业务审批；业务审批由自建系统内模块调用发起。  
- 对外由第三方通过对外应用（appId/secret）+ flowId 调用同一套引擎。  

### 9.2 主链路（摘要）

1. 配置模板；业务绑定触发。  
2. 创建 `flow_record` 并复制节点为运行时 `record_node`。  
3. 推进审批；过程数据在 after 与轨迹中。  
4. 写整单终态；通过后的业务动作由调用方模块决定。  

### 9.3 单实例与或签

- 同一业务不允许并行多条进行中审批单。  
- 或签：最后一次通过类提交生效；若触发拒绝/作废等终态，则整单按终态结束。
