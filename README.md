## 方案（平台账号 → 系统 → 模块 → 流程，低代码一体化）

### 目标形态（你要的产品模型）
- **平台账号**：用户先注册平台账号登录平台；不依赖是否已创建任何系统/模块。
- **系统（低代码创建）**：平台内用低代码建立“系统”（类似飞书的应用/钉钉的应用空间）。系统内部有自己的组织、用户、角色、权限、模块与数据。
- **系统用户复用平台账号**：系统内的“用户”本质来自平台账号，但系统可为用户附加系统专属扩展信息（可扩展字段）。
- **多租户开关**：每个系统可选择是否开启多租户（关闭=单租户；开启=同一系统下支持多个租户/空间隔离）。
- **模块（低代码创建）**：系统内继续用低代码创建各个模块（业务对象/表单/列表/规则/动作/自定义能力）。
- **流程（独立引擎）**：流程独立，可被模块配置使用，也可通过接口接入外部系统使用；流程本身可配置且能处理异常策略。

### 领域拆分（建议的服务边界）
- **platform（平台层）**
  - 平台账号注册/登录、会话、平台控制台
  - 平台级资源：系统列表、系统创建、进入系统（选择系统 + 选择租户上下文）
- **system（系统层：组织与权限）**
  - 组织：部门/成员/上下级
  - 权限：角色、成员-角色、多部门、多数据权限
  - 成员扩展：给“系统成员”加可配置字段（不污染平台账号）
  - 多租户开关：租户/空间管理（按系统维度开启/关闭）
- **lowcode（模块与元数据层）**
  - 模块树、字段、表单布局、列表列配置、字典、关联关系
  - 标准动作(Action)配置 + 执行引擎
  - 自定义能力扩展点（插件/脚本/自定义接口）
- **data（数据引擎层）**
  - 通用 CRUD、查询 DSL（SearchBO）、权限过滤（数据域/字段域）
  - 导入导出/打印等通用能力的实现
- **flow（流程/审批引擎，独立部署也可内嵌）**
  - 流程定义、节点（普通/条件/抄送/转交）、实例流转、异常策略（超时、找不到审批人等）
  - 对外接入协议（外部业务系统也能走审批）

### 平台账号 vs 系统成员（关键：既复用账号，又能扩展）
- **平台账号（PlatformAccount）**：只负责登录身份与平台级资料（手机号/邮箱/密码/头像等），尽量稳定。
- **系统成员（SystemMember）**：平台账号加入某系统（以及某租户）后形成成员身份：
  - **成员基础字段**：部门、岗位、上下级、入职状态等（系统内）。
  - **成员扩展字段**：系统可通过低代码给成员加“自定义字段”，例如：工号规则、证件信息、业务标签、区域、成本中心等。
  - **一人多部门**：支持主部门 + 兼职部门（并允许部门维度的数据权限策略）。
  - **一人多角色**：支持系统角色与模块角色（或统一角色但可按范围生效）。
  - **多数据权限**：同一成员可同时拥有多条数据域规则（按模块/部门/条件生效）。

### 系统多租户开关（按系统维度）
- **关闭多租户**：系统只有一个默认租户（所有模块数据在同一空间内，权限仍生效）。
- **开启多租户**：系统内部存在多个租户/空间：
  - 组织与成员通常按“租户”隔离（也可配置共享组织，取决于产品策略）
  - 模块元数据可选择：全租户共享 / 每租户独立（建议默认共享，降低维护成本）
  - 数据强隔离：所有数据查询必须带租户上下文（后端硬过滤）

### 模块（低代码）能力设计
#### 模块元数据
- **字段(Field)**：字段类型、校验、默认值、选项/字典、是否可搜索/排序、是否扩展字段等
- **表单(FormLayout)**：布局、分组、联动、显示/隐藏规则、只读规则
- **列表(ListView/Header)**：列配置、排序、筛选器、常用视图
- **关系(Relation)**：关联模块（主从、引用、lookup）、级联规则

#### 标准动作（Action）= 系统默认规则能力的统一抽象
- **标准动作集合**（可配置启用/禁用/排序/权限/流程挂载）：
  - LIST / DETAIL / ADD / EDIT / DELETE
  - IMPORT / EXPORT / PRINT
  - TRANSFER（转移：owner/负责人/部门归属/团队）  
  - TRANSFORM（转化：字段映射，把模块A数据转成模块B数据）
  - LOCK / UNLOCK
  - TEAM（团队成员：协作人、只读、可编辑等）
- **ActionPolicy（策略）**：
  - 权限要求（角色/成员/条件）
  - 字段级可写范围（哪些字段可编辑、哪些必填）
  - 数据域约束（可操作的数据范围）
  - 是否需要走流程（关联某流程定义 + 触发条件）
  - 审计与通知（谁、何时、改了什么）

#### 自定义能力扩展（不是只做 CRUD）
- **自定义动作**：新增自定义 Action（UI + 后端执行器），并可复用标准能力（校验/权限/流程/审计）。
- **自定义字段类型**：在标准字段类型之上扩展（例如地址组件、签章、地理围栏等）。
- **自定义校验/规则**：以规则表达式/脚本/DSL 形式配置（建议可灰度/可禁用/可回滚）。
- **自定义集成**：Webhook / 回调 / API Connector（对接外部系统、同步数据）。

### 数据引擎（模块数据存储与查询策略）
#### 主记录（Record）常用类型列（性能优化）
为避免“所有字段都落到 `record_data` 导致查询慢”，主记录表建议预留一批**常用类型列**作为热点字段承载层，例如：
- **数字列**：`number1..number5`
- **小数列**：`decimal1..decimail5`
- **文本列**：`text1..text10`
- **日期列**：`date1..date5`
- **枚举列**：`option1..option5`（存选项 code）
- **用户/部门列**：`user1..user2`、`dept1..dept2`（存 id）

使用规则（重要）：
- 字段元数据里要能配置“该字段映射到主表哪个槽位列”，并做**唯一性校验**（一个槽位同一模块只能映射一个字段）。
- 只有“高频筛选/排序/统计”的字段才占槽位；其余走扩展表。
- 槽位列需要建索引（按模块常用查询组合设计），避免全表扫。

#### `record_data`：一字段一行 vs 单 JSON（推荐一字段一行）
两种实现都能做，但从“可查询性 + 后续治理”角度，推荐 **一字段一行（EAV 行存储）**：
- **推荐：一字段一行**：
  - 优点：天然支持按字段做筛选、索引扩展（可加冗余 typed value 列）、做字段级审计/回滚也更直观
  - 适用：字段经常被查询、后续要做统计/清洗/权限控制
- **不推荐：整条记录一个 JSON**：
  - 缺点：查询与索引困难、字段权限/审计粒度差、后续字段迁移成本高
  - 适用：仅当“几乎不查询、只展示”的大字段块（如复杂配置）才考虑 JSON

折中建议（最常见可落地方案）：
- 主表放“槽位列”承载热点字段
- 扩展表按字段一行承载长尾字段
- 超高频报表再做物化/ES/聚合表（后期）

### 附件（上传/下载/预览）做成通用基础能力
附件建议独立为平台/系统的基础服务，模块与流程都复用：
- **能力范围**：上传、下载、私有鉴权、临时签名 URL、预览、批次/目录、引用计数、删除回收站（可选）
- **关联方式**：业务侧只保存 `file_id` 或 `batch_id`，不要把文件元信息复制到每张业务表
- **权限**：下载必须经过后端鉴权（基于系统/租户/模块/记录权限），再生成临时可访问链接

### 流程（flow）设计：独立、可配置、可对外接入
#### 流程本体能力
- **流程定义**：可视化配置、版本化（发布/停用/历史可追溯）
- **节点类型**：普通审批、条件分支、抄送、转交/加签、开始/结束
- **审批人类型**：指定人员、发起人自选、上级、角色、部门等
- **多人审批策略**：或签、会签、依次审批
- **异常策略**：
  - 超时策略（自动通过/转他人/提醒升级）
  - 找不到审批人（按上级层数、转管理员、兜底角色）
  - 撤回/驳回后的重新流转规则

#### 模块内接入（配置驱动）
- 模块的 ActionPolicy 可配置：某动作在满足条件时触发审批流程
- 条件审批可读取模块字段（含自定义字段）作为条件输入

#### 审批中允许编辑模块数据（模块接入流程的增强能力）
当“模块动作(Action)接入审批流程”时，支持在配置审批流程/节点时，声明该节点的审批人是否拥有**临时编辑该模块数据**的能力：
- **配置点**：在流程节点配置中增加 `edit_policy`（或同等配置），定义：
  - 允许编辑的字段范围（字段级白名单/黑名单；区分可读/可写）
  - 允许编辑的时机（仅当前待办审批人、或同节点所有候选人；是否允许抄送人编辑）
  - 是否允许修改会影响条件分支的字段（建议默认不允许，避免路径不一致）
- **数据一致性（推荐）**：
  - 审批中的编辑不直接写入最终业务数据，而是写入**变更单(ChangeRequest)的 after 快照**或“审批草稿层”
  - 审批通过后一次性落库；审批驳回/撤回则保留审计但不落库
- **冲突处理**：
  - 若发起人/其他动作在审批中也产生新变更：按版本号/时间戳做并发控制，必要时要求重新发起审批
- **审计要求**：
  - 记录“谁在审批中改了哪些字段、改前改后值、发生在哪个节点、原因/备注”，并可在流程轨迹中展示

#### 外部系统接入（协议驱动）
- 流程提供一套稳定的接入协议（HTTP API），外部系统只需传入规范参数即可发起/查询/审批：
  - **发起**：流程key/版本、业务标识(bizType/bizId)、发起人、表单数据快照、候选人上下文
  - **查询**：实例状态、当前节点、待办列表、历史轨迹
  - **审批**：同意/拒绝/转交/加签/撤回等
- 流程引擎只保证“流程正确流转 + 产出结果”，业务系统通过回调/事件接收“通过/拒绝/撤回”结果并自行落库。

#### 运行中动态插入节点（业务系统接入后的“临时加节点”能力）
业务系统接入审批后，在**审批未走完**的情况下，允许有权限的人对“运行中的流程实例”做临时调整：
- **支持的调整类型**：
  - **插入审批节点**：在某个节点前/后插入“临时审批人/审批组”
  - **插入条件节点**：插入条件分支（用于补充判断逻辑）
  - **插入抄送节点**：追加抄送（邮件/人员/角色/部门等）
- **权限控制（必须后端强约束）**：
  - 默认仅允许：系统管理员 / 流程管理员 / 该实例的发起人（可配置是否允许）/ 指定角色
  - 必须同时满足“数据域权限”（能看到该业务）+ “流程操作权限”（能改该实例）
  - 所有修改必须写审计（谁在何时插入了什么，原因是什么）
- **插入位置与一致性规则**：
  - 仅允许插入在“当前节点之后”的路径上（避免回滚已完成节点引发一致性问题）
  - 若插入影响到“已生成但未处理”的待办任务：需要定义策略
    - 策略A：撤销原待办，按新链路重新生成待办
    - 策略B：保留原待办，但将新节点作为后续追加（更简单但表达能力弱）
  - 条件节点插入必须绑定“可计算的上下文输入”（来自模块字段/表单快照/外部参数），并记录当时的输入快照，保证可追溯
- **推荐实现方式（兼容可审计与可回滚）**：
  - 将“临时节点/临时规则”作为 **instance_patch（实例补丁）** 记录，不直接篡改原流程定义
  - 引擎在运行时对“定义 + 补丁”做合成，生成实际执行链路
  - 支持补丁撤销（撤销后对未执行部分生效，已执行部分只做审计记录）

### 数据一致性（审批与数据变更如何不打架）
- **推荐模式：变更单(ChangeRequest)**：模块动作先生成变更单（保存变更前/后快照），审批通过后再真正写入模块数据。
- **替代模式：业务状态机**：直接写入但标记“审批中”，对外只展示“已生效版本”，审批通过再切换版本（实现复杂但查询更快）。

### 最小可用迭代路线（按优先级）
- **阶段 A**：平台账号体系 + 创建系统 + 进入系统（上下文：systemId + tenantId 可选）
- **阶段 B**：系统组织（多部门/多角色/多数据权限）+ 权限后端硬约束
- **阶段 C**：模块低代码（字段/表单/列表）+ 标准动作(Action)落地（CRUD/导入导出/转移/锁定/团队）
- **阶段 D**：流程引擎独立化 + 模块内接入 + 外部接入协议 + 异常策略完善

### 当前工程需要尽快修正的点（避免后期踩坑）
- 父 `pom.xml` 已统一为 **Java 21**（`maven-compiler-plugin` 使用 `release=21`）；若本地构建仍异常，检查 JDK 版本与 IDE 配置。
