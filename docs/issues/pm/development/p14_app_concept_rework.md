# P14 完整系统基线重整

## PM 裁决

P13 验收结论撤回。当前系统可以跑通登录、创建系统、系统内业务应用、模块、字段、发布和运行台新增记录，但这条链路不能证明这是一个普通人可以连续使用的完整系统。

从 P7 到 P14 多次出现“局部通过、整体割裂”的问题，说明不能继续用碎片化分期补丁推进。P14 升级为完整系统基线重整，基线文件为 `docs/product/integrated-system-baseline.md`。

本轮先由 PM/Analyst/UIUX 冻结完整系统目标、术语、信息架构和连续验收剧本，再决定前端和后端返工范围。

## 术语裁决草案

| 术语 | 层级 | 含义 | 当前问题 |
| --- | --- | --- | --- |
| 平台级对外应用 | 平台层/集成层 | 面向外部系统的应用接入主体，维护 appKey/secret、scope、IP 白名单、限流、调用日志，用于把系统内数据、模块动作、流程或文件能力授权给外部。 | 当前前端没有作为平台级核心入口呈现，容易被 OpenAPI 客户端页面弱化。 |
| 系统内业务应用 | 自定义系统层 | 用于组织系统内模块、字段、页面、菜单和运行台入口。 | 当前 UI 把它简称为“应用”，和平台级对外应用混淆。 |
| 模块 | 自定义系统层 | 业务数据模型和运行台记录的承载对象。 | 当前链路已跑通，但页面需要在术语纠偏后重新命名和定位。 |
| OpenAPI 客户端 | 集成技术对象 | 平台级对外应用下的凭证、授权和调用策略实体。 | 当前页面偏技术化，缺少“对外应用如何把数据开放出去”的产品流程。 |

## 原型流程追踪要求

P14 必须补齐以下流程图和页面追踪矩阵，并纳入 `docs/product/integrated-system-baseline.md`：

1. 平台管理员创建平台级对外应用，生成并只展示一次密钥。
2. 为对外应用选择可访问的系统、租户、模块、字段、操作和数据范围。
3. 外部应用调用系统内记录、文件或流程能力。
4. 系统管理员和平台管理员查看调用日志、错误、限流和审计。
5. 系统内业务应用继续作为模块配置和运行台组织入口，但页面文案不得再与平台级对外应用混用。
6. 普通用户从登录、我的系统、系统总览、建模配置、发布、运行台、出错恢复的连续使用剧本必须有页面和 E2E 证据。
7. 普通业务用户必须单独验证：只能看到自己所在系统、授权模块和自己数据范围内的数据，不得看到管理配置入口。
8. 平台级对外应用必须单独验证：可授权业务系统数据/能力，也可授权平台级流程能力，外部调用和日志追踪闭环。
9. 日志必须单独验证：平台日志和业务系统日志分层展示、分层权限、分层检索。
