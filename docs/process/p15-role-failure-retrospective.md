# P15 多角色失效复盘

## 结论

本轮问题不是“角色数量不够”，而是角色没有围绕同一个真实用户剧本工作。角色看似齐全，但 PM、UI/UX、frontend、test、reviewer 的验收目标发生了偏移。

## 失效点

| 角色 | 应该做到 | 实际偏差 | P15 修正 |
| --- | --- | --- | --- |
| PM | 定义普通人怎么用，并用它裁决范围 | 接受了功能点、接口和构建证据，没有推翻部署体验问题 | 先冻结 `docs/product/p15-project-rebaseline.md`，所有任务按用户剧本验收 |
| UI/UX | 输出页面级布局、主次按钮、状态和业务文案 | 页面仍像接口和配置表单堆叠 | 重新设计登录、系统首页、成员、建模、运行台、对外应用、日志 |
| frontend | 把设计变成可部署业务页面 | 仍按 API 模块堆表格、表单和按钮 | 按任务流重组页面，不再让用户理解技术 ID |
| backend | 把底层模型包装成业务动作 | 平台账号和系统成员割裂暴露给用户 | 支持“开通系统成员账号并加入系统” |
| test | 以陌生普通用户跑完整页面 | 管理员或局部链路通过被放大成完整可用 | 必须用 `che` 账号跑车系统普通用户剧本 |
| reviewer | 主动推翻 PM 结论 | 过度相信已有 E2E 和文档状态 | 部署反馈优先于内部 pass，发现矛盾必须 fail |
| validator | 验证构建和包 | 构建通过被误读为可用 | 只声明构建事实，不替 PM 宣称可用 |

## 根因

1. 任务拆分以模块/API 为中心，不是以“普通人完成一件事”为中心。
2. E2E 以证明已有实现能跑为目标，不是以发现用户卡点为目标。
3. 进度文档没有在用户反馈后强制撤回 accepted。
4. UI/UX 的输出没有具体到每个关键页面的新建入口、分页、退出和普通用户默认落点。
5. PM 没有把“车系统/che 普通用户”这种真实样例作为验收主剧本。

## P15 硬规则

1. 任何角色不能再用“接口存在”“页面可点”“构建通过”证明系统可用。
2. P15 的验收对象是一个真实业务例子：车系统、车辆模块、che 普通成员。
3. 普通成员视角失败时，整期失败。
4. 页面解释不了下一步时，UI/UX 失败。
5. 退出、分页、新建入口、权限隔离缺任一项时，frontend 失败。
6. 平台账号/系统成员模型仍需要用户手工跨页面拼接时，backend/PM 失败。
