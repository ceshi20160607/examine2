# unexamine 需求理解分析

## 原始需求摘要

原始需求要求将 unexamine 重构为一个“可配置业务系统平台”，面向普通企业团队长期使用，而不是低代码演示项目或接口调试后台。产品目标是让不写代码的管理员能够创建系统、创建应用、配置模块字段、生成页面、配置菜单权限、配置审批流程、邀请用户使用，并让普通用户在运行台完成登录、业务数据新增编辑、查询筛选、详情查看、附件上传、审批处理、消息查看和数据导出。

需求将产品分为两个层级和五个区域：

- 平台层：管理平台账号、平台角色、平台菜单、系统、租户、全局授权、全局 OpenAPI 策略、文件存储、日志、健康检查、版本和部署。
- 自定义系统层：管理单个业务系统内部的用户、部门、角色、权限、应用、模块、字段、页面、菜单、字典、流程、打印模板、KPI、仪表盘、模块数据、附件、导入导出、系统内 OpenAPI 和审计日志。
- 平台中心：登录、注册、我的系统、系统管理、租户、平台账号角色、平台消息和日志。
- 应用配置中心：系统基础信息、应用管理、模块建模、字段设计、页面设计、菜单配置、权限配置、流程配置、导出配置、OpenAPI 配置。
- 应用运行台：业务菜单、列表、表单、详情、关联数据、子表、附件、评论或备注、历史、流程状态、导出和仪表盘。
- 流程工作台：待办、抄送、我的申请、实例详情、审批处理、流程图、流程模板、流程设计器和发布。
- 系统管理和运维中心：配置检查、数据库/Redis/文件存储检查、日志、操作审计、OpenAPI 调用日志、导出任务、异常消息、版本和部署信息。

需求强调重构不是照搬旧代码，而是用旧系统确认业务范围、表和接口积累，再用新的产品结构、页面交互和后端接口重新组织能力。

## 产品边界

### 必须纳入的产品边界

- 平台与自定义系统必须分层：平台层负责全局系统、账号、授权、运维；自定义系统层负责单个系统的配置和业务运行。
- 所有业务数据必须包含并遵守 `systemId`、`tenantId`、`appId`、`moduleId` 或等价模型上下文。平台数据和业务数据分表、分域管理。
- 自定义系统之间必须数据隔离；多租户系统还要按租户隔离；普通用户不能跨系统、跨租户访问数据。
- 配置态和运行态必须分开：配置中心负责建模、页面、流程、权限、模板；运行台只呈现业务菜单和业务操作。
- 动态模块必须支持字段、页面、列表、表单、详情、关联模块、子表、附件、历史、导入导出、流程触发和权限控制。
- 权限必须覆盖平台权限、系统内权限、菜单权限、按钮/操作权限、字段可见/可编辑权限、数据范围权限、OpenAPI 授权。
- 配置类能力需要考虑草稿、发布、版本和回滚，至少在 PRD 中明确哪些能力 MVP 实现发布机制，哪些先保留扩展字段。
- 运行类数据需要保存快照或历史，避免配置修改后影响历史记录、审批实例、打印结果和审计解释。
- OpenAPI 必须与内部接口分开，支持 AK/SK 或签名、调用日志、幂等键、限流、IP 白名单、授权范围和错误码。
- 文件上传必须与业务记录、字段值、导出任务建立引用关系，支持权限校验、类型和大小限制、预览/下载、安全路径、后续对象存储扩展。

### 不应纳入或需要延后的边界

- 不做重型 ERP、MES 行业套件。采购、销售、库存台账、生产工单、质检记录、设备巡检等只作为可配置轻量场景，由模块、字段、流程、规则、KPI、打印模板和仪表盘组合实现。
- 不在 MVP 引入复杂脚本引擎。业务规则建议使用结构化条件配置，避免自由脚本带来的权限、安全和运维风险。
- 不把智能助手作为核心闭环的前置条件。智能生成字段、流程、错误解释、审批总结等可以作为后续增强，生成内容必须人工确认并受权限控制。
- 不以旧项目的调试式 CRUD 控制器作为新产品接口。旧 `queryById/add/update/queryPageList/deleteByIds` 形态只可参考基础 CRUD，不可作为对外契约。
- 不在运行时依赖“自动补表/修复 schema”完成正式迁移。旧项目的兼容修复只可作为旧库排障参考，新项目应以清晰 migration 和初始化 SQL 为准。
- 不把后台管理页面做成接口测试页。前端应围绕配置向导、设计器、运行台和工作台重新设计。

## MVP、后续增强与暂不做

### MVP 必做

- 账号登录：注册、登录、退出、刷新 token、当前用户、账号停用、登录日志、基本失败限制。
- 平台中心：我的系统、创建系统、启用/停用系统、系统成员、平台账号、平台角色、平台菜单、平台操作日志。
- 系统上下文：进入系统、系统基础信息、单租户/多租户上下文、默认应用初始化、默认角色和菜单权限初始化。
- 应用和模块：应用创建/编辑/启停、模块创建/编辑/停用、字段设计、字段类型元数据、默认列表/表单/详情配置。
- 字段基础：文本、数字、金额、日期、日期时间、单选、多选、开关、人员、部门、附件、图片、自动编号、关联数据、子表、地址、标签、JSON；高级公式/汇总可先预留。
- 动态数据运行：列表查询、服务端分页/筛选/排序、表单新增编辑、详情展示、软删除、变更历史、附件字段引用校验。
- 权限：系统成员、部门、角色、菜单权限、操作权限、数据范围、运行态菜单过滤，字段权限至少要在 PRD 中拆出 MVP 做法和后续字段级增强。
- 流程：流程模板、版本、设计器基础图、发布检查、发起审批、待办、抄送、我的申请、同意、拒绝、撤回、终止、审批历史、流程图当前节点。
- 导入导出：MVP 至少支持导出模板、导出任务、任务状态、结果文件下载；导入可以先做预演/任务模型设计，按阶段实现。
- 上传文件：本地存储、文件表、上传、列表、详情、预览、下载、软删除、扩展对象存储配置。
- OpenAPI：平台 OpenAPI 客户端、凭证轮换、签名或 AK/SK、业务记录接口、流程接口、幂等、调用日志。
- 运维基础：健康检查、配置检查、Flyway/migration 状态、请求日志、错误日志、操作审计、版本信息。

### 后续增强

- 应用模板、应用复制、应用配置导入导出。
- 页面设计器拖拽、复杂联动、保存视图、虚拟滚动、高级表格列拖拽。
- 公式字段、汇总字段、计算字段、引用字段、状态字段。
- 打印模板可视化设计、套打、PDF 下载、模板版本和历史打印。
- KPI 目标闭环、指标看板、未达标提醒。
- 自定义仪表盘组件、组件缓存、点击跳转规则、个人仪表盘。
- 导入预演、导入批次、导入回滚、行级错误明细。
- 流程模拟、子流程完整编排、异常节点策略可视化。
- 命令中心、全局搜索、最近访问、收藏菜单、快捷创建。
- 智能助手：字段草稿、流程草稿、错误解释、审批历史总结、导出字段建议。
- 多环境管理、备份恢复、灰度和功能开关、容量配额、数据脱敏、移动端完整适配。

### 暂不做

- 不做通用脚本执行平台。
- 不做行业级 ERP/MES 深度业务包。
- 不做复杂 BI 报表平台，先做模块统计、仪表盘和 KPI 基础能力。
- 不做独立消息队列架构，需求明确 event 采用本地接口承载通知和自动化扩展。
- 不把平台管理员设计成可无视业务规则直接修改所有业务数据的超级入口，除非 PRD 明确运维授权和审计边界。

## 用户角色

- 平台管理员：管理全局系统、租户、平台账号、平台角色、授权、OpenAPI 全局策略、文件存储、健康检查和平台日志。
- 系统拥有者：创建并拥有某个自定义系统，进入系统后管理应用、模块、用户、部门、角色、权限、流程、模板、仪表盘和运行设置。
- 系统管理员：在特定自定义系统内管理系统配置、成员、角色权限、字典、日志和功能开关，不能管理平台全局。
- 应用管理员：管理某个应用内的模块、字段、页面、菜单、流程绑定、导入导出和仪表盘。
- 普通用户：通过系统直登入口进入运行台，只看到有权限的菜单，进行业务数据新增、编辑、查询、导出、附件和评论操作。
- 审批人：在流程工作台处理待办，查看业务数据、流程图、审批历史，执行同意、拒绝、转交、领取、取消领取等动作。
- 运维人员：查看健康检查、配置状态、日志、导出任务、OpenAPI 调用日志、版本和部署信息。
- 外部系统：通过 OpenAPI 访问授权范围内的记录和流程能力，受签名、幂等、限流、IP 白名单和数据范围限制。

## 业务模块

- 平台中心：账号、系统、租户、平台 RBAC、平台消息、平台日志、全局配置、健康检查。
- 系统管理中心：系统基础信息、登录入口、参数、功能开关、系统内成员、部门、角色、权限、消息、日志。
- 应用配置中心：应用、模块、字段、字典、页面、菜单、列表视图、表单、详情、关联关系、子表。
- 应用运行台：业务列表、业务表单、业务详情、关联数据、附件、评论、历史、流程状态、导出、仪表盘。
- 流程工作台：流程模板、流程版本、设计器、发布检查、流程实例、待办、抄送、我的申请、审批日志、流程图。
- 文件中心：存储配置、文件上传、预览、下载、引用关系、软删除、大小和类型控制。
- 导入导出中心：导出模板、导出任务、文件结果、导入预演、导入批次、失败明细和回滚。
- OpenAPI 中心：客户端、凭证、权限范围、IP 白名单、签名、幂等、限流、调用日志。
- 仪表盘/KPI：组件配置、数据源、统计方式、筛选条件、目标值、达成率、预警。
- 运维审计：请求日志、操作日志、错误日志、后台任务日志、数据库脚本版本、系统体检、部署版本。

## 核心场景

- 创建系统闭环：平台用户注册/登录 -> 创建系统 -> 初始化默认应用、默认角色、默认菜单权限 -> 邀请成员 -> 成员进入系统。
- 搭建业务应用：系统拥有者创建应用 -> 创建模块 -> 配置字段和字典 -> 生成列表/表单/详情 -> 配置菜单 -> 配置角色权限 -> 发布给用户。
- 业务数据填报：普通用户进入运行台 -> 打开有权限菜单 -> 新增记录 -> 上传附件或选择关联数据 -> 保存草稿或提交 -> 列表和详情展示。
- 审批闭环：管理员配置流程模板和发布版本 -> 模块绑定流程 -> 用户提交记录触发审批 -> 审批人处理待办 -> 业务状态联动 -> 记录审批历史。
- 导出闭环：管理员配置导出模板 -> 用户按权限筛选列表 -> 创建导出任务 -> 后台生成文件 -> 文件中心可下载 -> 审计记录导出行为。
- OpenAPI 接入：平台或系统管理员创建客户端 -> 生成和轮换凭证 -> 配置授权范围和白名单 -> 外部系统签名调用 -> 后端幂等处理 -> 调用日志可追踪。
- 运维体检：运维进入健康页面 -> 检查数据库、Redis、文件存储、密钥、脚本版本、最近错误 -> 定位 requestId 和日志链路。

## 关键流程

### 平台初始化到业务运行

1. 平台管理员初始化全局配置、密钥、数据库迁移、Redis、文件存储和默认平台角色。
2. 平台用户注册或由管理员创建账号，登录后进入“我的系统”。
3. 用户创建自定义系统，系统记录 owner、系统编码、租户模式、状态和默认首页。
4. 系统创建后初始化默认应用、默认系统管理员角色、默认菜单/API 权限和成员关系。
5. 系统拥有者进入应用配置中心，配置应用、模块、字段、页面、菜单、角色、流程、导出模板。
6. 配置发布后普通用户进入运行台，按角色权限操作业务数据。
7. 业务数据触发流程、附件、导出、OpenAPI、日志和审计。

### 动态模块建模到运行

1. 创建应用，配置应用编码、名称、图标、状态。
2. 创建模块，配置模块编码、名称、生命周期状态。
3. 创建字段，校验编码唯一、类型合法、必填、默认值、字典、关联模块、子表配置、自动编号配置。
4. 生成或配置列表页、表单页、详情页、菜单项和权限点。
5. 发布配置，运行端只读取已发布版本。
6. 用户新增或编辑记录时，后端按字段定义校验必填、类型、唯一性、附件引用、关联记录和权限。
7. 保存主记录、字段值、历史和触发事件；如绑定流程则按规则发起审批。

### 审批流程

1. 管理员创建流程模板草稿，配置节点、连线、条件、审批人、抄送、异常策略和业务状态联动。
2. 发布前执行完整性检查：必须有开始节点、可办理审批节点、结束路径、合法审批人、无孤立节点、条件兜底。
3. 业务提交时创建流程实例，保存模板版本快照、业务数据快照或变量、首个待办。
4. 审批人打开待办，同时看到业务详情、流程图、审批历史和操作区。
5. 同意推进节点，拒绝、撤回、终止必须填写原因并改变实例和业务状态。
6. 实例结束后写审批日志、业务状态、消息通知和审计日志。

### 文件和导出

1. 文件上传前校验系统上下文、租户上下文、文件大小、扩展名、MIME 类型和存储配置。
2. 上传成功后写文件元数据，业务字段只保存 fileId 或文件引用数组，详情展示时补充文件名称、预览和下载地址。
3. 导出任务创建时保存筛选条件、模板、操作者、系统租户上下文和任务状态。
4. 后台任务生成文件，写入文件表，更新任务结果；失败要记录错误原因和可重试边界。
5. 下载时按系统、租户、数据权限和文件状态校验。

## 需求边界与不确定项

- 字段类型的 MVP 清单需要冻结：需求列出大量字段类型和高级字段，PRD 需要明确首期哪些真正落地，哪些只预留配置。
- 页面发布机制需要细化：列表、表单、详情、仪表盘、字段显示配置是否都做草稿/发布/回滚，MVP 是否允许直接发布。
- 字段级权限需要明确粒度：可见、可编辑、导出明文、OpenAPI 返回、详情解密是否同一套规则。
- 数据范围权限需要明确范围：本人、本人及下属、部门、部门树、全部之外，是否支持自定义条件。
- 多租户模式需要明确创建路径：平台租户、系统内租户、用户加入租户、租户切换、单租户隐藏策略。
- 业务规则配置需要限定表达能力：条件显示、条件必填、状态下禁止编辑、金额超限走审批、删除限制等需要结构化规则定义。
- 流程和业务状态联动需要明确状态来源：模块字段中的状态字段、系统流程状态、业务草稿/审批中/通过/拒绝/撤回之间的映射。
- 导入能力需要阶段边界：原始需求要求预演、批次、回滚、重复检查，MVP 是否只做导出或同步做导入需要确认。
- 打印模板和 PDF 能力较重，PRD 需要拆为模板管理、版本、设计器、预览、套打、导出 PDF 的阶段。
- 智能助手涉及模型接入、脱敏、权限和人工确认，建议作为增强项，不影响 MVP 骨架。
- 移动端要求包含扫码或拍照上传，但本阶段是否交付移动端应用、H5 适配还是仅预留 API 需要确认。
- 备份恢复、多环境、部署回滚、容量配额、数据脱敏属于成熟上线能力，PRD 需要区分基础配置与完整运维能力。

## 旧项目可复用能力

| 可复用能力 | 可支撑的新项目场景 | 旧实现参考 | 需要规避的问题 |
| --- | --- | --- | --- |

## 旧项目需要重构或规避的问题

- 页面和产品形态缺失：旧目录未发现独立前端源码或页面文件，后端接口大量是生成式 CRUD，无法支撑现代配置中心和运行台体验。
- 中文编码异常：多处 POM 描述、Java 注释、Swagger 文案、错误提示和 `service_info.md` 显示为乱码，后续文档和代码必须统一 UTF-8。
- 配置态和运行态混杂：生成式 Controller、系统态 Controller、OpenAPI Controller 并存，路径和命名需要按平台/系统/运行/OpenAPI 重新收敛。
- 旧 Controller 直接暴露 PO 或 Java record 作为入参出参，缺少 SaveBO/UpdateBO/VO 分层，不适合冻结 API 契约。
- 错误码不成体系，存在 `new BusinessException(400, "...")` 和字符串错误混用；新项目需按模块定义错误码。
- Flyway 存在手工标记版本和运行时 schema repair，说明旧库有手工 SQL 和迁移不一致风险；新项目 DBA 不能照搬这种迁移治理方式。
- 实体与 Flyway DDL 不完全一致：旧实体中存在 `un_flow_definition`、`un_flow_instance` 等表映射，但已扫描 migration 未见对应建表，DBA 需重新核对表清单。
- EAV 查询、字段值索引、关联字段展示、字段删除后历史展示、唯一性校验和大数据量性能需要重新设计。
- 数据权限当前以创建人/部门范围为主，字段权限、操作权限、OpenAPI 权限和权限解释不完整。
- 流程引擎具备基础推进能力，但流程发布检查、条件配置 UI、业务状态联动、审批原因强制、幂等并发处理需要加强。
- 文件上传本地路径和重定向逻辑可以参考，但对象存储、分片、引用关系、事务补偿和安全扫描不足。
- 导出服务已拆出文件内容生成和响应写出思路，但 xlsx 生成方式基础，复杂模板、打印、PDF 和脱敏导出需要重做。
- `examine-web` 承载较多聚合和业务控制器，新项目应让 `examine-web` 只承载启动、Web 装配、全局配置和必要聚合入口。

## 竞品/参考资料结论

原始需求没有提供具体竞品名称，只给出技术参考方向和主流官方资料方向：Vue、Vite、Spring Boot、Ant Design Vue、Element Plus、Naive UI、TanStack Table、AG Grid、Vue Flow。

对 PRD 的结论：

- 前端需要现代组件化设计，运行台和配置台风格统一但信息密度不同；不要复刻旧接口调试页。
- 表格是核心体验，列表页需要服务端分页、筛选、排序、列显示、列宽、保存视图，后续可引入虚拟滚动或专业表格库。
- 流程设计器建议基于图编辑能力规划，后端保存结构化节点、连线、条件和发布版本，不只保存不可读 JSON。
- Schema 驱动页面是核心方向，字段、表单、详情、列表、仪表盘应尽量统一 schema，以支撑 Web、移动端和配置导入导出。
- UI 组件库需要在 PRD 或前端方案阶段做选择，避免 Ant Design Vue、Element Plus、Naive UI 混用。

## 对 PRD 的输入建议

### 推荐 Maven 多模块结构

建议新后端沿用多模块方向，不做单体平铺：

- `backend/pom.xml`：父工程，统一 Spring Boot、Java 21、MyBatis-Plus、Flyway、Knife4j、测试依赖版本。
- `examine-core`：通用返回、异常、错误码、上下文、安全工具、MyBatis-Plus 配置、基础实体、缓存协调、通用工具。
- `examine-plat`：平台账号、系统、租户、平台 RBAC、平台消息、平台日志、全局配置。
- `examine-module`：应用、模块、字段、字典、页面、菜单、成员、部门、权限、动态记录、关联、导入导出、仪表盘/KPI 基础。
- `examine-flow`：流程模板、流程版本、流程设计、流程实例、任务、抄送、审批日志、流程状态联动。
- `examine-upload`：文件存储配置、上传、预览、下载、引用关系、分片扩展。
- `examine-app`：OpenAPI 客户端、凭证、签名、授权范围、调用日志、幂等、限流。
- `examine-web`：启动类、Web 配置、过滤器/拦截器、全局异常、OpenAPI 文档、少量聚合入口。

### 推荐表名前缀规则

统一 `un_` 开头，按业务模块前缀分组：

- 平台与租户：`un_plat_`，不得增加额外字母或变体前缀。
- 动态模块/模型/记录：`un_module_`。
- 流程审批：`un_flow_`。
- 上传与文件：`un_upload_`。
- OpenAPI 客户端：旧项目使用 `un_app_` 作为历史表域，新项目 DB 设计应统一迁移为 `un_openapi_`，并在 DB 设计中写清 `un_app_*` 到 `un_openapi_*` 的映射；业务应用仍归属 `un_module_app`。
- 系统日志/审计：`un_sys_` 或 `un_audit_`，如果沿用 `un_plat_oper_log` 需明确平台日志与系统内业务审计的区别。

### 代码生成落位规则

- MyBatis-Plus 代码生成器只生成贴表基础 CRUD，放在各业务模块的 `base` 包，例如 `examine-plat/.../plat/base/`、`examine-module/.../module/base/`。
- `base` 包包含 `entity`、`mapper`、`service`、`service/impl`，不生成对外 Controller。
- 业务接口、事务、权限、校验、VO/BO 转换放在 `manage` 包，例如 `manage/controller`、`manage/service`、`manage/bo`、`manage/vo`、`manage/converter`。
- `examine-web` 不堆业务实现，只做启动和聚合路由；如需要平台和系统入口 Controller，应调用对应模块 manage service。
- 旧项目的生成式 Controller 只作反例，不作为新 API 暴露方式。

### 核心业务模块关系

PRD 需要明确以下关系：

- 平台账号可以拥有或加入多个系统。
- 系统可以启用多租户；租户下有应用、成员、角色、部门和业务数据。
- 应用包含模块、菜单、页面、角色成员和运行入口。
- 模块包含字段、记录、字段值、关联关系、页面配置、列表视图、导入导出模板和流程绑定。
- 流程模板按系统/租户管理，发布版本绑定到模块或业务动作，流程实例引用业务记录并保存版本快照。
- 文件属于系统/租户，可被字段值、导出任务、评论或打印模板引用。
- OpenAPI 客户端绑定平台或具体系统/租户，授权访问模块记录、流程和文件，所有调用写日志。
- 审计日志贯穿平台操作、系统配置、业务数据、流程、文件、导入导出和 OpenAPI。

### PRD 场景拆分建议

每个场景至少写角色、入口、前置条件、主流程、异常分支、权限边界、数据落点和验收点。建议拆分：

- 平台注册登录与账号状态。
- 创建系统与默认初始化。
- 系统切换、租户切换和系统停用访问。
- 应用创建、启停、模板创建和应用复制。
- 模块创建、字段设计、页面生成、菜单发布。
- 普通用户运行台新增/编辑/删除/查询记录。
- 附件上传和业务字段引用。
- 角色授权、菜单过滤、按钮禁用、字段权限、数据范围。
- 流程设计、发布检查、发起、审批、拒绝、撤回、终止、转交、抄送。
- 导出模板、同步导出、异步导出、失败重试、结果下载。
- OpenAPI 凭证创建、签名调用、幂等重放、调用失败、限流和日志。
- 系统体检、配置缺失、脚本版本异常、文件存储不可用。

## 给后续审查角色的重点输入

### DBA 审查重点

- 表名前缀必须按 `un_plat_`、`un_module_`、`un_flow_`、`un_upload_`、`un_openapi_` 分组，平台前缀只能使用 `un_plat_`；旧 `un_app_*` 只作为 OpenAPI 历史来源和迁移映射参考。
- 动态字段 EAV 表需要 typed columns、常用筛选索引、值表归档策略和字段删除后的历史展示策略。
- 配置表要考虑草稿、发布、版本、回滚和历史快照，尤其字段、页面、流程、打印模板、仪表盘、导入导出配置。
- 自动编号必须使用序号表和行锁或数据库原子能力，不能查询最大值加一。
- 导入导出任务、OpenAPI 幂等、调用日志、文件引用、审批状态联动都需要独立表或明确字段。
- 旧项目实体与 migration 不一致的 flow 表需要重新核对，不应直接按旧实体全量生成。

### Backend 审查重点

- API 必须分平台、系统配置、运行台、流程工作台、文件、导入导出、OpenAPI，避免旧调试式 CRUD。
- Controller 入参出参不能直接暴露 PO；需要 BO/DTO/VO 和模块化错误码。
- 权限校验必须在后端执行，覆盖系统上下文、租户上下文、菜单/API、按钮、字段、数据范围、OpenAPI scope。
- 动态记录保存要在一个事务内处理主记录、字段值、历史、附件引用校验、自动编号、流程触发和事件扩展。
- 文件上传与业务保存、导出任务与结果文件之间需要补偿和失败状态，不能靠 fire-and-forget。
- OpenAPI 幂等、签名、限流、IP 白名单、调用日志要形成稳定契约。

### Frontend 审查重点

- 需要区分平台中心、应用配置中心、运行台、流程工作台、运维中心的导航和信息架构。
- 配置态页面要提供向导、空态、字段设计器、页面设计器、流程设计器，不要呈现接口字段表单。
- 运行台只显示用户有权限的菜单和动作，表格、表单、详情、附件、历史、流程状态要有完整状态。
- 权限禁用态、系统停用态、租户未选态、无应用/无模块/无字段/无流程/无权限都要有交互设计。
- 前端必须基于冻结 API 生成 typed SDK 和接口映射，不能散落调用或伪造后端没有的编辑能力。

### Test 审查重点

- 核心闭环用例应覆盖从创建系统到运行填报、审批、导出、OpenAPI 和审计。
- 权限测试必须覆盖平台管理员、系统拥有者、系统管理员、应用管理员、普通用户、审批人、外部系统。
- 数据隔离测试必须覆盖跨系统、跨租户、跨应用、跨模块、跨角色数据范围。
- 并发测试应覆盖自动编号、审批任务重复处理、OpenAPI 幂等、导出任务领取、导入批次。
- 异常测试应覆盖配置缺失、文件存储不可用、流程发布不完整、字段类型非法、权限缓存刷新、迁移失败。

- 模块：平台中心、系统管理中心、应用配置中心、应用运行台、流程工作台、文件中心、导入导出中心、OpenAPI 中心、仪表盘/KPI、运维审计。
