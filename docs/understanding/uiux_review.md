# UI/UX 可用性审查：部署后普通用户最小链路

审查日期：2026-06-11

审查范围：

- `docs/ui/ui-design.md`
- `frontend/src/App.ts`
- `frontend/src/styles.css`

审查目标：仅针对用户反馈“部署后的 `http://192.168.0.211:19999/` 普通人根本没法使用”做可用性审查，不修改代码。

## 一、结论

当前前端已经补入登录、我的系统、系统总览、应用配置、字段配置、发布、运行台等页面，但整体仍偏“接口调试台/工程工作台”。普通用户的最小链路可以被功能点覆盖，但缺少稳定的任务引导、分步验收、错误恢复、发布前置约束和运行台业务化表达。

按 `docs/ui/ui-design.md:9` 的产品结论，现状“不应作为最终用户可正常使用的产品体验”。本轮必须先修 P0，否则不能判定为普通用户可用。

## 二、P0 问题清单

### P0-01 登录失败反馈不满足普通用户恢复要求

证据：

- 登录页表单在 `frontend/src/App.ts:498-520`，错误/结果通过 `renderMessage()` 放在表单后方。
- `renderMessage()` 仅显示原始错误文本或“请求中: requestId”，没有稳定的错误标题、原因、下一步动作和复制 requestId 入口，见 `frontend/src/App.ts:4460-4467`。
- 设计要求登录错误提示显示在表单上方并包含 requestId，见 `docs/ui/ui-design.md:91-99`。

影响：

- 普通用户登录失败后不知道是账号、密码、网络、权限还是服务异常。
- 运维定位依赖 requestId，但 UI 没有“复制 requestId / 联系管理员”的可操作入口。

必须修：

- 登录页错误必须显示在账号/密码表单上方，格式为“登录失败：原因。requestId: xxx”。
- 登录按钮提交中必须禁用并显示“登录中”。
- 失败后必须保留账号输入，聚焦密码或错误字段。
- requestId 必须可复制。

### P0-02 “我的系统”缺少可恢复空态和明确进入成功反馈

证据：

- “我的系统”创建表单在 `frontend/src/App.ts:563-581`，列表直接渲染系统行，空列表时没有专门空态，见 `frontend/src/App.ts:588-605`。
- 创建按钮禁用条件为 `!createEnabled && !actions.create.enabled`，只在两个条件同时为 false 时禁用，见 `frontend/src/App.ts:580`，权限或业务任一不可用时仍可能显示可点。
- 设计要求创建后出现成功提示和进入按钮，见 `docs/ui/ui-design.md:114`。

影响：

- 新用户没有系统时看到的是创建表单和空列表结构，缺少“第一步做什么”的强提示。
- 无权限或后端禁用时按钮仍可能触发失败，普通用户会误以为系统坏了。

必须修：

- 无系统时显示独立空态：“暂无系统。请创建一个业务系统后进入配置。”
- 创建成功后必须给出明确反馈：“系统已创建，正在进入系统总览 / 进入系统总览”。
- 创建按钮只要权限不可用或创建能力不可用，必须禁用并显示禁用原因。
- 系统状态必须使用中文标签，不直接显示原始状态码。

### P0-03 进入系统后的主链路仍依赖用户猜路径

证据：

- 系统总览有推荐路径，见 `frontend/src/App.ts:764-824`。
- 但“我的系统”的右侧引导第 2 步仍指向 `#/platform/my-systems`，不是进入后的系统总览，见 `frontend/src/App.ts:2126-2129`。
- 设计要求创建系统成功后必须明确引导进入系统总览，见 `docs/ui/ui-design.md:76` 和 `docs/ui/ui-design.md:312`。

影响：

- 普通用户从“我的系统”进入后，如果上下文加载失败或未理解侧边栏变化，会在平台层和系统层之间迷路。
- “进入系统总览”的引导链接不成立，削弱关键路径。

必须修：

- 进入系统成功后默认落到系统总览，并显示当前系统、租户、成员、权限摘要。
- “我的系统”的下一步必须根据选中系统生成真实“进入系统总览”动作，不能指回本页。
- 若进入系统失败，必须说明失败原因，并给出“重试进入 / 返回我的系统 / 复制 requestId”。

### P0-04 应用配置步骤不符合冻结设计，发布检查和发布被合并

证据：

- 冻结设计要求 6 步：应用、模块、字段、页面、发布检查、发布，见 `docs/ui/ui-design.md:167-174`。
- 当前步骤条只有 4 步：应用、模块、字段、页面与发布，见 `frontend/src/App.ts:2019-2041`。
- 页面配置内同时放“加载配置、保存默认页面、保存菜单动作、发布检查、发布模块”，见 `frontend/src/App.ts:1396-1405`。

影响：

- 普通配置管理员无法判断“字段完成后是否必须先保存页面、是否必须先发布检查、检查失败怎么定位”。
- 发布前置条件不清晰，容易跳过检查或误发布。

必须修：

- 步骤条必须拆成 6 步，且每步有完成/当前/未完成/阻塞状态。
- “发布检查”必须是独立步骤，展示检查项、错误、警告和定位入口。
- “发布”必须是独立步骤，仅在检查通过后可点击。

### P0-05 发布操作缺少前置约束和用户确认

证据：

- “发布模块”按钮只按模块和权限禁用，没有按发布检查是否通过禁用，见 `frontend/src/App.ts:1404-1405`。
- 发布时使用固定发布说明“页面发布”，用户无法确认版本意图，见 `frontend/src/App.ts:3648-3659`。
- 设计要求发布检查失败时能定位问题，发布步骤填写发布说明并生成运行版本，见 `docs/ui/ui-design.md:173-180`。

影响：

- 用户可能在未检查或检查失败时尝试发布。
- 发布历史没有业务说明，后续追踪困难。

必须修：

- 未执行发布检查、检查失败、有阻塞错误时，“发布模块”必须禁用。
- 发布前必须二次确认，确认内容包含模块名、版本、发布说明、影响范围。
- 发布说明必须由用户填写，不能为空。
- 发布成功后必须给出“去运行台查看”的主按钮。

### P0-06 字段配置仍是工程输入，不支持普通用户稳定建模

证据：

- 字段表单只有字段名称、编码、类型、必填、唯一、选项和按钮，见 `frontend/src/App.ts:1348-1362`。
- 选项是单个文本输入，没有按字段类型切换专用配置区，见 `frontend/src/App.ts:1356-1359`。
- 设计要求字段要维护类型、必填、唯一、显示状态，并服务后续页面和运行台，见 `docs/ui/ui-design.md:171`。

影响：

- 普通配置管理员不知道字段编码怎么填、选项格式怎么填、哪些类型需要选项、哪个字段作为标题。
- 配置出的运行台表单容易不可读或不可用。

必须修：

- 字段编码需要自动建议，允许用户只填中文字段名。
- 字段类型切换后必须展示对应配置项，例如选项列表、日期格式、数值精度。
- 必须明确标题字段或默认标题策略。
- 字段保存后必须提示下一步：“继续添加字段 / 去页面配置”。

### P0-07 运行台仍暴露配置概念，业务用户不能直接开始填报

证据：

- 运行台首页需要手动点击“加载运行菜单”，见 `frontend/src/App.ts:1432-1454`。
- 运行模块页仍让用户选择“模块”、点击“加载运行台”、再“查询记录”，见 `frontend/src/App.ts:1468-1479`。
- 未加载 schema 时提示“发布模块后点击加载运行台”，见 `frontend/src/App.ts:1503-1508`。
- 设计要求运行台只展示已发布模块，不混入配置入口，见 `docs/ui/ui-design.md:78` 和 `docs/ui/ui-design.md:186-189`。

影响：

- 业务用户看到“模块、schema、加载运行台”等配置/技术概念，不知道如何填报业务。
- 已发布模块没有作为业务菜单自动展示，用户必须理解后台模型关系。

必须修：

- 进入运行台必须自动加载已发布模块菜单。
- 有且仅有一个已发布模块时，自动进入该模块列表。
- 文案不能出现 schema、模块 ID 等技术表述，改为业务对象和表单语言。
- 运行台空态必须给出“暂无可用业务模块，请联系管理员发布模块”。

### P0-08 运行台新建表单存在默认假数据风险

证据：

- 运行表单按 schema 字段直接渲染，见 `frontend/src/App.ts:1503-1521`。
- 默认值会自动填入 `100`、`2026-06-09`、`A`、`P10记录...` 等，见 `frontend/src/App.ts:4843-4861`。

影响：

- 普通用户可能直接提交默认假数据，污染业务数据。
- 表单没有按字段类型做明确控件、必填提示和校验反馈。

必须修：

- 运行台新建表单默认值必须为空，除非业务字段有真实默认值。
- 必填字段必须有标记和提交前校验。
- 字段类型必须使用对应控件：日期选择、数字输入、下拉/多选、开关等。
- 新建、编辑必须和列表分离成明确表单区、抽屉或独立页，不能和筛选/列表混在一个页面内。

### P0-09 操作结果反馈不可见且不可恢复

证据：

- 每个工作区固定渲染“下一步”和“操作结果”，见 `frontend/src/App.ts:533-560`。
- CSS `content-grid` 只有两列，但页面渲染三个面板，第三个“操作结果”会成为下一行面板，见 `frontend/src/styles.css:203-207`。
- `message` 只是文本容器，没有成功/失败动作区，见 `frontend/src/styles.css:912-930`。

影响：

- 主区域较长时，保存/发布/提交后的结果可能不在当前视口内。
- 错误后没有“重试、返回、复制 requestId、查看日志”的恢复动作。

必须修：

- 成功/失败反馈必须出现在当前操作区域附近，并同步进入全局消息区。
- 失败提示必须包含原因、requestId、下一步动作。
- 提交中必须禁用当前按钮，防重复点击。

## 三、P1 问题清单

### P1-01 登录页缺少基础易用性细节

证据：

- 登录页未体现默认焦点、回车提交、忘记密码/联系管理员入口，见 `frontend/src/App.ts:498-520`。
- 设计要求默认焦点在登录名、回车触发登录，见 `docs/ui/ui-design.md:98-99`。

建议：

- 默认聚焦账号输入框。
- 回车触发登录。
- 增加“忘记密码/联系管理员”次级入口。

### P1-02 页面表单密度过高，主次按钮混排

证据：

- 多数创建/配置表单使用 `.form-grid`，桌面为 `repeat(4) + auto`，见 `frontend/src/styles.css:738-747`。
- 应用、模块、字段、页面发布把多个字段和多个操作按钮放在同一网格，见 `frontend/src/App.ts:1265-1270`、`frontend/src/App.ts:1307-1313`、`frontend/src/App.ts:1396-1405`。

影响：

- 普通用户难以区分“填写内容”“切换对象”“保存”“发布”。
- 长字段、按钮和错误信息容易挤压。

建议：

- 创建/编辑使用抽屉或明确表单区。
- 每个表单区只保留一个主按钮，次要动作放在表单底部或更多操作中。

### P1-03 深层配置页面从侧边导航隐藏，缺少面包屑和返回

证据：

- 模块、字段、页面配置、运行模块被判定为 contextOnlyRoutes 并从侧边导航隐藏，见 `frontend/src/App.ts:470-473`。
- 依赖步骤条和行内按钮跳转，见 `frontend/src/App.ts:2019-2064`。

影响：

- 用户刷新页面或切换对象后，不知道自己处在应用、模块、字段、页面发布的哪一层。

建议：

- 配置页顶部增加面包屑：系统 / 应用 / 模块 / 当前步骤。
- 隐藏侧边导航可以保留，但必须有返回应用、返回模块的显式入口。

### P1-04 列表缺少筛选、分页和批量操作的可用表达

证据：

- 通用表格只渲染头、行、空态，没有分页和批量操作区，见 `frontend/src/App.ts:2066-2077`。
- 运行台设计要求列表页包含筛选区、表格、批量操作、分页，见 `docs/ui/ui-design.md:188`。

建议：

- 我的系统、应用、模块、字段、运行记录至少要有分页或“共 N 条”。
- 运行记录需要可见筛选区和重置按钮。

### P1-05 二次确认覆盖不足

证据：

- 部分状态变更使用 `window.confirm`，例如系统状态，见 `frontend/src/App.ts:2347-2350`。
- 运行台提交审批直接执行，见 `frontend/src/App.ts:3757-3769`。
- 设计要求删除、提交审批、导出等危险或批量操作必须二次确认，见 `docs/ui/ui-design.md:195`。

建议：

- 提交审批、取消导出、删除、停用等使用统一确认弹窗。
- 确认内容包含对象名、影响范围、不可逆说明。

### P1-06 响应式布局仍不适合低宽桌面和移动

证据：

- 主应用固定侧边栏 260px，见 `frontend/src/styles.css:33-36`。
- 数据行最小宽度 760px，见 `frontend/src/styles.css:764-767`。
- 移动端只是压成单列，未提供专门移动交互，见 `frontend/src/styles.css:961-1007`。
- 设计明确 P12 不做完整移动端适配，见 `docs/ui/ui-design.md:300-304`。

建议：

- 本轮验收只承诺 1366x768、1440x900、1920x1080 桌面可用。
- 小于 860px 显示“移动端暂未纳入 MVP”的清晰提示或只保证只读浏览。

### P1-07 文案仍混入技术词和接口思维

证据：

- 运行台出现“运行 schema”“模块 ID”“加载运行台”等技术表达，见 `frontend/src/App.ts:1435-1444`、`frontend/src/App.ts:1464-1479`。
- 操作成功文案展示接口 scope，如 `MOD-007 操作成功`，见 `frontend/src/App.ts:4418-4427`。

建议：

- 面向普通用户的页面隐藏 API 编号、schema、模块 ID。
- 成功文案改成业务结果，例如“模块已发布，可在运行台使用”。

## 四、本轮必须修的 UI 标准

本轮修复完成后，必须满足以下标准才能判定“普通用户最小可用”：

1. 登录：失败原因、requestId、复制入口和重试动作可见；登录中按钮禁用；登录成功进入“我的系统”。
2. 我的系统：无系统有空态和创建引导；创建成功明确进入系统总览；无权限时按钮禁用并说明原因。
3. 进入系统：顶部固定显示当前系统、租户、成员和权限摘要；进入失败有重试和 requestId。
4. 系统总览：用步骤卡明确“成员/权限 -> 应用 -> 模块 -> 字段 -> 页面 -> 发布 -> 运行台”，每步显示完成、未完成或阻塞。
5. 应用配置：应用、模块、字段、页面、发布检查、发布必须拆成 6 个可见步骤；不能把发布检查和发布合在一个按钮组里。
6. 字段配置：字段编码自动建议；字段类型驱动专用配置控件；必填、唯一、标题字段策略清晰。
7. 发布：发布检查未通过时不能发布；发布前二次确认；发布说明必填；发布成功后提供“去运行台查看”。
8. 运行台：自动加载已发布模块；业务用户不看到 schema、模块 ID、接口编号；无模块时提示联系管理员发布模块。
9. 运行填报：新建表单默认不填假数据；必填校验清晰；提交审批有二次确认和成功/失败反馈。
10. 状态反馈：所有主要页面具备加载态、空态、错误态、禁用态、权限不足态；错误包含 requestId 和恢复动作。
11. 布局：桌面 1366x768、1440x900、1920x1080 下主流程按钮和反馈不被折叠到不可见区域；移动端不纳入本轮 MVP。
12. 验收：必须用真实浏览器走通“登录 -> 我的系统 -> 进入系统 -> 创建应用 -> 创建模块 -> 配置字段 -> 页面配置 -> 发布检查 -> 发布 -> 运行台新建记录”的 smoke 路径，并记录失败截图或日志。
