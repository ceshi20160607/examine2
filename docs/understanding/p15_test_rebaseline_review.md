## 结论

P15 测试设计结论：`pass-for-test-design`。

P15-3 真实验收必须重新执行一条连续浏览器剧本：平台管理员从零创建“车”系统，开通 `che` 普通成员，创建并发布“车辆档案”模块，授权后退出管理员，使用 `che` 登录并直接进入“车系统 / 业务运行台”，完成车辆数据新增、查询、编辑，同时确认 `che` 看不到平台管理、建模、字段设计、发布和权限入口。

历史 P10/P13/P14 记录只能作为能力参考，不能替代 P15 通过证据。P14 已证明普通成员后端权限和运行台写入可以跑通，但 P15 基线明确撤回“完整系统可用”判断，因此必须用新的业务对象、普通成员账号、真实浏览器截图和失败断言重新验收。

## 浏览器 E2E 步骤

1. 启动完整本地或部署环境，使用生产构建或部署包访问前端，不使用只适合开发的 API 地址面板。记录前端 URL、后端 URL、构建版本、数据库环境。
2. 打开浏览器，登录 `platform_admin / 123123aa`，截图保存登录后首页。
3. 在“我的业务系统”创建“车”系统，编码使用 `car_{runId}` 防止重复。创建后确认进入“车”系统总览。
4. 在系统总览确认页面存在明确搭建步骤：成员与权限、建模配置、业务运行。
5. 进入“成员与权限”，用系统内表单开通 `che_{runId}` 成员账号，初始密码记录在测试报告中但不写入公开截图。
6. 创建或选择车辆普通成员角色，保存运行台相关菜单、操作、字段和数据范围授权。若页面必须手填角色 ID、菜单 ID 或字段 ID，应记录为可用性风险；若无法完成授权，P15 失败。
7. 进入“建模配置”，创建“车辆业务”应用。
8. 创建“车辆档案”模块。
9. 创建字段：车牌号、车型、颜色、状态。字段名称必须是业务中文，不允许只显示技术 ID。
10. 保存列表页、表单页、详情页，保存菜单动作，执行发布检查并发布模块。
11. 刷新业务运行台，确认管理员视角能看到“车辆档案”入口，但该证据不能替代普通成员验收。
12. 点击顶部“退出”，确认回到登录页。
13. 使用 `che_{runId}` 登录。
14. 断言登录后首屏 URL 为 `/systems/{车系统ID}/runtime` 或等价业务运行台路由，页面标题/上下文为“车”系统和“业务运行台”。
15. 断言 `che` 侧栏只显示普通业务入口。不得出现“平台账号”“平台角色”“平台配置”“平台审计”“运维诊断”“成员与权限”“建模配置”“字段设计”“页面与发布”“系统角色”等入口。
16. 点击“车辆档案”进入业务模块。若只有一个授权模块，可以接受直接进入模块页；如果先进入运行台首页，必须能清晰点击“车辆档案/进入”。
17. 填写一条真实车辆数据，例如车牌号 `沪A-{runId}`、车型 `SUV`、颜色 `蓝色`、状态 `在用`，点击“新建记录”。
18. 在列表中确认该记录出现，打开详情确认字段值正确。
19. 输入车牌号关键字查询，确认结果仍能看到该车辆，并记录分页/总数/当前页显示情况。
20. 修改车辆颜色或状态，点击“更新当前记录”，再次查询并确认更新生效。
21. 直访或通过地址栏尝试打开平台账号、平台角色、系统成员、建模、字段、页面发布等路由，确认隐藏、重定向或 403，不能出现可操作管理页面。
22. 保存至少 4 张截图：管理员创建系统/总览、系统内开通成员、`che` 登录首屏运行台、`che` 新增/查询/编辑车辆结果。测试记录写入 `docs/test_runs/p15-car-system-e2e-{date}.md`，最终汇总再由串行任务写 `docs/test_report.md`。

## API 辅助步骤

API 只允许辅助准备、定位和断言，不能替代浏览器验收。

| 辅助点 | 建议方式 | 不可替代的浏览器动作 |
| --- | --- | --- |
| 生成唯一 `runId` | 测试脚本生成时间戳，拼到系统编码、成员账号、模块编码、字段编码 | 所有关键创建动作仍优先由页面点击完成 |
| 登录和当前用户确认 | 观察 `AUTH-002`、`AUTH-001/me` 响应和 token | 必须在浏览器登录页输入账号密码 |
| 系统上下文确认 | 观察 `SYS-001` 返回 system、tenant、member、permissions | 必须验证页面上下文显示“车”系统 |
| 成员创建结果定位 | 必要时用成员列表 API 查询 `che` 的 memberId、roleIds | 开通成员必须由“成员与权限”页面完成 |
| 权限目录和动态菜单定位 | 可用 API 读取权限目录，确认车辆模块菜单和操作已同步 | 授权保存必须在页面可完成；若只能靠 API 拼装授权，P15 失败 |
| 运行台菜单断言 | 用运行台菜单 API 确认 `che` 只有车辆档案 | 浏览器侧栏和运行台首页也必须只显示车辆档案 |
| 记录增改查核对 | 用 `RUN-003/RUN-004/RUN-006` 响应和数据库/API 返回定位 recordId | 新增、查询、编辑必须由 `che` 浏览器页面触发 |
| 权限负向 | 直调成员管理、建模、平台账号等接口应返回 403 | 浏览器路由也不能显示管理入口 |
| 分页可观察性 | 查询 `PageResult.total/pageNo/pageSize/records` | 页面必须展示总数、当前页、下一页/翻页入口或明确分页状态 |

## 失败判定

出现以下任一情况，P15-3 结论必须为 `fail` 或 `blocked`，不得写 `pass`。

| 失败条件 | 判定 | target |
| --- | --- | --- |
| `che` 登录后进入平台工作台、我的业务系统、系统总览或任何管理配置页 | fail | frontend/backend |
| `che` 可以看到或点击平台账号、平台角色、平台配置、平台审计、运维诊断入口 | fail | frontend/backend |
| `che` 可以看到或点击成员与权限、系统角色、建模配置、字段设计、页面发布入口 | fail | frontend/backend |
| 系统管理员不能在系统内直接开通 `che`，必须绕到平台账号页先建账号 | fail | frontend/backend |
| 车辆模块发布后运行台不出现“车辆档案”入口 | fail | backend/frontend |
| `che` 无法在浏览器页面新增、查询、编辑一条车辆记录 | fail | frontend/backend |
| API 能增改查但浏览器页面不能完成 | fail | frontend |
| 浏览器能看到记录但后端权限负向接口未返回 403 | fail | backend |
| 页面仍大量展示 appId、moduleId、tenantId、memberId、schema、权限编码等技术信息作为主流程内容 | fail | frontend/pm |
| 列表无分页信息，无法解释 total/pageNo/pageSize 或下一页 | fail | frontend |
| 新建入口仍是调试式表单堆叠，普通用户看不出主动作 | fail | frontend/pm |
| 没有真实浏览器截图、请求记录和测试分报告 | blocked | test |
| 只复用 P13/P14 历史 pass，不重新跑“车/che/车辆档案”剧本 | fail | test |

## 不打包条件

P15-4 打包前必须满足 P15-3 全部通过。以下任一成立，不得打包：

1. `che` 登录后没有直接进入“车系统 / 业务运行台”。
2. `che` 能看到平台管理、建模、字段设计、发布或权限配置入口。
3. “成员与权限”不能直接开通并加入 `che`。
4. “车辆档案”不能通过浏览器页面完成新增、查询、编辑。
5. 列表没有分页可观察信息，或 PageResult 与页面展示不一致。
6. 普通用户主流程仍依赖测试人员解释文档才能找到下一步。
7. 仅有 API 脚本通过，没有浏览器点击流、截图和分测试记录。
8. `docs/test_runs/p15-car-system-e2e-{date}.md` 未记录环境、账号、关键 ID、截图路径、请求失败摘要和权限负向结果。
9. `docs/test_report.md` 未由串行汇总任务明确写入 P15-3 结论。

## 历史证据使用边界

- `p10-app-runtime-ui-e2e-20260609.md` 可复用运行台增改查步骤设计，但它不是普通成员隔离验收。
- `p13-usability-e2e.md` 可复用“创建系统后进入系统总览”“运行台业务化文案”检查，但它没有覆盖 `che` 普通成员首屏。
- `p14-ordinary-user-openapi-e2e-20260611.md` 可复用权限、菜单、字段、数据范围和 403 断言，但它偏 API/脚本补证，不能替代 P15 浏览器连续剧本。
- `frontend/src/App.ts` 当前存在普通用户无配置权限时进入 `/systems/{systemId}/runtime` 的路由逻辑、系统内开通成员页面、运行台菜单和记录增改查页面，P15 执行时必须用真实浏览器验证这些逻辑在部署环境成立。
