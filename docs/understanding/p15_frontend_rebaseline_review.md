# P15 frontend 可用性重整审阅

审阅角色：frontend
审阅范围：`docs/product/p15-project-rebaseline.md`、`frontend/src/App.ts`、`frontend/src/router/index.ts`、`frontend/src/pages/*`
结论：fail。当前前端已经补了一些中文引导、系统成员开通入口和退出按钮，但整体仍偏“工程调试台”。根因是登录落点、路由守卫、导航可见性、列表分页、新建方式和技术字段展示仍以接口/模型为中心，没有以普通成员的连续业务剧本为中心。

## 一、为什么仍像工程调试台

| 关注点 | 当前证据 | 风险 | 判断 |
| --- | --- | --- | --- |
| 登录后路由 | `frontend/src/router/index.ts:75-76` 固定 `DEFAULT_AUTHENTICATED_ROUTE = "/platform/my-systems"`；`frontend/src/App.ts:5546-5549` 无 hash 时仍按该默认路由进入；`frontend/src/App.ts:2367-2376` 只有“单系统且无平台工作权限”才尝试直接进入系统，否则回到平台我的系统。 | 普通成员只要存在多系统、权限判断异常或后端误给平台权限，就仍先看到平台视角；“我的系统”同时包含创建系统表单，像平台后台入口，不是业务系统选择。 | P0 必改 |
| 路由守卫未真正接入渲染 | `frontend/src/router/index.ts:257-324` 定义了 `createRouteGuard`，但 `frontend/src/App.ts:399-405` 渲染时只判断未登录重定向，没有调用 guard 的权限/上下文 block 结果。 | 用户直接输入系统管理、建模、审计等 hash 路径时，可能绕过导航隐藏进入页面渲染；普通成员隔离不能只靠侧边栏过滤。 | P0 必改 |
| 平台/系统导航混合 | `frontend/src/App.ts:431-490` 侧边栏按 `route.meta.layout` 与 `item.section` 过滤，品牌入口始终回 `#/platform/my-systems`；系统内也显示“切换系统”回平台。 | 对普通业务用户来说，平台层仍是显性返回点；导航没有区分“业务用户运行台壳”和“管理员配置壳”。 | P0 必改 |
| 普通成员可见性 | `frontend/src/App.ts:2381-2386` 进入系统后仅按若干配置权限决定总览或运行台；侧边栏可见性依赖 permissionStore 与 section，不形成普通成员白名单壳。 | 若权限目录、菜单授权或状态 stale，普通成员可能看到成员、角色、建模、字段、发布、审计等入口；不能满足 P15 “只能看到运行台和授权模块”。 | P0 必改 |
| 退出 | `frontend/src/App.ts:493-507` 顶部已有“退出”，`frontend/src/App.ts:2399-2404` 调用 AUTH-004。 | 退出按钮本身已部分补齐，但需要确认所有认证后页面都使用同一 shell；登录页/异常页无需退出。 | P2 验证 |
| 分页与 PageResult | 多个页面模型保留 `PageResult`，例如 `platformCenter.ts:497-504`、`runtimeWorkbenchPageModel.ts:196-205`、`openApiPageModel.ts:153-176`、`auditPageModel.ts:68-81`；但通用表格 `frontend/src/App.ts:2218-2229` 只渲染表头、行和空态，没有分页控件。运行台 `frontend/src/App.ts:1571-1574` 用 `records.length` 展示“列表记录”，不是 `total/pageNo/pageSize`。 | 数据超过一页后用户无法知道总数、当前页、下一页，也无法翻页；P15 不打包条件第 5 条不满足。 | P1 必改 |
| 新建入口 | 平台系统、账号、角色、成员、业务应用、模块、字段、运行台记录、导出、OpenAPI 均把创建表单直接堆在页面主区域，例如 `App.ts:753-762`、`925-940`、`1139-1153`、`1363-1374`、`1405-1417`、`1450-1465`、`1619-1629`、`1984-2012`。 | 用户进入页面先看到大量字段和按钮，像接口录入台；新建/编辑语义混杂，无法形成“列表 -> 主按钮 -> 新建页/抽屉 -> 保存返回”的业务路径。 | P1 必改 |
| 技术 ID 暴露 | 页面直接展示或要求输入 `系统 ID`、`部门 ID`、`角色 ID`、`租户 ID`、`菜单 ID`、`moduleId/recordId/apiId/scope`，例如 `App.ts:993`、`1077`、`1149-1151`、`1194`、`1527`、`1636-1640`、`1681-1686`、`1831`、`2048-2052`、`2118-2122`。 | 普通用户和系统管理员需要理解底层 ID 才能操作，违反 P15 “技术 ID 默认隐藏，必要时放到详情/诊断”。 | P1 必改 |
| 页面组织方式 | `renderWorkspace` 固定三块：主面板、下一步、操作结果；多数 route body 仍按 API 模块堆表格和表单。 | 形式上有引导，但页面主动作仍不是用户任务；管理员搭建路径和普通成员运行路径没有被拆成不同工作台。 | P1 必改 |

## 二、前端必须改的模块清单

| 模块 | 必改内容 | 风险 |
| --- | --- | --- |
| `frontend/src/router/index.ts` 与 `authPageModel`/登录后跳转 | 取消单一默认平台落点；登录后根据后端返回的成员系统、平台权限、系统权限和菜单授权计算 landing：单系统普通成员直接进 `runtime.home`，多系统普通成员进“选择业务系统”，平台管理员进平台首页/我的业务系统。 | 不改则普通成员仍可能先看到平台后台，P15 P0 失败。 |
| `frontend/src/App.ts` 路由渲染守卫 | 在 render 前统一执行 route guard；权限拒绝显示可理解的无权限页，不能渲染目标页面；上下文缺失时跳业务系统选择，不跳工程后台。 | 不改则直接 hash 可绕过侧边栏，权限隔离不可信。 |
| 应用壳与导航 | 拆出平台管理壳、系统管理壳、普通业务运行壳；普通业务运行壳只保留业务模块、个人信息、退出、必要系统切换，不显示品牌回平台、平台审计、建模配置等管理入口。 | 不改则普通用户仍看到平台/系统管理心智，系统像后台。 |
| `my-systems` 页面 | 改成“选择业务系统/我的业务系统”任务页；对无创建权限用户隐藏创建表单，只展示可进入系统；对管理员才显示创建系统主按钮和独立创建流程。 | 不改则普通成员登录后看到创建系统表单，像平台控制台。 |
| 系统成员与权限页 | 保留“开通并加入本系统”，但部门、角色、租户必须使用可读选择器，不允许输入 ID；成员详情中的 role/dept ID 也要转为名称，技术值放诊断。 | 不改则系统管理员仍要理解底层账号/成员/角色 ID。 |
| 建模配置页 | 业务应用、模块、字段、页面发布改成向导或分步任务；创建/编辑用独立页、抽屉或清晰任务区，列表页只保留主按钮和状态。 | 不改则配置管理员仍面对 API 表单堆叠。 |
| 运行台页面 | 运行台首页只展示已授权业务模块；模块页应是业务列表 + 查询 + 分页 + “新建”主按钮；新建/编辑表单独立呈现；字段显示使用字段名称，详情默认隐藏 fieldCode/recordId。 | 不改则普通成员无法自然新增、查询、编辑真实业务数据。 |
| 通用表格/PageResult 组件 | 为所有 `PageResult` 列表提供分页条、总数、当前页、每页条数、上一页/下一页；页面 load 方法接收当前 pageNo/pageSize，不再固定 1/20 或 1/50。 | 不改则数据多时不可用，且 PageResult 契约没有闭环。 |
| OpenAPI 与审计页 | 对外应用页保留专业信息但按“创建外部接入 -> 授权 -> 安全 -> 日志”组织；scope/API/requestId 只在集成/审计角色可见。审计页保留 requestId，但不要把 API ID 当普通业务列表字段。 | 不改则平台级对外能力继续像接口调试台。 |
| 错误/空态 | 区分无系统、无模块、无权限、未发布、接口失败；普通用户用业务文案，管理员给修复入口，诊断信息折叠展示 requestId。 | 不改则用户遇到失败不知道恢复路径。 |

## 三、建议实现顺序

1. 先改登录 landing 与 route guard 接入，确保普通成员不会进入平台/建模页面。
2. 拆应用壳：平台管理壳、系统管理壳、普通运行壳，先保证导航和退出正确。
3. 重做 `my-systems` 和 `runtime` 两个普通用户主路径：选择系统、进入运行台、查看模块、列表分页、新建/编辑记录。
4. 补通用 PageResult 分页能力，并替换平台、系统成员、流程、文件、导出、OpenAPI、审计列表。
5. 整理系统成员与权限页，把部门/角色/租户 ID 输入改成名称选择和权限目录选择。
6. 重组建模配置页为管理员向导，压低表单堆叠感。
7. 最后处理 OpenAPI、审计、运维的技术信息分层，把 API ID、scope、trace、requestId 放到集成/诊断视图。

## 四、需要后端支持的点

| 后端支持点 | 需要原因 | 优先级 |
| --- | --- | --- |
| 登录或当前用户接口返回可用系统摘要与推荐 landing | 前端不能只靠 `PLAT-001` 二次猜测；需要知道用户是平台管理员、系统管理员、普通成员、多系统成员还是无系统用户。 | P0 |
| `SYS-001` 稳定返回当前成员、租户、有效菜单、操作权限、字段权限、数据范围和普通/管理入口建议 | 前端路由、导航、按钮、字段显示都依赖它；不能靠写死角色名。 | P0 |
| 所有列表接口统一返回 `PageResult.records/total/pageNo/pageSize`，必要时补 `hasNext/hasPrevious` | 前端要实现真实分页，不能对数组响应伪造分页。 | P1 |
| 成员开通接口 `MEM-002` 明确支持“创建平台账号并加入当前系统”，返回成员详情、临时密码/是否已存在账号、角色/部门/租户名称 | 系统管理员不能绕平台账号页，也不能输入 ID 才能完成团队搭建。 | P0 |
| 部门、角色、租户、菜单、字段权限目录接口返回可选择项名称、编码和禁用原因 | 前端要用选择器替代 ID 文本框，并解释权限禁用态。 | P1 |
| `RUN-001` 只返回当前成员被授权的业务模块和菜单，包含业务显示名、模块状态、可执行动作 | 普通运行台不能自己过滤出权限边界。 | P0 |
| `RUN-003` 记录列表返回业务标题、状态显示名、更新时间、可用动作和分页信息 | 普通成员列表不能 fallback 到 `recordId`。 | P1 |
| OpenAPI scope catalog 返回业务化标签和层级，调用日志保留 requestId/apiId 但提供业务系统、模块、动作名称 | 集成管理员需要技术追踪，普通页面不能暴露调试字段。 | P2 |

## 五、复核条件

- 普通成员 `che` 登录后，单系统场景直接进入“车系统 / 业务运行台”，多系统场景进入“选择业务系统”，不出现平台创建系统表单。
- 直接访问 `#/platform/accounts`、`#/systems/{id}/apps`、`#/systems/{id}/modules/{moduleId}/fields` 等未授权路由时显示无权限/无上下文，不渲染页面内容。
- 普通运行台只显示授权模块，模块列表和记录列表有总数、当前页、每页条数、上一页/下一页。
- 新建业务记录由明确主按钮进入独立表单/抽屉，不与列表和查询控件堆在同一表单区。
- 普通用户页面不显示 `systemId/tenantId/memberId/appId/moduleId/recordId/apiId/scope`；管理员和审计页面如需展示，应放在详情或诊断信息中。
- 全局退出在所有认证后页面可见且能回到登录页。
