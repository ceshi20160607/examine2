# 后端需求理解审查

- 结论：fail
- 审查对象：`docs/requirement_analysis.md`，并参考 `docs/user_requirement.md`、当前 PRD 草稿、`docs/legacy_reference.md`、`docs/service_info.md`。

## 已满足项

- 模块职责方向基本清楚：已识别平台中心、系统管理、应用配置、运行台、流程、文件、导入导出、OpenAPI、运维审计等后端领域。
- 工程分层方向清楚：已明确参考旧项目 Maven 多模块，避免平铺单模块，要求 `examine-web` 只做启动和 Web 装配，业务进入各模块 `manage` 层。
- 旧项目风险识别充分：已指出旧 CRUD Controller、PO 直接暴露、错误码混乱、运行时 schema repair、`examine-web` 堆业务、流程/导出/文件能力不足等问题。

## 是否存在“页面愿望但缺少后端语义”

存在部分风险，主要集中在仪表盘/KPI、打印模板、移动端、智能助手、命令中心、复杂页面设计器等能力。需求分析已把部分能力列为后续增强，但当前 PRD 草稿仍覆盖较广。PM 需要在 PRD 中严格标注哪些只做页面入口或预留配置，哪些必须有后端表、API、任务、权限和验收标准。

## 回环建议

- analyst 不需要重扫旧项目；旧项目参考摘要已足够。
- PM 需要根据上述问题修正 PRD 草稿并冻结 MVP 边界、表前缀、状态矩阵、权限规则、事务/异步规则和 OpenAPI 安全契约。
- 在上述问题关闭前，不建议进入 API 契约冻结或后端实现阶段。
