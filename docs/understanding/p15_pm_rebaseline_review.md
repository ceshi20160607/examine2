## 1. P14/P15 之前 PM 失败原因

PM 失败不是单一页面缺陷，而是验收口径、信息架构和角色剧本同时失守。

| 失败点 | PM 判断 | 影响 |
| --- | --- | --- |
| 把功能存在误判为用户会用 | P14 证明了部分接口、页面和测试链路可运行，但没有证明普通人自然完成业务 | `fullProjectDeployable=true` 结论被部署反馈推翻 |
| 把管理员链路误判为普通用户链路 | 只用平台管理员或配置视角跑通，不能代表 `che` 这类普通成员可用 | 普通成员登录后仍可能进入平台/管理视角 |
| 把局部 E2E 拼接成完整系统验收 | API E2E、浏览器 smoke、构建通过被扩大解释为完整可用 | 主流程断点未在部署环境暴露前被 PM 阻断 |
| UI/UX 仍按接口模块组织页面 | 页面像管理后台或工程配置台，不像日常业务系统 | 用户不知道登录后第一屏要做什么 |
| 平台账号与系统成员概念暴露 | 系统管理员被迫理解底层账号模型，才能开通业务成员 | 管理员建团队流程不自然 |
| 缺少基础业务系统体验闸门 | 退出、分页、新建入口、权限隐藏、空态和错误恢复没有成为硬验收条件 | 页面能点但不像可交付系统 |
| review/progress 状态治理失效 | 文档曾出现用户反馈失败但阶段 accepted 的矛盾 | 状态文件没有及时阻断打包和最终交付 |

PM 当前撤回结论：

- P14 `fullProjectDeployable=true` 撤回。
- P14 `accepted` 不能再作为最终可用依据。
- 当前 `docs/review.json.status=fail`。
- 当前 `fullProjectDeployable=false`。
- 当前 `packageGate=blocked`。

## 2. 当前产品定位

当前产品不是单纯的平台后台，也不是接口调试管理台。

PM 冻结定位：

> 一个可配置业务系统平台：平台管理员创建业务系统；系统管理员在业务系统内搭建模块、字段、页面、权限和对外开放；普通业务用户登录后只处理自己业务系统里的数据。

用户视角边界：

| 用户画像 | 核心目标 | 登录后默认看到 | 不应看到 |
| --- | --- | --- | --- |
| 平台管理员 | 创建业务系统、维护全局账号、平台角色、对外应用和平台审计 | 平台工作空间、我的业务系统、平台账号、平台角色、对外应用、平台审计 | 某个业务系统的普通数据操作默认入口 |
| 系统管理员 | 在某个业务系统内建团队、建模块、配权限、发布业务 | 当前系统总览、成员与权限、建模配置、运行台预览、系统审计、系统对外授权 | 其他业务系统数据、平台全局配置 |
| 配置管理员 | 配置模块、字段、页面、菜单、发布 | 建模配置、发布检查、运行台预览 | 平台账号、平台角色、运维诊断 |
| 普通业务用户 | 新增、查询、编辑、提交被授权业务数据 | 自己所属系统的业务运行台和授权模块 | 平台管理、系统建模、字段设计、发布、权限配置 |
| 集成管理员 | 管理外部系统接入和授权 | 对外应用中心、密钥、安全策略、调用日志 | 普通业务数据维护入口，除非另有业务权限 |
| 审计/运维人员 | 追踪问题和风险 | 平台日志、系统日志、requestId、健康状态 | 数据编辑和业务配置入口 |

产品原则：

1. 角色是用户画像和权限模板，不是写死角色名。
2. 页面必须由真实权限、系统成员上下文、菜单授权、字段权限和数据范围驱动。
3. 平台级对外应用和系统内业务应用必须在菜单、文案、接口授权和验收中分离。
4. 技术概念可以存在于系统内部，但不能作为普通用户完成任务的前置知识。

## 3. 必须冻结的用户剧本

P15 进入任何实现修复前，以下剧本必须作为 UI/UX、frontend、backend、test、reviewer 的共同验收基线。

### 剧本 A：从零创建“车”系统并让普通成员使用

1. 平台管理员登录。
2. 在“我的业务系统”创建“车”系统。
3. 创建成功后进入“车系统首页”，页面明确展示搭建步骤。
4. 在“成员与权限”中直接开通 `che` 成员账号并加入“车”系统，不要求先去平台账号页。
5. 在“建模配置”中创建业务应用，例如“车辆业务”。
6. 创建模块，例如“车辆档案”。
7. 添加字段，例如车牌号、车型、颜色、状态。
8. 配置列表页、表单页、详情页、菜单动作并发布。
9. 给 `che` 授予“车辆档案”的菜单、查看、新增、编辑和必要字段权限。
10. 平台管理员退出。
11. `che` 登录。
12. `che` 直接进入“车系统 / 业务运行台”。
13. `che` 只能看到车辆档案模块。
14. `che` 可以新增、查询、编辑车辆数据。
15. `che` 看不到平台账号、平台角色、系统建模、字段设计、发布和权限配置入口。

### 剧本 B：把“车系统”的车辆数据开放给外部

1. 平台管理员或集成管理员进入“对外应用中心”。
2. 创建对外应用，例如“车联网外部接入”。
3. 生成 appKey/secret，secret 只展示一次。
4. 在“车系统 / 系统对外授权”中授权车辆档案模块。
5. 选择可访问字段、动作和数据范围。
6. 配置 IP 白名单、限流和过期时间。
7. 外部系统调用 OpenAPI 成功。
8. 调用日志能按 requestId 查到签名、scope、限流、系统、模块和结果。

### 剧本 C：普通用户遇到问题后的恢复

1. 登录失败时，明确区分账号不存在、密码错误、账号停用和服务异常。
2. 没有业务系统时，提示联系管理员加入系统；有权限时才给创建系统入口。
3. 没有业务模块时，提示当前系统未开放业务模块。
4. 无权限时，入口不可见或禁用，并给出可理解原因。
5. 发布失败时，指出缺字段、缺页面、缺菜单或缺权限。
6. 外部调用失败时，能按 requestId 追踪原因。

## 4. P15 分期和不打包条件

P15 不是继续追加功能，而是撤回过度验收后的全项目可用性重整。

| 子阶段 | 目标 | 退出标准 | PM 闸门 |
| --- | --- | --- | --- |
| P15-0 PM 重整 | 撤回 P14 过度验收，冻结定位、剧本、问题清单和角色裁决 | PM 能回答系统给谁用、登录第一屏是什么、普通成员怎么只处理自己系统数据、管理员怎么开通成员、怎么退出、怎么分页、在哪里新建业务数据 | 只允许写文档，不允许打包 |
| P15-1 UI/UX 重画 | 输出关键页面结构、导航、主次动作、状态和中文文案 | 登录、系统首页、成员、建模、运行台、对外应用、日志都有可执行设计和流程追踪矩阵 | 缺少冻结 UI/UX 时 frontend 不得继续真实页面实现 |
| P15-2 实现修复 | 按冻结 UI/UX 修复前端，补必要后端业务操作 | 普通成员入口、系统成员开通、退出、分页、新建入口、权限隐藏可用 | 不能按 API 模块堆页面；不能暴露技术 ID 作为主流程 |
| P15-3 真实验收 | 用部署环境或完整本地环境跑“车系统”剧本 | `che` 登录后直接使用车辆模块，且看不到平台管理、建模、字段设计、发布、权限配置 | 必须有真实浏览器记录和截图/证据 |
| P15-4 打包 | 只有 P15-3 通过后再打包 | 包内包含 P15 证据、部署说明和 review 通过结论 | 任一不打包条件成立则禁止打包 |

不打包条件：

1. 普通成员登录后仍进入平台管理视角。
2. 系统管理员仍需要绕到平台账号页面才能开通系统成员。
3. 运行台无法新增、查询、编辑一条真实业务数据。
4. 普通成员能看到平台管理、建模、字段设计、发布或权限配置入口。
5. 列表仍无分页，或无法解释总数、当前页、下一页。
6. 新建入口仍是散落在表头的工程化表单堆叠。
7. 页面主流程只能靠文档解释，普通人看页面不知道下一步。
8. 没有真实浏览器验收截图或记录。
9. UI/UX 未冻结，或冻结设计没有覆盖信息架构、主次按钮、空态、错误态、权限禁用态、中文文案和流程追踪矩阵。
10. `docs/review.json.status` 未恢复为 `pass`，或 `fullProjectDeployable` 未经 P15 剧本复验重新判定为 `true`。

## 5. PM 对其他角色的裁决

| 角色 | PM 裁决 | 必须执行 | 禁止事项 | 关闭条件 |
| --- | --- | --- | --- | --- |
| analyst | 重新从普通用户任务断点审视需求，不再只复述功能范围 | 输出“普通用户不能完成任务”的断点清单，确认平台层/系统层/运行层/对外开放层边界 | 不得把技术对象直接等同为用户任务 | 断点能映射到 UI/UX、backend、frontend、test 任务 |
| uiux | 必须重画页面级信息架构和交互设计 | 冻结 `docs/ui/ui-design.md` 或 P15 等效设计，覆盖登录、系统首页、成员、建模、运行台、对外应用、日志 | 不允许只给接口模块页面，不允许让 frontend 临场补产品设计 | 设计能解释剧本 A/B/C 的页面、路由、主按钮、状态反馈和权限禁用 |
| planner | 重新拆 P15，不沿用 P14 accepted 作为开发入口 | 按 P15-0 至 P15-4 拆任务，标明任务类型、依赖、输出路径、退出标准和不打包条件 | 不得把局部修复直接推到打包任务 | 任务计划能支撑 UI/UX、实现、测试、review 顺序闭环 |
| frontend | 按用户任务和冻结 UI 实现，不按 API 模块堆表格 | 修复普通成员登录落点、系统成员开通、退出、分页、新建入口、权限隐藏、空态错误态 | 不得暴露 API 地址配置、技术 ID、schema 摘要、调试文案作为普通用户主流程 | 真实浏览器中 `che` 只能看到授权车辆模块并可完成新增/查询/编辑 |
| backend | 将底层模型包装成可理解业务操作 | 提供或确认“开通系统成员账号并加入系统”等业务级接口/编排；保证权限上下文和数据范围真实生效 | 不得要求前端或用户理解平台账号/系统成员拆分后才能完成管理任务 | `che` 权限、菜单、字段和数据范围由后端真实约束，越权返回明确错误 |
| test | 用陌生普通用户视角跑真实页面流程 | 独立执行剧本 A/B/C，至少包含普通成员验收、平台级对外应用验收和日志追踪验收 | 不得只跑接口或管理员链路后判定完整可用 | 测试报告明确 UI、权限、数据、日志、失败恢复均通过 |
| validator | 只验证构建和产物，不替代可用性判断 | 验证前后端 clean build/package、产物路径、nginx `/api/` 前缀保留转发 | 不得用 build pass 推导 fullProjectDeployable | 构建通过且明确“可用性以 test/reviewer/P15 剧本为准” |
| reviewer | 必须主动推翻 PM 结论 | 对照部署体验、普通成员剧本、对外应用、日志分层、不打包条件逐项审查 | 不得因文档写了 pass 或构建通过就放行 | 无任何 P15 不打包条件，且 `review.json` 可合法表达 pass |
| PM | 重新承担产品判断责任 | 维护定位、剧本、闸门、角色裁决和验收口径；在证据矛盾时撤回 accepted | 不得再用局部 E2E、构建通过或单角色 pass 宣布完整系统可用 | P15-3 真实验收通过后，才允许进入 P15-4 打包 |

## 6. 当前 PM 最终判断

1. 当前项目后端可视为具备试部署基础，但这不等同于完整系统可用。
2. 当前前端已有可部署工程和若干页面能力，但部署反馈已证明普通用户可用性不足，不能继续打最终包。
3. 当前数据库和接口链路不是主要阻塞点，主要阻塞在产品入口、信息架构、普通用户剧本、成员开通和页面任务化表达。
4. P15 必须先冻结 PM 重整结论和 UI/UX 设计，再进入实现修复。
5. 在 P15-3 真实验收通过前，`fullProjectDeployable=false` 和 `packageGate=blocked` 必须保持。
