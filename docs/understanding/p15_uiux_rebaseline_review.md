# P15 UI/UX 可用性重整审查

状态：`review`

审查角色：`uiux`

审查范围：

- `docs/product/p15-project-rebaseline.md`
- `docs/product/integrated-system-baseline.md`
- `frontend/src/App.ts`
- `frontend/src/styles.css`

本次只做页面级 UI/UX 梳理，不修改前端代码。

## 一、结论

当前前端已经开始向 P15 基线靠拢：顶部有退出入口，登录后有单系统普通用户进入运行台的分流逻辑，系统总览能区分管理者和普通使用者，系统成员页提供“开通并加入本系统”，运行台、对外应用、日志也加入了业务化说明。

但现状仍不能判定为“普通业务用户可自然使用”。主要问题不是缺单个按钮，而是页面组织方式仍以“表单 + 表格 + 行按钮 + 结果面板”为主，很多页面看起来仍像接口调试台或后台配置台。P15 前端返工应按用户任务重组页面，而不是继续在现有表格上补字段。

UI/UX 结论：`blocked(rebaseline-required)`。必须完成以下页面级重整后，frontend 才能进入实现验收。

## 二、核心用户与默认入口

### 2.1 普通业务用户

目标：登录后只处理自己有权限的业务系统、业务模块和数据。

默认入口规则：

1. 只属于一个业务系统且没有平台管理权限：登录后直接进入该系统的“业务运行台”。
2. 属于多个业务系统：进入“选择业务系统”，页面标题使用“选择业务系统”，不能叫“平台后台”。
3. 没有业务系统：显示“你还没有加入任何业务系统，请联系系统管理员开通成员账号”，如果有创建系统权限，再提供“创建业务系统”。
4. 进入系统后：只显示“业务运行台、授权模块、个人信息、退出”。不得显示平台账号、平台角色、建模配置、字段设计、权限配置、发布入口。

### 2.2 系统管理员

目标：在当前业务系统内开通成员、搭建模块、发布页面、授权普通成员使用。

默认入口规则：

1. 登录后进入“我的业务系统”或最近使用的“系统总览”。
2. 系统总览必须展示搭建进度和下一步，而不是统计数字堆叠。
3. 系统管理员应能在“成员与权限”内直接开通登录账号并加入当前系统，不需要绕到平台账号页。

### 2.3 平台/集成/审计用户

平台级能力只放在平台层：

- 我的业务系统
- 平台账号与角色
- 平台级对外应用
- 平台审计
- 运维诊断

系统级能力只放在系统层：

- 系统总览
- 成员与权限
- 建模配置
- 业务运行台
- 系统对外授权
- 系统审计

“业务应用”和“对外应用”必须持续分离，不能在菜单、页面标题、按钮或空态里只写“应用”。

## 三、信息架构建议

### 3.1 平台层导航

一级导航：

- 我的业务系统：默认平台入口，支持创建、进入、切换业务系统。
- 平台管理：平台账号、平台角色、平台配置。
- 对外应用中心：外部接入应用、密钥、授权、安全策略、调用日志。
- 平台审计：平台账号、业务系统、对外应用、运维事件。
- 运维诊断：健康检查、版本、迁移、配置诊断。

平台层不展示任何具体业务模块的数据操作入口。

### 3.2 系统层导航

一级导航：

- 系统总览：显示当前系统可用性、搭建进度和下一步。
- 成员与权限：成员开通、部门、角色、菜单/操作/字段/数据范围授权。
- 建模配置：业务应用、业务模块、字段、页面、菜单动作、发布检查。
- 业务运行台：普通业务用户主工作区，只展示已发布且已授权模块。
- 协同与文件：流程、文件、导出。
- 系统对外授权：把当前系统数据和能力授权给平台级对外应用。
- 系统审计：系统内配置、数据、流程、文件、OpenAPI 调用追踪。

普通业务用户在系统层只看到“业务运行台”和必要的个人/退出入口。

## 四、页面级重整建议

### 4.1 登录页

现状问题：

- 标题仍是“登录平台”，普通成员会误以为自己进入平台后台。
- 登录失败只依赖通用操作结果面板，不够明确地区分账号不存在、密码错误、账号停用、服务异常。

建议布局：

- 主标题：`登录业务系统平台`
- 副文案：`登录后系统会根据你的权限进入业务运行台或管理工作台。`
- 表单字段：账号、密码。
- 主按钮：`登录`
- 次按钮：`注册平台账号` 只在允许自注册时展示。
- 错误区固定在按钮下方，用中文解释原因和恢复动作。

登录后落点验收：

- 普通成员 `che` 登录后直接进入“车系统 / 业务运行台”。
- 多系统成员登录后进入“选择业务系统”。
- 平台管理员登录后进入“我的业务系统”。

### 4.2 我的业务系统 / 选择业务系统

现状问题：

- 创建表单直接铺在页面顶部，仍有后台表单感。
- 系统列表没有分页控件。
- 多系统普通用户的选择页和平台管理员的创建页需要区分主任务。

建议布局：

- 页面标题：`我的业务系统`
- 主动作：
  - 有创建权限：`创建业务系统`
  - 无创建权限：不展示创建入口
- 主区域：
  - 系统卡片或表格，字段为系统名称、我的身份、状态、最近进入时间、操作。
  - 操作为 `进入系统`，不要暴露 systemId。
- 创建入口：
  - 使用抽屉或独立任务区，字段为系统名称、系统编码、租户模式、说明。
  - 创建成功后直接进入系统总览。
- 空态：
  - 普通用户：`你还没有加入任何业务系统，请联系系统管理员开通成员账号。`
  - 管理员：`还没有业务系统，创建一个系统后即可配置成员和业务模块。`

### 4.3 系统总览

现状可保留：

- 已有管理者/普通使用者区分。
- 已有推荐路径和就绪度检查。

必须调整：

- 普通用户不应默认停留在系统总览。系统总览对普通用户只作为无模块、无权限时的解释页或只读入口。
- 管理员总览要把“成员、建模、发布、运行、对外开放”做成进度任务，不要只展示数量指标。

建议布局：

- 顶部：系统名称、当前身份、租户、退出、切换系统。
- 主任务卡：
  - `1 开通成员`
  - `2 配置业务模块`
  - `3 发布运行台`
  - `4 授权对外应用`
- 每个任务显示状态：未开始、待处理、已完成、异常。
- 右侧：最近问题，展示发布检查失败、权限缺口、最近错误 requestId。

### 4.4 成员与权限

现状问题：

- 虽然已提供“开通并加入本系统”，但字段仍暴露“部门 ID、角色 ID、租户 ID”，普通管理员不可理解。
- 角色授权仍有“操作权限编码、菜单 ID”等工程字段。
- 新建成员表单长期展开，占据页面主区域，像调试台。

建议布局：

- 页面主动作：`开通成员`
- 成员列表字段：姓名、登录账号、部门、角色、状态、最近登录、操作。
- 筛选：关键字、部门、角色、状态。
- 新建成员抽屉：
  - 基本信息：登录账号、姓名、初始密码、手机号/邮箱。
  - 加入范围：租户选择、部门选择。
  - 授权：角色多选、数据范围。
  - 完成页：展示成员登录账号和下一步“用该账号登录验证”。
- 角色授权页：
  - 使用权限树，不让管理员填写菜单 ID 或权限编码。
  - 字段权限按模块分组：可见、可编辑、必填。
  - 数据范围使用单选：本人、本部门、本部门及下级、全部、自定义。

权限态：

- 无成员开通权限：隐藏 `开通成员`，显示 `你没有开通成员权限，请联系系统管理员。`
- 角色目录未加载：显示 `权限目录尚未加载，点击“加载权限目录”后配置角色。`

### 4.5 建模配置

现状问题：

- 已有业务应用、模块、字段、页面与发布四步，但每一步仍是直接表单铺开。
- “模块编码、字段编码、菜单编码”仍占据主输入区，技术感过强。
- 发布检查结果只是表格，缺少按问题回跳修复。

建议布局：

- 使用横向或左侧步骤条：
  - 业务应用
  - 业务模块
  - 字段设计
  - 页面配置
  - 发布检查
- 业务应用页：
  - 主动作 `新建业务应用`
  - 列表字段：业务应用名称、说明、模块数、状态、操作。
- 模块页：
  - 主动作 `新建业务模块`
  - 模块编码放入“高级设置”，默认由名称生成。
- 字段设计页：
  - 左侧字段列表，中间字段属性，右侧表单预览。
  - 字段编码默认生成，允许高级编辑。
- 页面配置页：
  - Tabs：列表页、表单页、详情页、菜单动作。
  - 列表页配置列顺序、筛选字段、默认排序。
  - 表单页配置分组、必填、帮助文案。
  - 详情页配置字段展示和附件/审批/历史区。
- 发布检查页：
  - 问题按字段、页面、菜单、权限分组。
  - 每条问题提供 `去修复`。
  - 通过后主按钮 `发布到业务运行台`。

### 4.6 业务运行台

现状问题：

- 运行台模块页同时出现模块选择、打开业务表单、查询记录、新建表单、列表、详情 Tabs，任务层级混杂。
- 新建记录表单直接铺在列表上方，像调试表单。
- 记录列表没有分页。
- 列表字段没有按已发布列表页 schema 展示，仍用标题、状态、版本、更新时间等通用字段。

建议布局：

- 运行台首页：
  - 左侧或顶部为可用业务模块。
  - 主区展示最近使用模块和待处理数据。
  - 空态：`当前系统还没有给你开放业务模块，请联系系统管理员。`
- 模块列表页：
  - 标题：模块名称，例如 `车辆档案`
  - 主动作：`新建车辆档案`
  - 筛选区：由发布的列表页筛选字段生成。
  - 表格列：由发布的列表页 schema 生成，不默认展示技术 ID。
  - 行操作：查看、编辑、提交，按权限可见或禁用。
  - 分页：总数、每页条数、当前页、上一页、下一页。
- 新建/编辑：
  - 使用抽屉或独立页面。
  - 字段按表单页 schema 分组。
  - 底部固定操作区：保存、保存并提交、取消。
- 详情：
  - 使用详情页，Tabs 为详情、附件、审批、历史、关联。
  - 历史显示操作人、动作、时间、requestId。

权限态：

- 无新增权限：不展示 `新建`，或禁用并显示 `当前角色不能新增该模块数据。`
- 无字段可见权限：字段隐藏，不显示空列。
- 数据为空：`暂无车辆档案，点击“新建车辆档案”录入第一条数据。`

### 4.7 对外应用中心与系统对外授权

现状问题：

- 页面已经区分“对外应用”，但创建、授权范围、安全策略全部堆在一个表单里。
- “平台能力”仍是自由输入，不可验收。
- 对外应用中心和系统对外授权边界还需要更清晰。

建议布局：

- 平台层“对外应用中心”：
  - 列表字段：对外应用名称、负责人、状态、授权系统数、最近调用、操作。
  - 主动作：`创建对外应用`
  - 创建向导：
    1. 基本信息：名称、编码、负责人、说明。
    2. 凭证生成：展示 appKey/secret，secret 只展示一次。
    3. 安全策略：IP 白名单、限流、过期时间。
    4. 完成：提示去系统内做数据授权。
- 系统层“系统对外授权”：
  - 选择对外应用。
  - 选择可访问模块、字段、动作、数据范围。
  - 保存授权。
- 调用日志：
  - 支持按 requestId、对外应用、系统、模块、结果、时间筛选。
  - 详情展示签名结果、scope 结果、限流结果、错误原因。

### 4.8 平台日志与系统日志

现状可保留：

- 已区分平台层日志和业务系统层日志。

必须调整：

- 日志检索不能只有“日志类型、请求号、业务类型”，需要面向排障。
- 详情不能只展示快照字段，要明确失败原因和恢复建议。

建议布局：

- 筛选区：requestId、操作人、对象类型、结果、时间范围。
- 列表字段：时间、操作人、动作、对象、结果、requestId、操作。
- 详情抽屉：
  - 基础信息
  - 请求链路
  - 鉴权/权限结果
  - 变更前后
  - 失败原因
  - 建议处理动作

普通业务用户只能看到自己授权范围内的业务日志或记录历史。

## 五、全局组件与状态规范

### 5.1 列表与分页

所有 PageResult 列表必须展示分页：

- 总数：`共 128 条`
- 当前页：`第 1 / 7 页`
- 每页条数：默认 20，可选 10、20、50
- 操作：上一页、下一页
- 加载态：保留表头，行区显示骨架或 `正在加载...`
- 空态：必须说明原因和下一步

当前 `renderDataTable` 只渲染表头、行和空态，未渲染分页；所有列表页必须统一补分页组件。

### 5.2 新建入口

禁止把长期新建表单直接堆在列表上方。统一规则：

- 高频业务数据：主按钮 + 抽屉。
- 复杂配置：独立步骤页。
- 少字段配置：可用抽屉。
- 创建成功后：提示成功，并给出下一步，例如 `继续添加字段`、`去发布检查`、`进入运行台验证`。

### 5.3 权限态

按钮权限处理规则：

- 无权限且用户不需要知道该能力：隐藏。
- 用户需要理解为什么不能操作：禁用并展示原因。
- 禁用按钮必须有原因文案，不能只有 disabled。

推荐文案：

- `你没有新增该模块数据的权限。`
- `当前模块尚未发布，发布后才能进入运行台。`
- `当前角色没有字段编辑权限，只能查看。`

### 5.4 空态

空态必须区分业务原因：

- 无系统：`你还没有加入任何业务系统，请联系系统管理员开通成员账号。`
- 无模块：`当前系统还没有给你开放业务模块，请联系系统管理员发布并授权。`
- 无数据：`暂无数据，点击“新建”录入第一条记录。`
- 无权限：`你没有访问该页面的权限。`
- 未选择上下文：`请先选择业务系统。`

### 5.5 错误与成功反馈

错误提示必须包含：

- 人能理解的原因。
- requestId。
- 下一步处理建议。

示例：

- `保存失败：字段“车牌号”为必填。请补充后重试。请求号：xxx`
- `发布失败：列表页未配置展示字段。请前往“页面配置 / 列表页”修复。`

成功提示必须包含下一步：

- `成员已开通。可使用账号 che 登录验证运行台权限。`
- `模块已发布。普通成员获得授权后即可在业务运行台使用。`

### 5.6 危险操作

以下操作必须二次确认：

- 停用系统、成员、角色、对外应用。
- 删除部门、字段、字典项、文件。
- 轮换 secret。
- 发布覆盖已有运行台版本。

确认文案必须说明影响范围，例如：

`停用该成员后，成员将无法登录当前系统。已创建的数据不会删除。`

## 六、必须重做的现有页面

### P0 必须重做

1. 登录与登录后落点
   - 目标：普通成员直接进入运行台，多系统成员进入选择业务系统。
   - 当前页面文案仍偏“登录平台”，错误恢复不足。

2. 业务运行台
   - 目标：成为普通用户主工作区。
   - 当前模块页混合模块选择、查询、新建、列表、详情，且无分页。

3. 成员与权限
   - 目标：系统管理员不理解底层账号模型也能开通成员。
   - 当前仍暴露部门 ID、角色 ID、租户 ID 和权限编码。

4. 全局列表组件
   - 目标：所有列表具备分页、空态、加载态、错误态、权限态。
   - 当前 `renderDataTable` 没有分页，也没有按 PageResult 展示页码。

### P1 必须重做

1. 建模配置
   - 当前已有步骤条，但每步仍是表单堆叠。
   - 需要改为“业务应用 -> 模块 -> 字段设计 -> 页面配置 -> 发布检查”的任务流工作台。

2. 对外应用中心
   - 当前创建、授权、安全策略混在一个表单。
   - 需要拆成平台层创建凭证、系统层授权范围、日志追踪三类页面。

3. 平台日志 / 系统日志
   - 当前已有分层说明，但排障路径不足。
   - 需要按 requestId、失败原因、恢复建议组织详情。

4. 我的业务系统
   - 当前创建表单常驻页面。
   - 需要区分普通用户选择系统和管理员创建系统。

## 七、P15 UI 验收标准

frontend、test、validator、reviewer 应按以下标准判断页面是否可用：

1. 普通业务用户登录后，不看到平台管理、建模配置、字段设计、权限配置和发布入口。
2. 普通业务用户能在业务运行台完成新增、查询、编辑、详情、提交至少一条真实业务数据。
3. 系统管理员能在当前系统内开通成员账号并加入系统，不需要跳转平台账号页。
4. 所有 PageResult 列表有分页信息和翻页入口。
5. 所有新建入口是主按钮、抽屉或独立任务页，不是页面顶部长期散落表单。
6. 无系统、无模块、无数据、无权限、加载失败都有中文说明和下一步。
7. 对外应用与业务应用在导航、标题、按钮和空态文案中完全分离。
8. 平台日志和系统日志分层，日志详情能按 requestId 追踪失败原因。
9. 顶部退出在所有认证后页面可见，退出后清理认证态并返回登录页。
10. P15 “车系统”剧本必须用真实浏览器跑通，不能只用接口调用或地址栏参数证明。

## 八、给 frontend 的实现边界

可以保留：

- 现有 API client、store、权限判断、系统上下文、动态页面 schema 能力。
- 顶部退出、系统总览基础结构、运行台 schema 表单生成能力、日志分层基础能力。

必须重组：

- 页面层级、导航分组、主动作位置、列表分页、抽屉/步骤流、权限态文案、空态恢复路径。

不得继续：

- 在页面上暴露技术 ID 作为主要输入。
- 用通用表格和通用表单替代业务页面。
- 让普通业务用户通过“刷新、切换模块、打开业务表单”理解主流程。
- 用按钮 disabled 代替权限解释。
