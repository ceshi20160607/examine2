# DBA 需求理解审查

## 审查范围

- `docs/user_requirement.md`
- `docs/prd.md`（当前草稿，仅作参考）
- `docs/requirement_analysis.md`
- `docs/legacy_reference.md`
- `docs/service_info.md`

本次只执行需求理解审查，不生成 SQL，不做数据库设计。

## 审查结论

**结论：fail**

analyst 的需求理解已经覆盖主要业务域、旧项目表前缀、旧项目模块结构、EAV、流程、上传、导入导出、OpenAPI、审计、并发和迁移风险，整体方向正确；但仍有若干会直接影响 PRD 冻结和后续数据库设计的关键未决项。尤其是当前 PRD 草稿与需求分析、旧项目参考在平台表前缀上出现硬冲突，且多个运行态数据规则只停留在“需要明确”，尚不足以直接进入数据库设计。

## 通过项

- 业务模块边界基本清楚：平台中心、系统管理中心、应用配置中心、应用运行台、流程工作台、文件中心、导入导出中心、OpenAPI 中心、仪表盘/KPI、运维审计均已识别。
- 核心层级关系已识别：平台账号、系统、租户、应用、模块、字段、页面、菜单、记录、流程、文件、OpenAPI 客户端和审计日志的主线关系基本完整。
- 旧项目参考充分：已明确旧项目 Maven 多模块、`un_plat_`、`un_module_`、`un_flow_`、`un_upload_`、`un_app_` 表前缀、Flyway 组织方式，以及旧实体与 migration 不一致风险。
- EAV 风险识别充分：已指出 typed columns、常用筛选索引、唯一性、字段删除后历史展示和大数据量查询性能需要重设计。
- 并发与幂等风险已识别：自动编号不能用最大值加一，审批任务、OpenAPI、导出任务、导入批次需要幂等和并发控制。
- 迁移风险已识别：旧项目运行时 schema repair、手工标记 Flyway、flow 实体与 DDL 不一致均被标记为不能照搬。

## 必须补充或修正的问题

### 1. 表前缀存在硬冲突，必须由 PM 修正 PRD

- `docs/requirement_analysis.md` 和 `docs/legacy_reference.md` 均要求平台表前缀使用 `un_plat_`。
- 当前 `docs/prd.md` 草稿写成平台与租户表统一使用 `un_platt_`，且声明“不沿用旧项目 `un_plat_`”。
- Orchestrator/DBA 约束要求默认平台、租户、账号、角色、权限、系统上下文使用 `un_plat_`。

**必须修改：** PRD 中平台与租户前缀统一改为 `un_plat_`，不要使用 `un_platt_`。如 PM 坚持使用新变体前缀，必须先修改上游约束并说明迁移原因，否则 DBA 不能进入设计。

### 2. OpenAPI 表域仍有命名歧义，需 PM 决策

旧项目使用 `un_app_` 承载 OpenAPI 客户端、凭证、scope、白名单和调用日志；当前规范又给出 `un_openapi_`。需求分析中仍写“`un_app_` 或拆出 `un_openapi_` 由后续 DBA 决策”。

**必须明确：**

- `un_app_` 是否只保留“应用/版本/客户端应用配置”。
- OpenAPI 客户端、凭证、scope、白名单、幂等、限流、调用日志是否统一迁移到 `un_openapi_`。
- 若从旧 `un_app_` 迁移到 `un_openapi_`，PRD 需说明命名差异和迁移映射。

### 3. 租户、系统、应用、模块层级需要冻结唯一口径

当前材料同时出现“系统下有租户”“租户下有系统/应用”“系统内多租户”等表达，方向可理解，但落库时会影响外键维度、唯一索引和权限过滤。

**必须明确：**

- `system` 与 `tenant` 是一对多、可选多租户，还是租户为更上层容器。
- 单租户系统是否仍落 `tenant_id`，默认租户如何初始化。
- 应用、模块、部门、成员、角色、业务记录的唯一约束维度到底是 `system_id`、`tenant_id + system_id`，还是其它组合。
- 平台账号加入多个系统、系统内成员、租户管理员之间的关系表边界。

### 4. 配置发布、版本、快照范围需要明确 MVP 落点

需求分析已指出配置类能力需要草稿、发布、版本和回滚，但当前仍未冻结哪些表必须首期做版本快照。

**必须明确 MVP 首期范围：**

- 字段、页面、列表视图、菜单、流程模板、导出模板、打印模板、仪表盘、权限配置中哪些必须有草稿/发布/版本表。
- 运行态业务记录保存哪些配置快照：字段名称、字段类型、字典项显示值、页面版本、流程版本、打印模板版本。
- 配置发布后历史记录、审批实例、导出文件、打印结果如何继续按旧版本解释。

### 5. EAV 物理模型和索引规则需补到可设计层级

当前理解已指出 EAV 需要 typed columns 和索引，但未明确字段值存储、索引值、唯一性和筛选之间的关系。

**必须补充：**

- `record` 主表、字段值表、字段索引值表、关联关系表、子表明细记录之间的边界。
- 文本、数字、金额、日期、人员、部门、附件、关联数据、子表、JSON 各类字段的值落点和展示快照。
- 模块内唯一字段、组合唯一规则如何落库，是否支持软删除后重用。
- 列表筛选、排序、模糊搜索、范围查询依赖哪些 typed/index 字段。

### 6. 状态枚举和状态流转需要产品冻结

当前已有若干状态描述，但还不足以直接建约束和索引。

**必须明确：**

- 系统、租户、账号、应用、模块、字段、页面发布、业务记录、流程模板、流程实例、流程任务、文件、导入导出任务、OpenAPI 客户端的状态枚举。
- 各状态允许的操作，例如停用模块是否允许查看历史、是否允许导出、是否允许发起流程。
- 业务记录状态与流程状态的映射，是否使用固定系统状态字段，还是由模块字段配置。

### 7. 导入能力 MVP 边界不清，影响任务表设计

需求分析写导入可先做预演/任务模型设计，按阶段实现；原始需求又要求预演、批次、回滚、行级错误明细。

**必须由 PM 决策：**

- MVP 是否真正实现导入执行，还是只设计表结构预留。
- 若首期只做导出，是否仍创建导入任务、导入批次、导入明细、回滚记录表。
- 导入回滚与历史数据、软删除、再次修改后的冲突处理规则。

### 8. 文件引用和事务补偿规则需更明确

当前已识别文件元数据、分片、存储配置和引用关系，但业务字段、流程附件、导出结果、配置附件的引用生命周期还不够明确。

**必须明确：**

- 文件从临时到已引用的转换时机。
- 业务记录保存失败、流程提交失败、导出任务失败时文件如何补偿。
- 一个文件被多个对象引用时的删除规则和唯一约束。
- 文件权限是跟随业务对象、流程任务、上传人，还是单独授权。

### 9. 权限落库颗粒度需要收敛

材料要求菜单、操作、字段、数据范围、OpenAPI、导出脱敏权限，但 MVP 粒度仍有不确定。

**必须明确：**

- 字段权限是否首期覆盖列表可见、详情可见、表单可写、筛选可用、导出明文、OpenAPI 返回。
- 数据范围枚举和部门树路径策略。
- 权限预览和权限缓存是否首期落库。
- OpenAPI scope 与内部角色权限是独立授权还是叠加校验。

### 10. 初始化数据范围需要提前冻结

后续 `sql/init.sql` 必须包含可直接导入的初始化数据，但当前只描述了默认应用、默认角色、默认菜单权限等方向。

**必须明确：**

- 默认平台管理员账号、默认密码策略和是否允许 seed 明文初始密码。
- 默认平台角色、平台菜单、权限点。
- 系统创建时初始化哪些系统内角色、菜单、默认应用、默认模块或字段类型元数据。
- 默认字典、字段类型枚举、OpenAPI scope、文件存储配置、健康检查配置是否初始化。

## 需要 analyst 补充的问题

- 将旧项目表清单与新模块边界再整理为“可参考、需重命名、禁止沿用”三类，特别是 flow 实体与 DDL 不一致表。
- 补充旧项目中动态记录、字段值、关联关系、导出任务、OpenAPI 幂等、流程任务并发的旧表字段级参考摘要，供 DBA 后续判断是否沿用字段。
- 明确旧项目 `un_app_` 作为 OpenAPI 域的历史原因，供 PM 决策是否拆为 `un_openapi_`。

## 需要 PM 在 PRD 中修正或决策的问题

- 立即修正平台表前缀为 `un_plat_`，消除 `un_platt_` 冲突。
- 冻结系统/租户/应用/模块层级与唯一约束维度。
- 冻结 MVP 中配置版本、运行快照、字段权限、导入导出、打印模板、仪表盘/KPI 的落库边界。
- 冻结所有核心状态枚举和允许操作矩阵。
- 冻结初始化数据清单。

## DBA 后续设计风险提示

- 在上述问题未修正前，不建议开始 `docs/db_design.md` 和 `sql/init.sql`，否则很容易产生表前缀返工、唯一索引维度返工、版本快照补表、EAV 索引重建和历史数据解释困难。
- 数据库设计阶段必须以 `docs/legacy_reference.md` 的 `un_plat_`、`un_module_`、`un_flow_`、`un_upload_`、`un_app_` 参考为基础，并结合最终 PRD 决策决定是否新增 `un_openapi_`、`un_sys_`、`un_audit_`。
