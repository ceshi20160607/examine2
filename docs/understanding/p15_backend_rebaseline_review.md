# P15 后端可用性重整审阅

审阅角色：backend
审阅范围：`docs/product/p15-project-rebaseline.md`、`backend/examine-plat`、`backend/examine-module` 中认证、平台账号、系统成员、SYS-001、MEM-002、RBAC、运行台权限相关代码。
执行约束：本轮只做梳理，不修改后端代码。

## 结论

当前后端已经有“平台账号 + 系统成员 + 系统角色 + 菜单/操作/字段/数据范围权限 + 运行台记录”的底层模型，也已经有系统成员开通时自动创建平台登录账号的雏形。但它还没有完整包装成 P15 要求的普通用户业务动作。

P15 进入实现前，backend 需要补齐登录后默认业务入口、账号与成员上下文绑定、系统成员开通的业务化接口语义、运行台字段/数据权限落地，以及平台账号和系统成员的边界校验。否则前端即使重画页面，也会继续依赖技术 ID、请求头和人工路由，普通成员登录后仍无法自然进入“车系统 / 业务运行台”。

## 已具备能力

1. 认证会话已具备注册、登录、刷新、退出和当前用户查询：
   - `/api/v1/auth/login` 返回 token。
   - `/api/v1/auth/logout` 可撤销 accessToken。
   - `/api/v1/auth/me` 返回平台账号、平台角色编码、平台操作权限编码。

2. 创建业务系统时已初始化系统内基础数据：
   - `PlatformCenterServiceImpl.createSystem` 创建平台系统和默认租户。
   - `MybatisPlatformModuleInitializationWriter` 自动创建系统创建人成员、默认系统超级管理员角色、系统管理菜单、`SYS_MANAGE_ALL` 操作和默认应用。

3. 系统成员接口已有“邀请或绑定成员”雏形：
   - `SystemRbacServiceImpl.inviteMember` 支持按 `accountId` 绑定已有平台账号，或按 `loginName + initialPassword` 自动创建平台账号并加入当前系统。
   - 同一事务内创建 `un_module_member`、成员租户、部门、角色关系。

4. 系统权限模型已有基础闭环：
   - 角色权限保存支持菜单、操作、字段权限、数据范围、OpenAPI scope。
   - `ModulePermissionSnapshotProvider` 基于当前 `systemId/memberId/tenantId` 加载有效权限。
   - `DefaultPermissionService` 提供操作、字段写入、OpenAPI scope 和数据范围校验。

5. 运行台已有分页记录查询和基本增删改查：
   - `RuntimeRecordServiceImpl.queryRecords` 返回 `PageResult`，包含 `total/pageNo/pageSize/hasNext`。
   - 详情、编辑、删除、提交已调用 `requireDataScope`。
   - 创建/更新字段值时调用 `requireFieldWritable`。

## 关键缺口

| 编号 | 缺口 | 当前表现 | P15 影响 | 建议优先级 |
| --- | --- | --- | --- | --- |
| B-P15-01 | 登录后默认入口缺失 | 登录结果和 `/auth/me` 只返回平台账号与平台权限；没有可进入系统、默认系统、默认运行台、是否需要选择系统等信息 | 普通成员 `che` 登录后无法由后端直接判断应进入“车系统 / 业务运行台” | P0 |
| B-P15-02 | “我的业务系统”只查 owner | `/api/v1/platform/my-systems` 只按 `ownerAccountId` 查询系统，不包含账号作为系统成员加入的系统 | 普通成员不是系统 owner 时看不到自己所属业务系统 | P0 |
| B-P15-03 | 成员上下文依赖前端请求头续传 | `RequestContextFilter` 直接读取 `X-System-Id/X-Member-Id/X-Tenant-Id`；`/systems/{id}/enter` 只在当前请求中绑定上下文，没有写入 token/session | 后续接口必须靠前端手工带技术 ID；普通用户入口和权限判断不够业务化 | P0 |
| B-P15-04 | 请求头成员身份未绑定登录账号 | 权限快照按请求头里的 `memberId` 加载角色；多数系统接口只校验 token 有效，不校验该 member 是否属于当前 token 的账号 | 存在伪造 `X-Member-Id` 获得他人系统权限的契约和安全风险 | P0 |
| B-P15-05 | 系统成员开通接口仍偏技术化 | 接口名是 `/members/invitations`，BO 暴露 `accountId/defaultTenantId/deptIds/roleIds` 等底层 ID；返回值没有说明“新建账号/绑定账号/初始密码处理/首次改密” | UI 很难包装成“开通成员账号并加入系统”的自然动作 | P0 |
| B-P15-06 | 平台权限与系统权限边界不够硬 | 平台中心多个接口只做 `validateLogin`，未看到平台级操作权限强制校验；系统接口依赖系统权限但上下文可信度不足 | 普通成员可能仍能调用平台管理接口，前端隐藏不能代表后端隔离 | P0 |
| B-P15-07 | 运行台列表未按数据范围过滤 | `queryRecords` 只校验 `RECORD_VIEW`，没有对列表结果逐条应用数据范围或把数据范围转成查询条件 | 普通成员列表可能看到超出授权的数据 | P0 |
| B-P15-08 | 字段可见权限未作用于读取和 schema | 详情/列表直接返回所有 `RecordValue`；`moduleSchema` 返回完整字段定义和页面 schema；`recordDetail.fieldPermissions` 为空 | 普通成员可能看到未授权字段，前端也缺少可靠字段禁用/隐藏依据 | P0 |
| B-P15-09 | 动作可见性未按权限过滤 | `availableActions` 返回模块所有启用动作，未根据当前权限裁剪 | 普通成员可能看到不可执行动作，或前端需要再猜测隐藏规则 | P1 |
| B-P15-10 | 普通用户空态缺后端语义 | 没有“无系统/无模块/无权限”的后端聚合状态；只能从多个接口失败或空列表推断 | 前端难以展示 P15 要求的可理解恢复提示 | P1 |

## 需要补/改的后端能力

### 1. 登录落点和“我的可用系统”聚合能力

建议新增或调整一个后端业务聚合能力，用来回答“当前登录账号下一步去哪”：

- 返回当前账号可管理和可使用的业务系统，来源应包含系统 owner、系统成员关系、平台权限允许创建系统的情况。
- 返回每个系统下的成员身份、默认租户、是否系统超级管理员、可见运行台模块数量、可见管理菜单摘要。
- 返回推荐落点：
  - 一个系统且仅有运行台权限：直接进入该系统运行台。
  - 多个系统：进入业务系统选择页。
  - 无系统但有创建系统权限：进入创建业务系统引导。
  - 无系统且无创建权限：返回“联系管理员加入系统”的业务状态。
- 该能力不应要求前端先知道 `systemId/memberId`。

### 2. 服务端可信成员上下文

后端需要把 `Authorization` 与系统成员上下文绑定起来，不能让前端任意声明 `X-Member-Id`：

- `/systems/{systemId}/enter` 应校验 token 对应账号在系统内的成员身份，并返回可用于后续请求的可信上下文。
- 后续系统内接口应由后端根据 token + systemId 解析当前 member，或者校验请求头 memberId 确实属于 token 账号。
- 对 `X-Tenant-Id` 也要校验其属于当前成员可访问租户。
- 权限快照加载前必须完成 accountId、systemId、memberId、tenantId 的一致性校验。

### 3. 系统成员开通业务动作

现有 `inviteMember` 方向正确，但需要包装成 P15 可理解业务动作：

- API 语义建议从“邀请”调整为“开通系统成员账号/加入当前系统”，至少在契约说明中明确它会自动创建或绑定平台账号。
- 入参应支持面向用户的字段：登录名、姓名、手机号/邮箱、初始密码或重置策略、岗位、部门、角色模板、数据范围模板。
- 返回应明确：是否新建平台账号、是否绑定已有账号、成员状态、首次改密要求、默认进入系统、可分配角色和默认权限结果。
- 对已有平台账号要校验账号状态；禁用/锁定账号绑定为启用成员会造成“成员已开通但无法登录”的体验断点。
- 对重复登录名、重复成员、角色不可用、租户不可用给出可直接展示的业务错误。

### 4. 平台账号与系统成员边界

后端需要明确两条边界：

- 平台账号是全局登录身份，不等于可以看到平台管理。平台接口必须使用平台权限码强校验，不只是 token 有效。
- 系统成员是某个业务系统内的身份，不等于平台账号管理入口。系统接口必须校验账号确实是该系统成员，并按系统角色权限访问。

### 5. 运行台权限落地

运行台必须让权限从“配置存在”变成“数据真实受控”：

- 列表查询按数据范围过滤，不能先查全量再分页。
- 详情、列表、导出、历史、关联都要统一执行数据范围。
- schema、列表值、详情值按字段可见权限裁剪；编辑/新建表单按字段写权限裁剪或返回禁用原因。
- 必填字段校验需要结合字段写权限；不可写且必填的字段应由系统默认值、后端规则或配置校验解决，不能让普通用户被隐藏必填字段卡住。
- `availableActions` 应按当前操作权限和状态规则过滤，返回“可见/禁用/原因”，支持前端展示可理解反馈。

### 6. 空态和错误恢复语义

P15 需要普通人能理解“为什么不能继续”。建议后端补充统一业务状态码或聚合字段：

- `NO_ACCESSIBLE_SYSTEM`：没有可进入业务系统。
- `MULTIPLE_ACCESSIBLE_SYSTEMS`：需要选择业务系统。
- `NO_RUNTIME_MODULE`：当前系统未开放业务模块。
- `NO_RUNTIME_PERMISSION`：当前成员没有运行台权限。
- `MEMBER_DISABLED`、`ACCOUNT_DISABLED`、`TENANT_NOT_ACCESSIBLE` 等错误应可被前端稳定映射成中文提示。

## 接口契约风险

1. 如果冻结 API 仍把登录结果定义为纯 token，无法支撑 P15 的登录后自动落点。需要新增或调整契约，不能只让前端自行拼 `/auth/me + /platform/my-systems + /systems/{id}/enter`。

2. 如果契约要求前端长期传 `X-Member-Id`，必须补充后端校验规则：该 member 必须属于 token 账号、systemId 必须匹配、tenantId 必须属于该 member。否则这是权限绕过风险。

3. `/platform/my-systems` 的语义风险较高：名称像“我的业务系统”，实现却只返回 owner 系统。P15 语义需要“我可管理或可使用的系统”，建议契约拆分或重定义字段。

4. `/systems/{systemId}/members/invitations` 的契约名称和 BO 不符合 P15 业务动作。前端若要展示“开通成员账号”，契约需要说明自动创建平台账号、绑定已有账号、初始密码和首次改密规则。

5. 权限目录接口返回大量菜单 ID、操作编码、字段编码、数据范围类型，适合配置管理员，不适合普通成员。P15 需要额外的运行态权限视图，返回业务菜单、动作可见性、字段可见/可写和禁用原因。

6. 运行台 schema 和记录接口若继续返回完整字段和值，只靠前端隐藏，会违反字段权限边界。契约应明确后端已经裁剪，或至少明确哪些字段是权限后的结果。

7. 平台账号接口、平台角色接口、系统配置接口现在多处只要求登录态。契约需要明确平台级权限点，否则普通成员“看不到平台管理”只能靠前端菜单隐藏，不能通过后端验收。

## 测试建议

### P0 后端单元/集成测试

1. 登录落点测试：创建“车系统”，开通普通成员 `che`，`che` 登录后后端返回一个可直接进入的系统和运行台落点，不返回平台管理落点。

2. 多系统成员测试：同一账号加入两个系统时，后端返回“选择业务系统”状态，并列出两个业务系统的业务名称、成员身份和可见模块摘要。

3. 无系统成员测试：普通账号无成员关系时，后端返回“联系管理员加入系统”；平台管理员无系统时返回“创建业务系统”引导。

4. 成员开通测试：系统管理员在当前系统内用登录名和初始密码开通 `che`，验证同一事务内创建平台账号、系统成员、租户关系、角色关系，并且无需先调用平台账号接口。

5. 上下文防伪测试：账号 A 携带账号 B 的 `X-Member-Id` 访问系统接口必须被拒绝；账号 A 携带无权租户 ID 也必须被拒绝。

6. 平台边界测试：普通成员 token 调用平台账号、平台角色、平台系统全局管理接口必须返回 forbidden。

### P0 权限与运行台测试

1. 菜单权限测试：`che` 只授予“车辆档案”运行菜单时，运行台菜单只返回该模块，不返回建模、字段设计、发布、权限配置。

2. 字段可见测试：角色只授予车牌号可见，不授予颜色可见时，列表、详情、schema 都不能返回颜色字段和值。

3. 字段写入测试：角色对字段无写权限时，新建/编辑提交该字段返回 forbidden；不可写必填字段需要有明确配置失败或默认值策略测试。

4. 数据范围测试：角色为 SELF 时，列表、详情、编辑、删除、提交都只能访问本人创建的数据；列表分页总数也必须是过滤后的总数。

5. 动作权限测试：未授予编辑/删除/提交时，`availableActions` 不应返回可执行动作，或应返回禁用态和原因。

### P15 端到端验收脚本

建议 backend 配合 test 固化一条接口级剧本，作为前端 E2E 前置数据和验收证据：

1. 平台管理员登录。
2. 创建“车”业务系统。
3. 在该系统内开通 `che` 成员账号。
4. 创建应用、模块“车辆档案”、字段、页面和菜单。
5. 给 `che` 分配运行台菜单、查看、新增、编辑和字段权限，数据范围设为 SELF 或指定范围。
6. `che` 登录，后端返回默认进入“车系统 / 业务运行台”。
7. `che` 查询运行台菜单，只看到车辆档案。
8. `che` 新增、查询、编辑一条车辆记录。
9. `che` 调用平台账号、平台角色、建模配置、权限配置接口均被拒绝。
10. 审计日志能按 requestId 串联登录、进入系统、开通成员、授权、记录新增/编辑。

## 后端执行建议

P15 后端实现顺序建议如下：

1. 先修正认证上下文和权限防伪，保证任何系统接口都不能由前端伪造 memberId。
2. 再补登录后落点/我的可用系统聚合接口，支撑普通成员默认进入运行台。
3. 然后把成员开通接口业务化，保证系统管理员不用绕平台账号页。
4. 最后补运行台字段、数据范围、动作权限裁剪，并补 P15 接口级剧本测试。

这些变更中，登录返回结构、我的系统语义、成员开通接口和运行态权限结果都可能触发 API 契约调整，需要先交 PM/API 冻结流程确认，不能在实现时私自改冻结契约。
