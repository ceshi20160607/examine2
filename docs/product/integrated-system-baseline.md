# 完整系统基线重整

状态：`P14 active`

## PM 结论

P7 到 P14 的反复问题说明：继续按零散页面、零散接口、零散验收往前补，会得到“每块都像完成了，合起来却不像一个系统”的结果。后续必须先按普通用户视角冻结一个完整系统基线，再让 PM、UI/UX、frontend、backend、test、validator、reviewer 统一围绕这条基线工作。

在本基线通过前：

- 不再打新的最终可用包。
- 不再把某个页面、某条接口、某段 E2E 通过称为完整系统可用。
- 不再让用户通过反复指出“哪里不能用”来驱动 PM 发现主流程问题。

## 一句话产品目标

普通人登录系统后，不需要懂接口、不需要懂技术 ID、不需要看文档，也能完成：

1. 创建一个业务系统。
2. 在系统里配置可用的业务模块和字段。
3. 发布页面和菜单。
4. 使用运行台新增、查询、编辑和提交业务数据。
5. 配置平台级对外应用，把系统内数据或能力按权限开放给外部。
6. 在日志和审计里看到谁做了什么、哪里失败、怎么恢复。

## 三条不可偏离的主线

### 1. 业务系统内的日常使用

业务系统必须先能被普通业务用户正常使用，而不是只给管理员配置。

普通业务用户登录后：

1. 只能看到自己加入的业务系统。
2. 进入某个业务系统后，只能看到自己有菜单权限的业务模块。
3. 在业务模块里，只能看到自己或自己数据范围内的记录。
4. 可以基于已发布页面新增、查询、编辑、查看详情、提交审批和导出自己有权限的数据。
5. 看不到字段设计、页面配置、发布检查、权限配置等管理员入口，除非被授予配置权限。
6. 无权限、无模块、无数据时必须看到中文说明和下一步，不是空白页或技术错误。

系统管理员或配置管理员在同一个业务系统内：

1. 可以维护业务模块、字段、页面、菜单、功能按钮和管理配置。
2. 可以控制哪些角色看到哪些菜单、功能、字段和数据范围。
3. 发布后普通业务用户才能在运行台看到模块和数据入口。

### 2. 平台级对外应用

平台级对外应用不是业务系统内的“业务应用”。它是平台提供给外部系统或第三方集成方的接入配置。

平台级对外应用必须支持：

1. 创建对外应用，生成 appKey/secret，并且 secret 只展示一次。
2. 授权它可以访问哪些业务系统、租户、模块、字段、操作和数据范围。
3. 授权它可以调用哪些平台级能力，例如平台级流程、文件、消息或后续扩展能力。
4. 管理 IP 白名单、限流、过期时间、密钥轮换和停用。
5. 外部调用时按应用身份记录调用日志、鉴权结果、scope 结果、业务系统上下文和 requestId。

因此页面上必须把“业务应用”和“对外应用”分开：

- 业务应用：系统内建模块和运行台用。
- 对外应用：平台级外部接入、授权和调用用。

### 3. 平台日志与业务系统日志分层

日志和审计必须分平台层和业务系统层。

平台层日志：

1. 平台账号登录、退出、失败和锁定。
2. 平台账号、平台角色、平台配置、业务系统创建和停用。
3. 平台级对外应用创建、授权、密钥轮换、停用。
4. 平台级流程或平台级能力调用日志。
5. 全局运维、健康检查、接口文档和部署相关日志。

业务系统内日志：

1. 系统成员、部门、角色、字典等系统管理操作。
2. 业务模块、字段、页面、菜单、发布配置变更。
3. 运行台数据新增、编辑、删除、提交审批、导出。
4. 系统内流程实例、任务处理、附件和文件引用。
5. 该系统相关的 OpenAPI 调用日志和数据变更历史。

普通业务用户只能看到自己被授权的业务日志或数据历史；平台超级管理员不能默认绕过业务系统权限直接修改业务数据，但可以在审计/运维边界内查看平台级日志和必要的只读诊断信息。

## 统一术语

| 名称 | 面向谁 | 放在哪里 | 用户理解 |
| --- | --- | --- | --- |
| 业务系统 | 平台用户、系统创建人 | 平台层“我的系统” | 我要搭建的一个独立业务空间。 |
| 系统内业务应用 | 系统管理员、配置管理员 | 系统层“建模配置” | 用来组织模块和运行入口的业务分组，不是对外开放应用。 |
| 业务模块 | 配置管理员、业务用户 | 系统层“建模配置”和“运行台” | 一类业务数据，例如客户、订单、折扣规则。 |
| 运行台 | 普通业务用户 | 系统层主工作区 | 使用已发布模块处理真实业务数据。 |
| 平台级对外应用 | 平台管理员、集成管理员 | 平台层或集成中心 | 给外部系统使用的接入应用，包含 appKey、secret、授权、限流和日志。 |
| OpenAPI 客户端 | 集成管理员、开发者 | 对外应用详情内 | 对外应用的技术凭证和调用策略。 |

页面文案必须避免只写“应用”。需要明确写“业务应用”或“对外应用”。

## 信息架构

平台层：

- 我的系统：创建系统、进入系统、查看自己可管理或可使用的系统。
- 平台管理：平台账号、平台角色、平台配置、平台审计。
- 对外应用：创建外部接入应用、管理密钥、授权可访问业务系统/模块数据/平台级流程能力、查看调用日志。

系统层：

- 系统总览：告诉用户下一步做什么，显示系统设置、建模、运行、集成状态。
- 系统设置：租户、成员、部门、角色、字典。
- 建模配置：业务应用、模块、字段、页面、菜单、功能按钮、管理配置、发布检查、发布。
- 业务运行：普通业务用户可见的已发布模块数据列表、表单、详情、附件、审批、历史。
- 协同与文件：流程、文件、导出。
- 系统审计：系统内操作日志、数据变更、OpenAPI 调用追踪。

## 完整用户剧本

### 剧本 A：创建并使用一个业务系统

1. 平台管理员登录。
2. 进入“我的系统”，点击“创建业务系统”。
3. 创建成功后进入系统总览。
4. 系统总览显示“系统设置、建模配置、业务运行、对外开放”的完成度和下一步。
5. 系统管理员确认默认租户、成员和角色可用。
6. 进入“建模配置”，创建业务应用或使用默认业务应用。
7. 创建业务模块，例如“客户”或“折扣规则”。
8. 添加字段，例如名称、类型、折扣比例、生效时间。
9. 配置列表、表单、详情和菜单。
10. 执行发布检查，失败时能定位到具体字段或页面问题。
11. 发布成功后进入运行台。
12. 普通业务用户在运行台新增、查询、编辑、提交数据。
13. 页面给出中文成功、失败、无权限和恢复提示。
14. 普通业务用户重新登录后，只看到自己所在系统、授权模块和自己数据范围内的数据。

### 剧本 B：把系统数据或能力对外开放

1. 平台管理员进入“对外应用”。
2. 创建对外应用，填写名称、编码、联系人和说明。
3. 系统生成 appKey/secret，secret 只展示一次。
4. 管理员选择该对外应用可访问的系统、租户、模块、字段、动作和数据范围。
5. 管理员选择该对外应用可调用的平台级流程或平台级能力。
6. 管理员配置 IP 白名单、限流和过期时间。
7. 外部系统使用 appKey/secret 调用 OpenAPI。
8. 调用成功后，系统返回业务结果和 requestId。
9. 调用失败时，页面能在调用日志里按 requestId 查到失败原因、签名结果、scope 结果和限流结果。
10. 管理员可停用对外应用或轮换 secret。

### 剧本 C：普通人遇到问题后的恢复

1. 登录失败：显示账号、密码、停用、网络或服务异常的可理解提示。
2. 没有系统：引导创建系统或联系管理员加入系统。
3. 没有权限：按钮禁用并说明原因，不让用户点击后才看到技术错误。
4. 没有模块：提示先发布模块，并提供去建模配置的入口。
5. 发布检查失败：定位到字段、页面、菜单或权限问题。
6. 外部调用失败：能按 requestId 查日志，看到签名、授权、限流或数据权限原因。

## 页面验收矩阵

| 页面 | 必须回答的问题 | 不合格表现 |
| --- | --- | --- |
| 登录 | 我是谁，失败怎么处理？ | 只显示错误码或接口返回。 |
| 我的系统 | 我能进入哪个系统，没有系统怎么办？ | 空表格，没有引导。 |
| 系统总览 | 下一步该做什么？ | 只堆接口状态或统计数字。 |
| 系统设置 | 谁能使用这个系统？ | 成员、角色、权限割裂，创建后无法使用。 |
| 建模配置 | 怎么把业务对象建成可运行模块，并配置页面、菜单、功能和字段？ | 让用户填技术 ID、schema 或 current 路由。 |
| 运行台 | 普通业务用户怎么只看自己系统、模块和数据范围内的真实数据？ | 配置按钮混入运行台，表单默认测试数据，或看到无权限模块。 |
| 对外应用 | 怎么把系统数据/能力和平台级流程开放给外部？ | 只展示 OpenAPI 技术字段，没有授权流程。 |
| 平台日志 | 平台账号、系统、对外应用、平台级流程出了什么问题？ | 平台日志和业务日志混在一起。 |
| 系统日志 | 某个业务系统内谁改了配置、数据、流程和文件？ | 只有列表，无 requestId、授权结果、失败原因。 |

## 一次过验收门槛

PM 只能在以下全部成立后声明“完整系统可用”：

1. UI/UX 已按本基线重画页面和流程。
2. 前端页面按“平台层”和“系统层”分清入口。
3. “业务应用”和“对外应用”文案、路由、页面、权限完全分离。
4. 本地或部署环境完成剧本 A、B、C 的真实浏览器 E2E。
5. 普通业务用户账号复测通过：只看到自己所在系统、授权模块和自己数据范围内的数据。
6. 平台级对外应用复测通过：可授权业务系统数据/能力和平台级流程能力，外部调用与日志追踪闭环。
7. 平台日志与业务系统日志分层复测通过。
8. E2E 不使用测试默认值、不依赖地址栏调试参数、不要求用户知道接口 ID。
9. 后端 clean package 通过，前端 clean build 生成 `frontend/dist/`。
10. nginx `/api/` 转发、接口文档、静态页面刷新都通过。
11. `docs/review.json.fullProjectDeployable=true` 只能由 reviewer 在以上证据齐全后设置。

## P14 执行方式

P14 不再是“小修某个概念”的阶段，而是完整系统基线重整阶段：

1. Analyst 对照原始需求、旧项目和当前实现，确认产品概念偏差。
2. PM 冻结本基线和验收剧本。
3. UI/UX 产出 `docs/ui/p14-integrated-ui.md`，按剧本给页面线框和流程。
4. Planner 拆出一次性返工任务，不再按割裂模块验收。
5. Frontend 按 UI/UX 返工页面和导航。
6. Backend 只在发现契约或权限缺口时补，不为了页面临时造假接口。
7. Test 执行剧本 A、B、C 的连续 E2E。
8. Reviewer 对照本基线审查，任何割裂或概念混用都 fail。
