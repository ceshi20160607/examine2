# 产品愿景与运行模型

状态：`P14 active`

## PM 结论

本项目不能再按“接口有了、页面有了、测试点过了”来判断是否完成。PM 的职责不是把用户指出的问题追加进清单，而是先定义一个普通人能理解、能连续使用、能恢复问题的产品运行模型，再让 UI/UX、前端、后端、测试和审查围绕这个模型收敛。

本系统的产品定位是：一个可配置业务系统平台。它要让平台管理员创建业务系统，让系统管理员搭建模块、字段、页面、菜单和权限，让普通业务用户进入自己的系统处理真实数据，让集成管理员通过平台级对外应用把系统数据或平台能力开放给外部。

## 用户角色

本节的角色是产品使用视角，不是必须写死在数据库里的固定角色。落地时仍应通过平台角色、系统内角色、权限点、数据范围和菜单配置组合出来。

默认初始化可以提供一组便于开箱使用的角色模板，例如平台超级管理员、系统管理员、建模配置人员、普通业务用户、集成管理员和审计人员。但这些模板只是初始配置，后续系统管理员应能按业务需要调整角色名称、权限点、菜单可见性、字段权限和数据范围。

验收时必须同时验证“默认角色模板可用”和“通过配置生成的自定义角色可用”。如果页面逻辑只识别写死角色名称，而不是识别权限点、成员上下文和数据范围，reviewer 必须判定失败。

| 角色 | 关注点 | 不应该看到什么 |
| --- | --- | --- |
| 平台超级管理员 | 平台账号、系统创建、平台级对外应用、平台审计、运维诊断。 | 不应绕过系统权限直接修改业务系统内真实数据。 |
| 系统创建人/系统管理员 | 系统总览、成员、角色、字典、模块、字段、页面、菜单、发布、系统审计。 | 不应被迫理解 OpenAPI 签名细节才能完成系统建模。 |
| 建模配置人员 | 把业务对象配置成可运行模块，维护字段、表单、列表、详情、按钮和发布检查。 | 不应在运行台里混用平台级对外应用、平台运维或技术调试入口。 |
| 普通业务用户 | 进入自己所属系统，看到授权模块，新增、查询、编辑、提交、导出自己的业务数据。 | 不应看到字段设计、页面发布、权限配置、租户配置等管理员入口。 |
| 集成管理员 | 创建平台级对外应用，维护 appKey/secret、授权范围、限流、IP 白名单、调用日志和排错。 | 不应把系统内业务应用误认为外部接入应用。 |
| 审计/运维人员 | 按平台层和系统层查看日志、失败原因、requestId、调用链和恢复建议。 | 不应看到无法归属层级的混合日志。 |

## 现代主流系统原则

1. 任务优先，而不是菜单优先：页面先回答“我现在能做什么、下一步是什么”，再展示列表和按钮。
2. 角色感知：同一个系统对平台管理员、系统管理员、配置人员、普通用户和集成管理员应呈现不同入口。
3. 上下文清晰：平台层、业务系统层、运行台、对外集成层必须在导航、文案和权限上分开。
4. 渐进披露：普通用户只看业务运行；管理员才看配置；集成管理员才看密钥、scope、签名、限流。
5. 权限可解释：无权限按钮要隐藏或禁用并给中文原因，不让用户点完才看到技术异常。
6. 错误可恢复：失败页面要说明原因、影响范围、下一步和可追踪编号。
7. 数据真实：验收不能依赖测试默认值、地址栏调试参数、接口 ID 或需要用户看文档才能继续。
8. 证据连续：完整可用只能由一条连续用户剧本证明，不能由多个割裂 smoke 点拼出来。

## 四个工作空间

### 平台工作空间

面向平台超级管理员和平台审计人员。

必须包含：

- 我的系统：创建业务系统、进入可管理或可使用的业务系统。
- 平台管理：平台账号、平台角色、平台配置、系统状态。
- 对外应用：创建外部接入应用、管理密钥、授权系统数据和平台能力、查看调用日志。
- 平台审计与运维：平台登录、系统创建、对外应用授权、密钥轮换、平台能力调用和部署诊断。

### 系统管理工作空间

面向系统创建人、系统管理员和建模配置人员。

必须包含：

- 系统总览：展示系统设置、建模配置、业务运行、对外开放、审计状态和下一步。
- 系统设置：成员、部门、角色、字典、数据范围。
- 建模配置：业务应用、模块、字段、页面、菜单、功能按钮、管理配置和发布检查。
- 系统审计：系统内配置、数据、流程、文件和 OpenAPI 调用日志。

### 业务运行工作空间

面向普通业务用户。

必须包含：

- 我的任务或待处理入口。
- 授权业务模块列表。
- 数据列表、查询、表单、详情、附件、提交审批、导出和历史。
- 空态、无权限、无数据、保存失败、提交失败的中文恢复提示。

### 集成开放工作空间

面向集成管理员，也可以从平台工作空间进入。

必须包含：

- 对外应用基本信息。
- appKey/secret 生成、只展示一次、轮换、停用。
- 可访问的业务系统、租户、模块、字段、动作和数据范围授权。
- 可调用的平台级流程、文件、消息或后续平台能力授权。
- IP 白名单、限流、过期时间、幂等和签名策略。
- requestId、鉴权结果、scope 结果、限流结果和业务错误追踪。

## 信息架构

平台层导航：

1. 我的系统
2. 平台管理
3. 对外应用
4. 平台审计
5. 运维诊断

业务系统层导航：

1. 系统总览
2. 系统设置
3. 建模配置
4. 业务运行
5. 协同与文件
6. 系统审计

普通业务用户进入系统时，默认进入“业务运行”或“我的任务”；只有拥有配置权限的用户才看到“系统设置”和“建模配置”。

## 端到端用户剧本

### 剧本 A：搭建并使用业务系统

平台管理员登录，创建业务系统，进入系统总览。系统管理员完成成员、角色和基础字典配置。建模配置人员创建业务模块、字段、列表、表单、详情、菜单和功能按钮，完成发布检查并发布。普通业务用户重新登录，只看到自己所属系统和授权模块，在运行台新增、查询、编辑、提交和导出业务数据，并能看到成功、失败、无权限和空态提示。

### 剧本 B：把业务系统能力对外开放

集成管理员创建平台级对外应用，生成 appKey/secret。管理员授权该应用访问指定业务系统、租户、模块、字段、动作、数据范围和平台级流程能力。外部系统调用 OpenAPI，成功时返回业务结果和 requestId，失败时能按 requestId 查到签名、授权、scope、限流或业务校验原因。

### 剧本 C：审计与恢复

平台审计人员查看平台账号、系统创建、对外应用、密钥轮换和平台能力调用日志。系统管理员查看本系统内成员、权限、模块发布、数据变更、流程、文件和系统相关 OpenAPI 调用日志。普通业务用户只能查看自己有权限的数据历史或业务操作结果。每个失败都能给出中文原因和下一步。

## 页面设计底线

不合格页面包括：

- 只显示接口字段、技术 ID、schema、current 路由或调试参数。
- 平台级对外应用和系统内业务应用都叫“应用”，用户无法判断用途。
- 普通业务用户看到管理配置、字段设计、权限配置或发布入口。
- 系统创建后进入空白运行台，不告诉用户下一步应该设置成员、建模或发布。
- 表格为空时没有中文空态和下一步。
- 错误只显示后端异常、错误码或堆栈文本。
- 平台日志和业务系统日志混在同一个无法分层的列表。

## 前端重设计策略

当前 P13 前端只能作为可运行技术底座，不能作为最终 UI 继续小修小补。P14 前端必须按产品运行模型重设计信息架构、导航、页面布局、视觉层级和交互状态。

必须重做的部分：

1. 全局导航和系统内导航：按平台工作空间、系统管理工作空间、业务运行工作空间、集成开放工作空间重新组织。
2. 首页和总览页：不能只是统计卡片和表格，要告诉用户当前状态、下一步、缺什么配置、能去哪里处理。
3. 建模配置页：按“业务对象从字段到页面再到发布”的任务流组织，而不是暴露接口字段和技术配置。
4. 业务运行页：普通用户看到的是业务模块和数据操作，不应混入配置入口。
5. 对外应用页：从“OpenAPI 技术客户端”升级为“平台级对外应用”，突出授权、密钥、scope、调用日志和排错。
6. 日志页：平台日志和系统日志分层，按用户能理解的对象、动作、结果、原因和 requestId 展示。
7. 视觉和中文文案：统一组件、按钮层级、表单密度、空态、错误态、加载态和禁用态，不保留调试页气质。

可以复用的部分：

1. 已经验证可用的 API client、类型定义、权限 store 和系统上下文处理。
2. 动态字段渲染、动态表单校验、运行台数据读写等底层能力。
3. 已通过验证的后端接口和数据库结构，除非 P14 UI/UX 发现真实业务缺口。

因此 P14 不是推倒重写全部前端，也不是在旧 UI 上修颜色；它是保留底层能力，重做面向人的产品界面。

## 开发前问题预判

PM 在 UI/UX 和开发前必须主动完成以下预判，避免开发后反复返工：

1. 角色与权限：每个入口由哪个权限点控制，普通用户和管理员分别看到什么。
2. 系统上下文：平台层和业务系统层的切换是否清楚，是否依赖真实 `SYS-001` 上下文。
3. 名词边界：系统内业务应用、平台级对外应用、模块、OpenAPI 客户端是否在页面上完全区分。
4. 业务主线：创建系统、建模、发布、运行台使用、对外开放、日志追踪是否能连续走通。
5. 空态和异常：没有系统、没有模块、没有权限、发布失败、调用失败、保存失败时用户是否知道下一步。
6. 角色模板与自定义角色：默认角色是否够开箱使用，自定义角色是否仍按权限点正常展示页面。
7. 测试数据和真实数据：E2E 是否使用真实页面创建的数据，而不是内置测试默认值。
8. 部署环境：nginx `/api/` 转发、静态资源刷新、接口文档、启动脚本是否纳入验收。

这些问题必须在 `docs/ui/p14-integrated-ui.md` 和 P14 任务拆分中逐条有落点。没有落点时，不允许进入前端实现。

## 验收方式

PM 只能在以下证据齐全时声明“普通人可用”：

1. UI/UX 已按本运行模型输出信息架构、线框、关键流程、状态和中文文案。
2. 前端页面按四个工作空间组织，不再把所有功能堆成接口调试台。
3. 浏览器 E2E 连续跑通剧本 A、B、C。
4. 至少使用两个账号验收：管理员/配置用户和普通业务用户。
5. 普通业务用户权限隔离通过，不能看到管理配置入口。
6. 平台级对外应用能完成创建、授权、调用和日志追踪。
7. 平台日志和系统日志分层通过。
8. clean build/package 通过并生成可部署前端 `dist/` 和后端包。
9. reviewer 对照本文件和 `docs/product/integrated-system-baseline.md` 判定通过。

## P14 执行要求

P14 的正确顺序是：

1. PM 冻结本产品运行模型。
2. UI/UX 基于本模型输出 `docs/ui/p14-integrated-ui.md`。
3. Planner 按剧本 A/B/C 拆分返工任务。
4. Frontend 按 UI/UX 重组导航、页面、文案和权限可见性。
5. Backend 只补真实缺口，不为了页面临时造假接口。
6. Test 执行连续浏览器 E2E 和 OpenAPI 调用链路。
7. Validator 执行 clean build/package。
8. Reviewer 对照产品运行模型审查是否真的给人使用。

本文件未冻结前，不允许进入新的前端实现；本文件与 UI/UX 未共同通过前，不允许再次打包。
