# P15 全项目可用性重整

状态：`draft`

## PM 重新判断

P14 不能继续作为“完整系统可用”结论。用户在部署环境中仍然无法理解系统怎么用，说明当前交付最多证明了部分接口、部分页面和若干测试链路可运行，没有证明普通人可以自然完成业务工作。

这不是单点缺陷，而是项目组织方式出了问题：

1. PM 把“功能存在”误判为“用户会用”。
2. UI/UX 仍然按接口模块和管理后台组织页面，没有按真实业务任务组织页面。
3. frontend 把表单、列表、按钮直接堆到页面，形成工程化配置台，不是日常业务系统。
4. test 偏向跑通指定链路，没有以陌生普通用户视角验证“登录后我该做什么”。
5. reviewer 过度相信测试证据，没有用部署环境和普通用户剧本推翻 PM 结论。
6. progress/review 文档存在“用户反馈失败但阶段 accepted”的矛盾，说明状态治理没有硬闸门。

因此 P15 不允许继续碎片化补丁。必须先重新梳理整个项目，再决定代码怎么改。

## 产品重新定位

本项目不是单纯的平台后台，也不是接口调试管理台。

本项目应定位为：

> 一个可配置业务系统平台：平台管理员可以创建业务系统；系统管理员可以搭建模块、字段、页面、权限和对外开放；普通业务用户登录后只处理自己业务系统里的数据。

普通用户不需要理解以下工程概念：

- 平台账号和系统成员的底层拆分。
- appId、moduleId、tenantId、memberId 等技术 ID。
- API 编号、接口状态、schema、权限编码。
- 前端路由、请求头、OpenAPI scope 的技术细节。

这些概念可以存在于系统内部，但页面要翻译成业务动作。

## 角色重新定义

| 角色 | 真实目标 | 登录后应该看到 | 不应该看到 |
| --- | --- | --- | --- |
| 平台超级管理员 | 管理平台、创建业务系统、维护全局账号和平台级能力 | 平台首页、业务系统、平台账号、平台角色、对外应用、平台审计、运维 | 某个业务系统的日常数据操作默认入口 |
| 系统创建人/系统管理员 | 搭建并管理某个业务系统 | 当前系统总览、成员、角色、建模、运行台、系统审计、对外授权 | 其他无关系统数据、平台全局配置 |
| 配置管理员 | 配置模块、字段、页面、菜单和发布 | 建模配置、发布检查、运行台预览 | 平台账号、平台角色、运维诊断 |
| 普通业务用户 | 新增、查询、编辑、提交业务数据 | 自己所属系统的业务运行台和授权模块 | 平台管理、系统建模、字段设计、权限配置 |
| 集成管理员 | 管理外部系统接入 | 对外应用、授权范围、密钥、调用日志 | 普通业务数据维护入口，除非同时有业务权限 |
| 审计/运维人员 | 追踪问题和风险 | 平台日志、系统日志、requestId 追踪、健康状态 | 数据编辑和业务配置入口 |

关键结论：角色是权限模板和用户画像，不是写死角色名。最终页面必须由真实权限、成员上下文和菜单授权决定。

## 信息架构重新整理

### 平台层

平台层只处理跨系统或全局能力：

1. 平台首页：解释当前平台状态和下一步。
2. 我的业务系统：创建系统、进入系统、查看我可管理或可使用的系统。
3. 平台账号：维护全局登录账号。该页面只给平台管理员使用。
4. 平台角色：维护平台级权限。
5. 对外应用中心：创建外部接入应用，管理密钥、授权、限流和调用日志。
6. 平台审计：平台账号、系统、对外应用和运维日志。
7. 运维诊断：健康、配置、版本、迁移。

### 系统层

系统层只处理当前业务系统：

1. 系统首页：告诉管理员或普通用户下一步该做什么。
2. 成员与权限：在当前系统内开通成员账号、分配角色和数据范围。
3. 建模配置：创建业务应用、模块、字段、页面、菜单动作和发布。
4. 业务运行台：普通用户日常新增、查询、编辑、提交业务数据。
5. 协同与文件：流程、附件、导出。
6. 系统对外授权：把当前系统的数据、模块动作、字段范围授权给平台级对外应用。
7. 系统审计：当前系统内的配置、数据、流程、文件和 OpenAPI 调用日志。

### 普通业务用户默认入口

普通业务用户登录后：

1. 如果只属于一个系统，直接进入该系统的业务运行台。
2. 如果属于多个系统，进入“选择业务系统”，但页面文案必须是业务系统选择，不是平台后台。
3. 进入系统后只看到运行台、自己可用模块、必要的个人信息和退出。
4. 没有模块时，显示“当前系统还没有给你开放业务模块，请联系系统管理员”，不能显示空后台。

## 主流程重新冻结

### 流程 A：从零创建“车”系统并让普通成员使用

1. 平台管理员登录。
2. 在“我的业务系统”创建“车”系统。
3. 创建成功后进入“车系统首页”，看到搭建步骤。
4. 在“成员与权限”中直接开通 `che` 成员账号，不要求用户先去平台账号页面。
5. 在“建模配置”中创建业务应用，例如“车辆业务”。
6. 创建模块，例如“车辆档案”。
7. 添加字段，例如车牌号、车型、颜色、状态。
8. 保存列表页、表单页、详情页，发布模块和菜单。
9. 给 `che` 成员授予“车辆档案”的菜单、查看、新增、编辑和必要字段权限。
10. 退出平台管理员账号。
11. 使用 `che` 登录。
12. 系统直接进入“车系统 / 业务运行台”。
13. `che` 只能看到车辆档案模块。
14. `che` 可以新增、查询、编辑车辆数据。
15. `che` 看不到平台账号、平台角色、系统建模、字段设计、发布和授权入口。

### 流程 B：把“车系统”的车辆数据开放给外部

1. 平台或集成管理员进入“对外应用中心”。
2. 创建对外应用，例如“车联网外部接入”。
3. 生成 appKey/secret，secret 只展示一次。
4. 在“车系统 / 系统对外授权”中授权车辆档案模块。
5. 选择可访问字段、动作和数据范围。
6. 配置 IP 白名单、限流和过期时间。
7. 外部系统调用 OpenAPI。
8. 调用日志能按 requestId 查到签名、scope、限流、系统、模块和结果。

### 流程 C：普通人遇到问题后的恢复

1. 登录失败：显示账号不存在、密码错误、账号停用或服务异常。
2. 没有系统：提示联系管理员加入系统，或有权限时创建系统。
3. 没有模块：提示当前系统未发布业务模块。
4. 没有权限：按钮不可见或禁用，并展示可理解原因。
5. 发布失败：指出缺字段、缺页面、缺菜单或缺权限。
6. 外部调用失败：按 requestId 查原因。

## P15 分期方式

P15 不是继续追加功能，而是重整交付质量。

| 子阶段 | 目标 | 退出标准 |
| --- | --- | --- |
| P15-0 PM 重整 | 撤回 P14 过度验收，冻结本文件和问题清单 | PM 能回答普通人怎么用 |
| P15-1 UI/UX 重画 | 输出关键页面结构和主操作设计 | 登录、系统首页、成员、建模、运行台、对外应用、日志都有可执行设计 |
| P15-2 实现修复 | 按 UI/UX 改前端，补必要后端 | 普通成员入口、系统成员开通、退出、分页、新建入口可用 |
| P15-3 真实验收 | 用部署环境或本地完整环境跑“车系统”剧本 | che 普通账号登录后直接使用车辆模块，不看到平台管理 |
| P15-4 打包 | 只有 P15-3 通过后再打包 | 包内包含 P15 证据和部署说明 |

## 不打包条件

以下任一成立，不得打包：

1. 普通成员登录后仍进入平台管理视角。
2. 系统管理员仍需要绕到平台账号页面才能开通系统成员。
3. 运行台无法新增、查询、编辑一条真实业务数据。
4. 普通成员能看到平台管理、建模、字段设计或权限配置入口。
5. 列表仍无分页或无法解释总数、当前页和下一页。
6. 新建入口仍是散落在表头或工程化表单堆叠。
7. 页面主流程只能靠文档解释，普通人看页面不知道下一步。
8. 没有真实浏览器验收截图/记录。

## PM 对多角色的修正规则

1. analyst 不再只复述需求，要找“普通用户不能完成任务”的断点。
2. PM 不再接受“功能点列表完成”，只接受“用户剧本完成”。
3. UI/UX 必须先给出页面级交互和主次按钮，不允许 frontend 自己临场设计。
4. frontend 必须按页面任务实现，不按 API 模块堆表格。
5. backend 必须把底层模型包装成可理解业务操作，例如“开通系统成员账号”。
6. test 必须用陌生普通用户账号跑真实页面流程。
7. reviewer 必须尝试推翻 PM 结论；只要部署体验和文档结论矛盾，就判 fail。
8. validator 只验证构建不代表可用；可用性必须由真实用户剧本决定。
