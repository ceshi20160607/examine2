# unexamine 全量重构产品方案

> 这是最终版综合方案。它以现有项目能看到的功能为参照，但不被旧代码限制。
> 重点是：完整产品应该包含哪些功能，现有功能应该如何升级，界面应该如何用更现代的方式呈现，系统如何更健壮、更好扩展。

## 1. 方案定位

unexamine 要做成一个“可配置业务系统平台”。

它要覆盖日常小型和中型内部系统的大多数需求：

- 搭业务表。
- 搭表单。
- 搭列表。
- 搭详情。
- 搭审批。
- 搭仪表盘。
- 分权限。
- 管成员。
- 上传附件。
- 导出数据。
- 外部系统接入。
- 查看日志。
- 稳定部署上线。

它不是单纯的“低代码演示项目”，也不是开发人员自己调接口的后台，而是一个普通企业团队可以长期使用的产品。

## 2. 总体目标

### 2.1 产品目标

- 管理员可以搭建一个业务应用。
- 普通用户可以顺手使用这个应用。
- 外部系统可以安全接入这个应用。
- 项目可以稳定部署、升级、维护。

### 2.2 覆盖范围

这套产品要能支撑：

- 客户管理。
- 合同管理。
- 订单管理。
- 项目管理。
- 任务管理。
- 报销审批。
- 请假审批。
- 采购申请。
- 资产管理。
- 工单处理。
- 设备巡检。
- 内部台账。
- 简单 CRM。
- 简单 OA。
- 简单进销存。
- 轻量 ERP 子场景。
- 轻量 MES 子场景。
- 简单售后系统。

说明：

- ERP 和 MES 不单独做重型行业套件。
- 通过模块、字段、子表、流程、规则、KPI、打印模板、仪表盘等能力，组合出轻量 ERP 和轻量 MES 场景。
- 例如采购、销售、库存台账、生产工单、质检记录、设备巡检，都可以作为普通模块配置出来。

### 2.3 核心判断标准

一个不会写代码的管理员，应该可以完成：

1. 创建一个系统。
2. 创建一个应用。
3. 创建一个业务模块。
4. 配置字段。
5. 生成页面。
6. 配置菜单。
7. 配置角色权限。
8. 配置审批流程。
9. 邀请用户使用。
10. 导出数据或开放接口。

普通用户应该可以完成：

1. 登录。
2. 进入自己有权限的系统。
3. 打开业务菜单。
4. 新增和编辑数据。
5. 查询和筛选数据。
6. 查看详情。
7. 上传附件。
8. 处理审批。
9. 查看消息。
10. 导出自己有权限的数据。

## 3. 现有功能应该如何看待

现有项目已经有不少后端能力，这些能力不能浪费。

可以保留或参考：

- 平台登录。
- 系统和租户。
- 应用。
- 模块。
- 字段。
- 字典。
- 部门。
- 页面。
- 菜单。
- 角色权限。
- 数据记录。
- 记录历史。
- 字段关联模块和子表。
- 流程模板。
- 流程运行。
- 待办和抄送。
- 上传。
- 导出。
- OpenAPI。
- 数据库迁移。
- 启动和部署脚本。

但现有项目的问题也很明显：

- 页面散，不像产品。
- 配置态和使用态混在一起。
- 很多页面像接口测试页面。
- 中文显示和编码存在问题。
- 权限体验不完整。
- 流程设计不够产品化。
- 表单设计不够直观。
- 数据详情不够友好。
- 部署配置和密钥检查还不够稳定。

所以重构时不是“照着旧代码重写一遍”，而是：

- 用现有功能确认业务范围。
- 用现有表和接口确认已有积累。
- 用新的产品结构重新组织功能。
- 用新的页面和交互重新设计体验。
- 用更清楚的后端业务接口重新收敛能力。

## 4. 产品整体结构

产品分成五个大的区域。

## 4.1 平台层和自定义系统层

成熟项目必须先分清两个层级：

- 平台层。
- 自定义系统层。

平台层是产品本身的管理层，负责管理所有系统、账号、授权、部署和全局运维。

自定义系统层是用户创建出来的业务系统，负责这个系统内部的模块、字段、页面、流程、权限、数据和业务设置。

两者不能混在一起。

### 平台层负责什么

平台层负责：

- 平台账号。
- 平台管理员。
- 平台角色。
- 平台菜单。
- 系统创建。
- 系统启用、停用、删除。
- 租户管理。
- 平台级授权。
- OpenAPI 全局策略。
- 全局密钥和安全配置。
- 全局文件存储配置。
- 全局日志。
- 系统健康检查。
- 版本信息。
- 部署信息。
- 数据库脚本版本。

平台层不直接管理某个业务模块里的业务数据。

### 自定义系统层负责什么

自定义系统层负责：

- 当前系统内用户。
- 当前系统内部门。
- 当前系统内角色。
- 当前系统内权限。
- 系统基础信息。
- 系统登录入口。
- 系统参数。
- 系统功能开关。
- 系统待办。
- 系统消息。
- 系统日志。
- 应用。
- 模块。
- 字段。
- 页面。
- 菜单。
- 字典。
- 流程。
- 打印模板。
- KPI 目标。
- 自定义仪表盘。
- 模块数据。
- 附件。
- 模块数据导入导出。
- 系统内消息。
- 系统内日志。
- 系统内 OpenAPI 授权。

自定义系统层只能管理当前系统自己的配置和数据。

### 层级边界

边界规则：

- 平台层创建和管理系统。
- 自定义系统层配置和使用业务。
- 平台账号可以进入多个自定义系统。
- 普通系统用户可以通过系统域名、系统编码或专属路径直接登录某个自定义系统。
- 一个自定义系统有自己的用户、部门、角色、权限、模块和数据。
- 自定义系统之间数据隔离。
- 平台管理员可以做平台运维，但不能绕过业务规则随意改业务数据，除非有明确的运维授权。
- 自定义系统管理员不能管理平台全局配置。

平台层菜单：

- 我的系统。
- 系统管理。
- 租户管理。
- 平台账号。
- 平台角色。
- 平台授权。
- 全局配置。
- 系统健康。
- 操作日志。
- 版本和部署。

自定义系统层菜单：

- 系统首页。
- 系统基础信息。
- 登录入口。
- 系统参数。
- 系统待办。
- 系统消息。
- 应用管理。
- 模块管理。
- 字段设计。
- 页面设计。
- 菜单配置。
- 用户和部门。
- 角色权限。
- 字典配置。

- KPI 目标。
- 自定义仪表盘。
- 文件管理。

运行台菜单：

- 我的待办。

- 我的申请。
- 自定义仪表盘。

### 权限划分

平台层权限：

- 创建系统。
- 管理系统状态。
- 管理租户。
- 管理平台账号。
- 管理平台角色。
- 查看平台日志。
- 管理全局配置。
- 查看系统健康。

自定义系统层权限：

- 管理应用。
- 管理模块。
- 管理字段。
- 管理页面。
- 管理菜单。
- 管理系统内用户。
- 管理系统内部门。
- 管理系统内角色。
- 管理流程。
- 管理打印模板。
- 管理 KPI。
- 管理仪表盘。
- 管理 OpenAPI。
- 查看、新增、编辑、删除、导入、导出模块数据。

### 数据隔离

所有业务数据都必须带：

- systemId。
- tenantId。
- appId。
- moduleId。

平台数据和业务数据分开。

自定义系统 A 不能查到自定义系统 B 的数据。

同一个系统内，如果启用租户，也要按租户隔离。

### 页面入口

平台管理员和系统拥有者登录后进入平台层的“我的系统”。

选择一个系统后，进入自定义系统层。

普通系统用户通过系统域名、系统编码或专属路径登录后，直接进入该系统的运行台或默认首页。

在自定义系统层里，用户可以进入：

- 管理设置。
- 应用配置。
- 应用运行台。
- 流程工作台。

这个入口关系必须清楚，不能让用户在平台层和业务层之间迷路。

面向平台管理员和系统拥有者。

- 注册。

- 我的系统。

- 系统启用和停用。
- 系统成员。
- 平台账号。
- 平台角色。
- 平台菜单。
- 平台消息。
- 平台日志。
- 全局配置。

页面：

- 登录页。
- 注册页。
- 我的工作台。
- 我的系统。

页面：

- 系统管理中心。
- 系统基础信息。
- 登录入口配置。
- 系统参数配置。
- 待办中心。
- 消息中心。
- 应用列表。

- 评论或备注。

- 自定义仪表盘。

页面：

- 应用首页。
- 业务列表页。
- 业务表单页。
- 业务详情页。
- 关联数据页。

优化点：

- 运行台不能出现复杂配置入口。

- 详情页要结构化展示，不能主要显示 JSON。
- 关联数据和子表要能直接维护。
- 仪表盘用于展示当前应用或当前模块的关键指标、图表、待办、列表和趋势。

## 5. 全量功能清单

## 5.1 账号和登录

必须包含：

- 登录。
- 注册。
- 退出登录。
- 刷新登录状态。
- 修改密码。
- 重置密码。
- 当前用户信息。
- 登录日志。
- 登录失败限制。

- 创建系统。

- 选择一个模块作为数据来源，再设置单选或多选，就是常见的关联关系。
- 子表字段用于订单明细、合同回款、项目任务这类一条主数据带多条明细的场景。
- 页面保存时保存关联数据的 ID，页面展示时显示关联数据的名称或主要字段，避免用户看到技术 ID。

优化：

- 字段设计器用拖拽，不只用表格。
- 字段配置用右侧抽屉。
- 不同字段类型显示不同配置项。
- 自动编号要用数据库或独立序号服务保证并发安全。
- 删除字段默认停用，不直接清历史数据。

优化：

设计能力：

- 流程模板。

- 节点审批人。
- 节点字段权限。
- 发布。

节点类型：

- 开始。
- 普通审批。
- 条件审批。
- 抄送。
- 条件分支。
- 子流程。
- 自动节点。
- Webhook 节点。
- 外部节点。
- 结束。

审批节点配置：

- 审批人来源支持指定人员、角色、部门、部门负责人、直属上级、发起人上级、字段内人员、上一节点处理人。
- 审批方式支持或签、会签、顺序会签。
- 或签表示任意一人同意即可通过。
- 会签表示所有人都要处理，可以配置全部同意才通过或达到比例后通过。
- 顺序会签表示按配置顺序一个一个审批。
- 节点可以配置超时提醒、自动通过、自动拒绝、自动转交。
- 节点可以配置字段查看、字段编辑、字段必填、字段隐藏。
- 节点可以配置审批操作按钮，比如同意、拒绝、转办、加签、退回、撤回。

流程触发：

- 流程要绑定模块和事件点。
- 支持新建触发。
- 支持编辑触发。
- 支持删除前或删除后触发。
- 支持状态变化触发。
- 支持手动点击发起。
- 支持 OpenAPI 发起。
- 每个触发点可以配置条件，比如金额大于 5000 才走审批。
- 一个模块可以绑定多个流程，但同一个事件点要配置优先级和互斥规则，避免重复发起。

流程运行数据：

- 流程模板保存基础信息和启用停用状态。
- 流程版本保存发布快照，草稿修改不能影响已发布版本。
- 流程高级设置保存撤回、拒绝、重审、超时、异常兜底、是否允许加签加节点等规则。
- 流程节点保存节点类型、审批方式、审批人来源、外部节点配置、子流程配置。
- 流程连线和连线条件保存条件分支。
- 流程实例保存某条模块数据正在走哪一个流程模板版本。
- 流程实例创建时，要把模板版本的节点、连线、条件、字段权限和高级设置复制成运行快照。
- 流程任务保存当前待办、候选人、处理状态和处理意见。
- 流程提交快照保存节点内字段修改，支持审批过程中修改字段。
- 流程日志保存每一次节点进入、审批、拒绝、转办、加签、加节点、撤回、终止。
- 流程运行时必须使用发布版本快照，流程修改不能影响已发起实例。

模块事件触发流程：

- 模块数据新建、编辑、删除、状态变化时，由模块业务层产生事件。
- 根据系统、模块、事件点查询流程绑定。
- 按优先级判断绑定条件。
- 满足条件后读取启用的流程模板和最新发布版本。
- 创建流程实例并复制版本快照。
- 回写模块数据的审批状态和当前流程实例 ID。
- 如果绑定是互斥的，命中第一个流程后停止继续匹配。

手动增加节点：

- 流程运行中支持管理员或有权限的人手动增加审批节点、抄送节点。
- 手动节点要记录添加人、添加原因、插入位置、节点配置快照。
- 手动节点可以插在当前节点后，也可以插在指定节点前后。
- 手动节点不能破坏已完成节点的历史记录。

运行能力：

- 发起流程。
- 待办。
- 抄送。
- 我的申请。
- 实例详情。
- 同意。
- 拒绝。
- 转办。
- 领取。
- 取消领取。
- 撤回。
- 终止。
- 加签。
- 减签。
- 催办。
- 评论。

优化：

- 流程设计器用画布加右侧配置。
- 创建流程时选择系统内一个模块，审批、业务详情、流程状态都围绕这个模块的数据展开。
- 发布前检查是否有断线、死路、空审批人。

- 导入错误要能下载错误行和错误原因。
- 导入配置只读取当前模块允许写入的字段，系统字段、审计字段、自动编号字段不能被外部覆盖。

- 订单模块打印订单或出库单。
- 工单模块打印维修单或处理单。

打印模板配置：

- 模板归属某个模块。
- 模板使用模块内字段。
- 模板可以使用关联字段的展示值。
- 模板可以配置页眉、页脚、签章区、明细表格。
- 模板支持打印预览。
- 模板支持下载或导出 PDF。

KPI 目标场景：

- 销售模块设置销售目标。
- 项目模块设置交付目标。
- 工单模块设置处理时效目标。
- 客服模块设置满意度目标。

优化：

- 系统管理设置放在自定义系统内，不放到平台全局。
- 用户、部门、审批、字典、打印模板、KPI 目标都按当前系统隔离。
- 打印模板和模块绑定，避免模板不知道读取哪些字段。
- 打印预览要在正式打印前展示效果。
- KPI 目标可以和模块数据统计关联。

## 5.19 智能能力

智能能力作为正式扩展能力纳入架构，所有动作都受权限、审计和人工确认控制。

包含：

## 7. 技术方案

## 7.1 前端技术

技术选型：

- 如果要快速做企业后台，优先 Ant Design Vue 或 Element Plus。
- 如果要主题更灵活，优先 Naive UI。
- 如果要设计更年轻一些，可以考虑 Arco Design Vue。
- 如果表格要非常强，考虑 AG Grid。
- 如果想完全掌控表格样式，考虑 TanStack Table。
- 流程设计器使用 Vue Flow 这类成熟画布库，不从零手写画布底层。

## 7.2 后端技术

推荐：

- Java 21。
- Spring Boot。
- 自研认证和权限拦截，不使用 Spring Security。
- MyBatis-Plus。
- MySQL。
- Redis。
- Flyway。
- MinIO 或 S3 兼容对象存储。
- 后台作业和轻量事件使用 Redis 实现，不使用 Quartz，不使用消息队列。
- OpenAPI 文档。

数据库规则：

- 当前数据库使用 MySQL。
- SQL 尽量使用标准 SQL，减少数据库私有语法，保证数据库切换成本可控。
- MySQL 特有能力可以用，但必须控制范围，比如分页、JSON、索引等。
- 特殊 SQL 统一写在 mapper xml 中，便于替换和维护。
- 业务接口不要依赖数据库方言太深。

## 7.3 架构方式

后端按功能分清楚：

- 平台。
- 应用。
- 模块。
- 字段。
- 页面。
- 权限。
- 数据。
- 流程。
- 文件。
- 导出。
- OpenAPI。
- 日志。

代码结构分两部分：

第一部分是基础生成代码：

- 放在 `base` 包。
- `base.entity` 保存表字段。
- `base.mapper` 处理基础 SQL。
- `base.mapper.xml` 保存基础 SQL 和允许存在的特殊 SQL。
- `base.service` 保存基础增删改查。
- `base.controller` 可以由工具生成，但新版前端不直接依赖这些散接口。

第二部分是业务开发代码：

- 放在 `manage` 包。
- `manage.controller` 只提供产品需要的业务接口。
- `manage.service` 处理产品业务逻辑和核心业务编排。
- `manage.validator` 做保存前校验。
- `manage.permission` 做权限判断。
- `manage.event` 承载事件扩展点。

命名规则：

- 基础生成代码统一放在 `base`。
- 业务开发代码统一放在 `manage`。
- 如果某类业务特别复杂，可以在 `manage` 下再拆 `domain`、`validator`、`permission`、`event`。
- 业务 controller 和业务 service 都放在 `manage`，不要和生成代码混在一起。

规则：

- 基础代码可以由工具生成。
- 业务逻辑不要写进 `base.controller`。
- 复杂业务不要塞进一个大 service。
- 特殊 SQL 可以写在 `base.mapper.xml` 中，这是允许的特例。
- `manage.service` 可以调用多个 `base.service`、`base.mapper`、`manage.permission`、`manage.event`，负责把一个完整业务动作串起来。
- event 采用接口和本地扩展点承载统一事件机制。

## 7.4 数据保存方式

业务数据可以采用“主记录 + 字段值”的方式：

- 主记录保存系统、应用、模块、状态、创建人。
- 字段值保存每个字段的值。
- 常用字段可以有 typed value，方便查询。
- 附件、人员、部门、关联关系单独处理。

优化：

- 列表查询要有索引设计。
- 常用筛选字段要能建索引。
- 自动编号要单独序号表。
- 关联关系要有明确表，不要只塞 JSON。

- 子流程状态要正确回到父流程。
- 任务不能重复处理。

权限必须做到：

- 前端隐藏。
- 后端拦截。
- 数据过滤。

完整权限至少包含：

- 菜单可见。
- 页面可见。
- 按钮可用。
- 数据范围。
- 字段可见。
- 字段可编辑。
- OpenAPI 可访问。

## 7.7 后台作业

后台作业包括：

- 导出。
- 导入。
- 文件处理。
- 流程通知。
- OpenAPI 异常重试。
- 数据统计。

实现方式：

- 系统只使用 Redis 做后台作业队列、作业锁、作业状态和重试记录。
- 不使用 Quartz。
- 不使用消息队列。
- 不引入复杂任务平台。
- 更多业务扩展统一进入 event 模块。

优化：

- 任务要有状态。
- 任务要有错误原因。
- 任务可以重试。
- 任务不能无限重试。
- 任务结果可下载。

- 请求日志。
- 错误日志。
- 操作日志。
- 慢查询日志。
- 后台作业日志。
- OpenAPI 调用日志。

上线要求：

- 每个请求有 requestId。
- 前端错误带 requestId。
- 后端日志带 systemId、tenantId、userId。
- 关键异常写入系统消息。

## 14. 上线能力完善规则

## 14.1 配置版本管理

关键配置必须有版本：

- 模块字段版本。
- 页面配置版本。
- 流程版本。
- 打印模板版本。

- 页面发布后，运行端使用发布版本，未发布草稿不影响用户。
- 流程发布后，老流程实例继续按老版本走，新实例使用新版本。
- 打印模板改版后，历史单据可以按历史模板打印。

## 14.2 配置发布机制

配置分成“草稿”和“已发布”：

- 管理员可以先在草稿里改。
- 预览没问题再发布。
- 普通用户只使用已发布版本。

适合的功能：

- 页面。
- 流程。
- 打印模板。
- 字段显示配置。
- 导入导出配置。

好处：

- 避免管理员改配置时影响正在使用的用户。
- 方便做预览和回滚。

## 14.3 模块生命周期

模块必须有清楚状态：

- 草稿。
- 已启用。
- 已停用。
- 已归档。

使用场景：

- 新模块配置中，不让普通用户看到。
- 模块停用后，保留历史数据但不允许新增。
- 模块归档后，只允许管理员查看。

## 14.4 业务规则配置

除了字段必填，还应该支持简单业务规则：

- 某字段满足条件时显示另一个字段。
- 某字段满足条件时必填。
- 某状态下不允许编辑。
- 某金额超过指定值时必须走审批。
- 某类数据只能指定角色删除。

实现规则：

- 使用结构化条件配置。
- 不引入复杂脚本。
- 条件规则要能预览和测试。

## 14.5 数据唯一性和重复检查

模块内可以配置唯一规则：

- 客户手机号不能重复。
- 合同编号不能重复。
- 资产编码不能重复。
- 工单号不能重复。

导入时也要检查重复：

- 和数据库已有数据重复。
- Excel 内部自己重复。
- 更新模式下找不到匹配数据。

## 14.6 导入回滚和预演

导入必须支持：

- 导入前先预演。
- 显示预计新增多少条、更新多少条、失败多少条。
- 真正导入后生成导入批次。
- 出问题时可以按批次回滚。

注意：

- 回滚只回滚本次导入影响的数据。
- 如果导入后数据又被别人改过，要提示不能直接回滚。

## 14.7 打印模板版本和套打

打印模板必须支持：

- 模板版本。
- 模板草稿。
- 模板发布。
- 套打位置调整。
- 明细表格自动分页。
- 打印前预览。
- PDF 下载。

使用场景：

- 合同版本变化。
- 报销单格式调整。
- 采购单需要固定签章区。
- 出库单需要固定纸张尺寸。

## 14.8 KPI 目标闭环

KPI 目标不只是录一个目标值，还应该能和模块数据联动。

必须支持：

- 目标对象：人员、部门、角色。
- 目标周期：月、季度、年。
- 目标来源：某个模块的某个字段汇总。
- 达成率计算。
- 目标看板。
- 未达标提醒。

使用场景：

- 销售目标按合同金额统计。
- 工单目标按按时完成率统计。
- 项目目标按任务完成数统计。

## 14.9 系统体检

管理端必须包含系统体检：

- 数据库连接。
- Redis 连接。
- 文件存储。
- 密钥配置。
- Redis 后台作业状态。
- 最近错误日志。
- 数据库脚本版本。
- 当前服务版本。

好处：

- 部署后能快速判断系统是否正常。
- 出问题时不用到服务器里翻半天日志。

## 14.10 权限预览

权限中心必须支持：

- 选择一个用户，预览他能看到哪些菜单。
- 预览他能操作哪些按钮。
- 预览他能看到哪些字段。
- 预览他能查看哪些数据范围。

使用场景：

- 管理员配置权限后，马上确认普通用户看到的效果。
- 排查“为什么某个人看不到菜单”。

## 14.11 数据归档和恢复

业务系统使用久了之后，数据会越来越多。

必须支持：

- 软删除。
- 恢复。
- 归档。
- 归档查询。
- 批量归档。

使用场景：

- 已完成很久的工单归档。
- 历史合同归档。
- 老项目归档。

## 14.12 审批和业务状态联动

审批通过或拒绝后，模块数据状态应该自动变化。

示例：

- 请假审批通过，状态变成“已通过”。
- 报销审批拒绝，状态变成“已拒绝”。
- 合同审批中，业务数据不能随意编辑。
- 审批撤回后，数据回到草稿状态。

这类状态联动要可配置，不能写死在某个模块里。

## 14.13 操作确认和风险提示

危险操作要有清楚提示：

- 删除模块。
- 删除字段。
- 停用字段。
- 发布流程。
- 停用流程。
- 导入覆盖数据。
- 批量删除数据。
- 修改权限。

提示内容要告诉用户会影响什么，不只问“是否确认”。

## 14.14 用户体验细节

必须包含：

- 全局搜索。
- 最近访问。
- 收藏菜单。
- 快捷创建。
- 表格列保存。
- 筛选条件保存。
- 表单草稿。
- 错误字段定位。
- 移动端扫码或拍照上传。

这些细节会明显提升日常使用手感。

## 14.15 开放扩展点

event 扩展点包括：

- 数据创建后。
- 数据更新后。
- 数据删除后。
- 流程发起后。
- 流程通过后。
- 流程拒绝后。
- 文件上传后。
- 导入完成后。
- 导出完成后。

event 采用本地接口承载，不接消息队列；统一承载通知和自动化能力扩展。

## 14.16 命名和编码规范

成熟项目需要统一命名和编码规则：

- 系统编码。
- 应用编码。
- 模块编码。
- 字段编码。
- 页面编码。
- 菜单编码。
- 角色编码。
- 流程编码。
- 打印模板编码。
- 字典编码。

规则：

- 编码只用英文、数字、下划线。
- 同一系统内编码不能重复。
- 编码创建后尽量不允许修改。
- 显示名称可以随时修改。
- 导入导出、OpenAPI、打印模板优先使用编码定位字段。

## 14.17 配置迁移和升级

系统升级必须考虑：

- 数据库脚本升级。
- 模块配置升级。
- 页面配置升级。
- 流程配置升级。
- 打印模板升级。
- 权限配置升级。

恢复规则：

- 配置变更有版本。
- 升级脚本可重复检查。
- 升级前能备份。
- 升级失败能回滚或给出明确处理方式。

## 14.18 备份和恢复

上线产品需要考虑备份：

- 数据库备份。
- 文件备份。
- 配置备份。
- 打印模板备份。
- OpenAPI 密钥备份策略。

恢复场景：

- 误删业务数据。
- 配置改错。
- 文件丢失。
- 部署升级失败。

注意：

- 密钥不能明文导出。
- 备份和恢复要记录操作日志。

## 14.19 多环境管理

成熟项目通常至少有：

- 开发环境。
- 测试环境。
- 生产环境。

要求：

- 不同环境配置分开。
- 生产环境不能使用占位密钥。
- 测试数据不能混入生产。
- 部署包和配置文件分开。
- 前端 API 地址按环境配置。

## 14.20 安全策略

安全策略要覆盖：

- 密码加密存储。
- token 过期。
- 刷新 token。
- 登录失败限制。
- 敏感操作二次确认。
- OpenAPI 签名。
- IP 白名单。
- 文件上传类型限制。
- 文件大小限制。
- 防止越权访问。
- 防止跨系统访问数据。

## 14.21 成熟项目文档

除了代码，还需要文档：

- 部署说明。
- 配置说明。
- 数据库说明。
- 启停脚本说明。
- 管理员手册。
- 普通用户手册。
- OpenAPI 接入说明。
- 常见问题。
- 错误排查说明。
- 升级说明。

## 14.22 API 规范

成熟项目需要统一 API 规范：

- URL 命名统一。
- 请求方式统一。
- 分页参数统一。
- 排序参数统一。
- 筛选参数统一。
- 返回结构统一。
- 错误码统一。
- requestId 统一。
- 时间格式统一。
- 金额和数字精度统一。

实现规则：

- 内部接口和 OpenAPI 分开。
- 平台层接口和自定义系统层接口分开。
- 所有接口返回都带 requestId，方便排查。
- 错误信息要给用户看得懂，错误码给开发和运维定位。

## 14.23 错误码和异常规范

错误不能都叫“参数错误”或“系统异常”。

按模块划分：

- 账号错误。
- 系统错误。
- 应用错误。
- 模块错误。
- 字段错误。
- 权限错误。
- 流程错误。
- 文件错误。
- 导入导出错误。
- OpenAPI 错误。

要求：

- 后端抛出明确错误码。
- 前端展示人能看懂的话。
- 日志记录详细原因。
- 用户提示和技术日志分开。

## 14.24 幂等和重复提交

成熟项目必须处理重复提交：

- 表单重复点击保存。
- 审批按钮重复点击。
- OpenAPI 重复请求。
- 导入任务重复提交。
- 导出任务重复提交。

实现规则：

- 使用 requestId 或 idempotencyKey。
- Redis 保存短时间幂等记录。
- 审批任务按任务状态防重复处理。
- OpenAPI 必须支持幂等键。

## 14.25 限流和防刷

需要限制：

- 登录接口。
- 注册接口。
- OpenAPI。
- 文件上传。
- 导入任务。
- 导出任务。
- 高频查询接口。

实现规则：

- 用 Redis 做限流计数。
- 按 IP、账号、appKey、系统维度限流。
- 超限时返回明确提示。

## 14.26 缓存策略

可以缓存：

- 当前用户权限。
- 系统配置。
- 字典项。
- 菜单。
- 页面发布配置。
- 字段发布配置。
- 打印模板发布配置。

要求：

- 缓存 key 带 systemId、tenantId。
- 权限变更后清缓存。
- 配置发布后清缓存。
- 不缓存敏感明文。
- 缓存失效时能自动回源。

## 14.27 前端设计系统

成熟项目要统一前端设计：

- 颜色。
- 字体。
- 间距。
- 按钮。
- 表格。
- 表单。
- 弹窗。
- 抽屉。
- 空状态。
- 错误状态。
- 加载状态。
- 图标。

要求：

- 不同页面不要各写各的样式。
- 运行台和配置台风格统一但重点不同。
- 移动端组件单独适配。
- 按钮主次关系清楚。

## 14.28 前端权限和路由守卫

前端需要：

- 路由守卫。
- 菜单过滤。
- 按钮权限。
- 字段可见控制。
- 字段可编辑控制。
- 直接访问无权限页面时提示。

注意：

- 前端只是体验优化。
- 后端必须再次校验。

## 14.29 性能优化

重点关注：

- 大表格查询。
- 字段值查询。
- 关联字段查询。
- 权限数据过滤。
- 导入导出。
- 文件上传下载。
- 流程待办查询。
- 仪表盘统计。

配置规则：

- 列表必须服务端分页。
- 仪表盘统计可以缓存。
- 常用筛选字段支持索引。
- 大导出走 Redis 后台作业。
- 前端表格支持虚拟滚动。

## 14.30 数据统计和报表基础

仪表盘之外，还应有基础统计能力：

- 按模块统计数量。
- 按字段分组统计。
- 按时间趋势统计。
- 按人员或部门统计。
- 按状态统计。

这些统计可以服务：

- 仪表盘。
- KPI。
- 运行台首页。
- 管理员概览。

## 14.31 变更审计

业务数据变更要记录：

- 谁改的。
- 什么时候改的。
- 改了哪个字段。
- 修改前是什么。
- 修改后是什么。
- 来源是 Web、移动端还是 OpenAPI。

使用场景：

- 业务追责。
- 数据恢复。
- 审计检查。
- 排查误操作。

## 14.32 灰度和功能开关

支持功能开关：

- 某个系统启用某功能。
- 某个角色可见新功能。
- 某个模块启用新页面。
- 某个流程启用新版审批页。

功能开关使用结构化配置：

- 系统级开关。
- 应用级开关。
- 模块级开关。

## 14.33 部署回滚

上线部署要考虑回滚：

- 后端版本回滚。
- 前端静态资源回滚。
- 配置回滚。
- 数据库脚本失败处理。

注意：

- 数据库结构升级要谨慎。
- 破坏性脚本必须单独确认。
- 部署前备份。
- 部署后记录版本。

## 14.34 容量和配额

平台可以设置配额：

- 每个系统最大用户数。
- 每个系统最大应用数。
- 每个应用最大模块数。
- 每个模块最大字段数。
- 单文件最大大小。
- 每日导出次数。
- OpenAPI 调用次数。

好处：

- 防止某个系统误操作拖垮平台。
- 支撑不同版本授权。

## 14.35 数据脱敏

敏感字段必须支持脱敏：

- 手机号。
- 身份证号。
- 银行卡号。
- 地址。
- 密码类字段。
- secret。

规则：

- 列表显示脱敏。
- 详情按权限显示。
- 导出按权限决定是否脱敏。
- OpenAPI 按授权决定是否返回明文。

## 14.36 可访问性和移动适配

虽然是后台系统，也要考虑：

- 文本可读。
- 表单错误清楚。
- 按钮大小合适。
- 移动端不拥挤。
- 关键操作不依赖鼠标悬停。
- 表格在小屏下有替代展示。

## 14.37 落地级数据设计原则

成熟产品的表结构不能只满足增删改查，还要满足真实上线后的配置变更、运行追溯、权限解释、性能查询和失败恢复。

必须做到：

- 配置类能力要有草稿、发布、版本和回滚。
- 运行类能力要保存快照，不能被配置修改影响。
- 模块字段、页面、流程、打印模板、仪表盘、数据源都要支持发布版本。
- 列表常用筛选字段要有索引值表，不能所有查询都扫字段值表。
- 自动编号要有独立序号表，不能查询最大值加一。
- 导入导出要有任务明细，能看到每一行成功或失败。
- KPI 要有计算记录，能解释实际值来自哪里。
- 仪表盘要有组件表、版本表、权限表和缓存表。
- OpenAPI 要有密钥版本、幂等、限流、调用日志。
- 文件要有分片、预览、引用关系。
- 权限要能解释为什么能看、为什么不能看。
- 大日志和大任务明细要支持归档。

这些不是内置能力，而是重构版从一开始就要按这个方向设计。
