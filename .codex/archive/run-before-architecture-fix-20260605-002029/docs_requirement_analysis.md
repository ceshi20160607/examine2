# 需求分析

## 原始需求摘要

本次重构目标是将 unexamine 建设为可长期使用的“可配置业务系统平台”，而不是旧项目的简单复刻或接口调试后台。产品需要覆盖普通企业内部小型、中型系统的搭建和运行，包括系统、应用、模块、字段、页面、菜单、角色权限、业务数据、审批、附件、导入导出、OpenAPI、日志审计、部署运维等完整能力。

需求明确要求区分平台层、自定义系统层和运行台：

- 平台层负责平台账号、平台管理员、系统创建、租户管理、平台授权、全局配置、健康检查、部署信息和全局日志。
- 自定义系统层负责某个系统内的用户部门、角色权限、应用、模块、字段、页面、菜单、字典、流程、打印模板、KPI、仪表盘、文件、OpenAPI 和日志。
- 应用运行台面向普通业务用户，提供业务列表、表单、详情、关联数据、附件、评论、历史、流程状态、导出和仪表盘。
- 流程工作台面向发起人、审批人和管理员，提供待办、抄送、申请、实例、流程模板、流程设计、发布和审批处理。
- 运维中心需要覆盖配置检查、数据库检查、文件存储检查、服务健康、接口日志、导出任务、异常消息、版本和部署信息。

功能范围是全量产品目标，实施顺序只用于组织开发，不改变最终范围。系统需要支持轻量 CRM、OA、进销存、ERP 子场景、MES 子场景等，通过模块、字段、子表、流程、规则、KPI、打印模板和仪表盘组合实现。

## 需求边界与不确定项

明确边界：

- ERP 和 MES 不作为重型行业套件单独建设，只通过可配置能力支撑轻量子场景。
- 配置态和使用态必须分离，运行台不应暴露复杂配置入口。
- 平台数据和业务数据需要分层，业务数据需按 systemId、tenantId、appId、moduleId 隔离。
- 前端权限只是体验优化，后端接口必须再次校验。
- 字段、页面、流程、打印模板、导入导出配置等需要支持草稿、发布、版本和回滚。
- OpenAPI 与内部接口需要分开，OpenAPI 需要签名、幂等、限流、调用日志、密钥版本和授权边界。
- 自动编号不能使用“查询最大值 + 1”，需要独立序号或数据库原子能力。
- 导入导出、文件上传、审批和远程调用等需要考虑任务状态、失败原因、补偿、幂等和审计。

不确定项：

- 首期交付范围未从全量目标中切出 MVP 边界，PRD 需要明确阶段拆分、必须先做的闭环和暂缓能力。
- 移动端被列为最终交付项，但当前 step 输入未给出移动端首期页面优先级，PRD 需要明确 Web 与移动端的交付节奏。
- 智能助手、命令中心、流程模拟、可视化打印模板等现代能力已提出方向，但未明确首期是否实现、采用何种模型或交互深度。
- 数据库连接、Redis 连接等公共配置来自 `docs/service_info.md`，该文件终端显示存在编码异常，后续 PRD/设计应只采纳可确认的路径、构建和基础配置约束，并避免把明文密钥写入运行配置。
- 旧项目包含较多运行日志、dist、node_modules、历史 SQL 和 zip 包，PRD 不应将这些历史产物视为新项目交付内容。

## 旧项目可复用能力

旧项目 `../examine2/` 可作为业务能力和路径索引参考，主要可复用以下能力模型：

- 后端工程结构：Maven 多模块，包含 `examine-core`、`examine-plat`、`examine-app`、`examine-module`、`examine-flow`、`examine-upload`、`examine-web`，技术方向为 Java 21、Spring Boot 3.3.5、MyBatis-Plus、Redis、Flyway、Knife4j。
- 平台账号与上下文：已有登录、注册、刷新、系统列表、进入系统、租户选择、Redis token 会话、systemId/tenantId 上下文切换等能力。
- 平台与系统权限：已有平台 RBAC、系统内 module RBAC、菜单权限、页面权限、字段权限、操作权限、数据范围、权限预览和权限缓存思路。
- 低代码元数据：已有应用、模块、字段、字段选项、字典、部门、页面、页面块、菜单、列表视图、筛选模板、导出模板、模块关系、模块动作、集成映射、模块版本等实体和接口。
- 业务数据运行：已有 `un_module_record` 与 `un_module_record_data` 的 EAV 数据结构、DSL 查询、记录详情、更新、软删、历史记录和关联查询。
- 审批流程：已有流程模板、版本、节点、连线、条件、实例、任务、参与人、变量、动作日志、轨迹日志、流程绑定、发起、同意、拒绝、撤回、终止、转交、抄送、会签/或签、子流程、图形设计器数据等能力。
- 文件上传：已有上传文件、分片、存储配置和系统上传入口，本地存储按 systemId/tenantId/日期分目录。
- OpenAPI：已有外部应用、凭证、授权范围、IP 白名单、访问日志、`/v1/open/**` 路径、AK/SK 或 HMAC 签名、`X-Acting-Plat-Id`、目标系统/租户、`Idempotency-Key` 等契约。
- 前端参考：Vue3 Web 已有登录、注册、系统、应用、模块、字段、字典、部门、页面、列表视图、导出、记录、流程、RBAC、上传、运行菜单和 OpenAPI 页面；UniApp 移动端已有认证、平台、运行台、记录、流程、模块配置、上传等页面。
- SQL 与迁移：旧项目有 `docs/sql` 与后端 Flyway migration，覆盖平台、上传、模块、流程、OpenAPI、平台 RBAC、字段扩展、序号、部门层级等表结构和种子数据。
- 测试与部署：已有 API smoke、Playwright Web UI、移动端 type-check/build、部署脚本、Nginx 配置、release 脚本和简单运行说明，可作为后续验证方案参考。

## 旧项目需要重构或规避的问题

产品与交互问题：

- 需求文件指出旧页面“散”，不像完整产品；配置态和使用态混在一起，部分页面像接口测试页。
- 表单设计、流程设计、详情页、权限体验不够产品化，运行台与配置台边界不清。
- 中文显示和编码存在问题；本次读取旧项目 README、架构文档、测试文档和 `service_info.md` 时也观察到终端输出有明显 mojibake，说明后续文档与源码编码需要统一校验。

工程与交付问题：

- 旧目录包含 `.git`、`.idea`、`.history`、`.worktrees`、`dist`、`logs`、`tmp`、`node_modules`、历史压缩包和大量运行日志，新项目不能继承这些产物。
- 旧项目存在多套 SQL 来源：`doc/`、`docs/sql/`、Flyway migration、manual repair 脚本等，后续数据库设计应统一到新项目 `sql/init.sql` 和明确迁移策略，避免版本来源混乱。
- 旧项目 OpenAPI 文档与 README 均提示部分能力以实现为准，且测试计划中仍有未覆盖项，例如 HMAC 签名模式、复杂流程 UI 操作、删除/批量操作、导出下载等，需要在 PRD 和验收中补齐。
- 旧项目已有 EAV 和 typed-value 扩展，但全量需求要求大表性能、索引值表、统计、KPI、仪表盘缓存、权限解释、归档等能力，不能只沿用旧 EAV 查询方式。
- 旧项目已有流程图设计器，但需求要求“流程设计器不能只是 JSON 编辑”，应重构为更强的可视化配置、校验、模拟和发布机制。
- 旧项目已有部署脚本和配置示例，但需求要求启动前检查密钥、数据库、Redis、文件存储和脚本版本，旧项目脚本只能参考，不能直接代表上线健壮性。

## 竞品/参考资料结论（如果需求文件提供）

需求文件未提供明确竞品名称或竞品链接，只提供了技术参考方向：

- 前端框架与构建：Vue、Vite。
- UI 组件方向：Ant Design Vue、Element Plus、Naive UI。
- 表格能力方向：TanStack Table、AG Grid。
- 流程图方向：Vue Flow。
- 后端方向：Spring Boot。

结论：

- PRD 不应绑定死具体版本，但要明确“Schema 驱动页面 + 高级表格 + 可视化流程 + 可配置仪表盘”的产品方向。
- UI 组件库需要在后续技术设计中二选一或明确组合边界，避免 Ant Design Vue、Element Plus、Naive UI 混用造成设计系统割裂。
- 高级表格需要服务端分页、排序、筛选、虚拟滚动、列拖拽、固定列、视图保存，普通组件表格不足以支撑全量需求。
- 流程图工具需要承载节点、连线、条件、审批人、会签/或签、抄送、子流程、发布检查和模拟，不应只作为图形展示。

## 对 PRD 的输入建议

建议 PRD 按“全量目标 + 阶段闭环”组织：

- 第一阶段应形成可运行闭环：账号登录、我的系统、创建系统、进入系统、应用/模块/字段/页面基础配置、运行台列表/表单/详情、基础 RBAC、文件上传、基础流程发起与待办、基础导出、基础日志、部署启动。
- 第二阶段补齐成熟产品能力：配置草稿/发布/版本、字段联动、唯一性、导入预演与回滚、流程可视化设计与模拟、权限预览、OpenAPI 签名与限流、导出任务、审计和系统体检。
- 第三阶段扩展体验与高级能力：命令中心、智能助手、打印模板可视化、KPI 闭环、自定义仪表盘、移动端完整运行体验、备份恢复、灰度和配额。

PRD 需要重点明确：

- 平台层、自定义系统层、应用配置台、应用运行台、流程工作台、运维中心的菜单和入口关系。
- 用户角色：平台管理员、系统拥有者、系统管理员、应用管理员、普通用户、审批人、运维人员、OpenAPI 调用方。
- 核心业务对象：系统、租户、应用、模块、字段、页面、菜单、角色、成员、字典、记录、记录值、流程模板、流程版本、流程实例、任务、附件、导入导出任务、OpenAPI 应用、审计日志、仪表盘、KPI、打印模板。
- 数据隔离与权限规则：systemId、tenantId、appId、moduleId 的使用边界，平台态 systemId=0 与系统态 systemId!=0 的切换规则。
- 配置发布规则：哪些配置需要草稿/发布/版本/回滚，运行时实例如何保存快照。
- 异常与验收：重复提交、越权、跨系统访问、上传失败、导出失败、流程重复处理、配置缺失、密钥占位、Redis 不可用、数据库脚本异常等必须纳入验收。
- 新旧关系：旧项目只作为能力和路径参考，新项目应重新设计产品结构、接口边界、数据结构和页面体验。
